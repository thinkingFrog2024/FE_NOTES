1. 模型的概念
 编写一个复杂的函数 ，这个函数具有输入和输出，给定一个输入，得到输出，通过计算输出值和真实值的差来调整参数，这样的函数就是大模型，这个大指的就是很大的参数量。
 这样的函数人类是写不出来的 所以我们会让机器来编写 这个编写的过程就称为机器学习 

 模型的参数 将会决定模型的行为 参数越多 训练成本往往越大
 不同的参数有不同的权重 权重越大 参数的影响就越大

 使用通用的数据进行训练 称之为 预训练 这样训练的模型具有很强的通用性 
 使用特定领域的数据训练 可以提高模型在特定领域的性能 这个叫做微调

 可以调整模型的温度 来调整输出 温度越高就会越自由


2. 智能体
模型本身就只能支持一问一答，也不具有查阅网页等能力。
但是我们希望模型具有这样的能力，所以我们给模型提供一个“函数”
模型调用这个函数 并且传递参数 拿到结果，这个模型就具有了他原来不具有的能力
我们把这样的模型 叫做智能体 因为他像人一样获得了使用工具的想法。

当我们向大模型描述我们的需求 大模型就能够分析我们的目的 根据我们的目的调用合适的agent来完成我们的任务

但是 大模型应该怎么和agent进行交流？
我们知道智能体的本质就是一个程序 所以我们最好规定好我这个程序需要需要的参数 大模型对我们的需求分析之后确定参数 再调用，其实就是大模型 agent之间约定的一个格式 这个格式称之为function call

我们的一个agent里面 通常不止一个功能 所以每个功能也肯定会有不同的参数需求，那么这个参数要求就叫做MCP,模型上下文协议 


所以这个过程中他们的角色就是这样的：
大模型：可以分析 但是没有操作的能力
AFENT：把模型的分析结果变成函数调用 再把调用结果返回
MCP：程序集 实现很多的功能




3. 检索增强生成 RAG
上文我们提到了我们希望模型能够具有某种能力，那么假设我们希望模型能够在一个巨大的知识库里面帮我们查询信息呢？
模型和我们的思维差不多：扫描这个知识库查找相关的部分
但是模型是使用向量进行查找的：查找数据里面和我们的输入的文本距离最小的那个向量
向量其实就是一个维度的描述 ，也就是说这些数据都会通过模型的一些算法 变成一个向量
向量的维度越多 那么就越精准


4. longChain
有一些任务比较固化 比如转换文件格式
这种任务的流程通常是固定的 ，所以为了不让ai因为自己每次都重新规划流程来浪费上下文窗口
所以我们可以规划一个固定的流程给ai使用，为了能够使用编程实现这个流程 我们创建了一个框架 叫做longchain


5. Skill
这个概念和longchain有点类似 但是longchain是一个流程 而skill是一个功能 
但是这个功能是一段纯prompt 不过当然 你可以在Skill里面调用一些工具

Skill其实也就是一个约定：我们和大模型之间的约定 在一些叫做Skill的文件里面可以存放各种各样的技能 模型可以根据这个Skill文件所在的文件夹的名称来确定这个Skill的作用 

这样的话 我们不需要在prompt里面输入大量的工具 大模型可以按需加载 

6. sub agent
上下文是一种宝贵的资源 为了避免上下文迅速膨胀 我们可以把一个任务的执行交给另一个agent 那么这个任务执行过程产生的上下文就不会保存在这个主要的agent里面 
这个被调用执行单个任务的agent就叫做sub agent


其实可以看出，除了agent MCP 是真正的在增强模型的能力 其他的一些概念做的都是：维护上下文窗口 提供优质的prompt 
lc的本质就是刚性的维护流程 不占用上下文 提供明确的步骤
sub也是把上下文隔离开 
我们的目标也会逐渐成：探索强大的模型 转换成 怎么维护一个模型友好地环境

7. 稠密模型 (Dense) vs. 稀疏模型 (Sparse/MoE)
我们在聊大模型的时候，经常听到“参数量”，比如 7B, 175B。但这些参数是怎么用的，决定了它是稠密还是稀疏。

### 稠密模型 (Dense Model)
**概念**：就像一个“全才医生”。不管病人是感冒、骨折还是肚子疼，这位医生都要动用脑子里**所有的知识**来诊断。
**特点**：
- 每次处理一个词（Token），模型里的**每一个参数**都要参与计算。
- 代表：GPT-3, Llama 2/3。
- 缺点：你想让模型更聪明（参数更多），计算量（显卡开销）就会线性暴涨。

### 稀疏模型 (Sparse Model / MoE)
**概念**：就像一家“综合医院”。医院里有心内科、骨科、肠胃科。病人来了，分诊台（Router）会判断，只把病人送去**对应的科室**。
**特点**：
- 虽然总参数量很大（比如 1.8 万亿），但处理某个词时，只激活其中的一小部分（比如 2 个专家）。
- 代表：GPT-4, Mixtral, DeepSeek-V3。
- 优点：用很少的计算量（省显卡），就能获得超大规模模型的知识储备。

### 核心架构：MoE (Mixture of Experts - 混合专家)
这是目前实现稀疏模型最主流的方式。它由两个核心部分组成：
1. **Router (路由/分诊台)**：负责判断当前的词该交给哪个专家处理。
2. **Experts (专家/科室)**：模型内部拆分出来的多个小网络，每个专家可能擅长不同的知识领域。

**直观对比图：**
```text
【稠密模型】                      【稀疏模型 (MoE)】
  输入 (Token)                   输入 (Token)
      ↓                             ↓
[ 所有参数都开火 ]               [ 路由 Router ]
      ↓                      /      |      \
     输出              [专家1]  [专家2(火)] [专家3]
                             \      |      /
                                  输出
```

### 总结对比
- **智力开销比**：稀疏模型（MoE）可以用更少的推理成本，达到和巨型稠密模型相当的智力。
- **显存占用**：稀疏模型虽然推理快，但因为总参数量在那，还是需要巨大的显存来“装下”那些没干活的专家。
- **训练难度**：稀疏模型很难训练，容易出现“某个专家太累，其他专家都在摸鱼”的情况（专家负载不均）。

8. 学术情报获取平台 (Paper Platforms)
AI 领域发展极快，了解最新的技术动向不能等教科书，必须直接看论文。

### arXiv (前哨站)
- **地址**：[arxiv.org](https://arxiv.org/)
- **定位**：预印本平台（未审稿先发布）。
- **特点**：速度极快，AI 圈的“朋友圈”。开发者在这里占坑和同步最新想法。

### OpenReview (辩论场)
- **地址**：[openreview.net](https://openreview.net/)
- **定位**：公开审稿平台。
- **特点**：能看到审稿人对论文的“毒舌”批判和作者的“回怼”，是深度理解论文优缺点的最佳去处。

### Papers with Code (实战库)
- **地址**：[paperswithcode.com](https://paperswithcode.com/)
- **定位**：论文 + GitHub 代码关联。
- **特点**：看完理论想跑代码？来这里找。它还有 SOTA (State-of-the-Art) 排行榜，告诉你谁是当前最强。

### Hugging Face Papers (讨论区)
- **地址**：[huggingface.co/papers](https://huggingface.co/papers)
- **定位**：AI 社区热门论文榜单。
- **特点**：最接地气，大佬们会在这里留言点赞，能帮你快速筛选出每天最值得读的那几篇。

### Nature / Science (殿堂)
- **地址**：[nature.com](https://nature.com/)
- **定位**：顶级学术期刊。

139→为了让 AI 能够像人一样操作浏览器，我们需要给它提供一套高效的“感知”方案。

141→### a11y 树 (Accessibility Tree)
142→- **概念**：无障碍树，是浏览器为屏幕阅读器（盲人使用）提供的“网页说明书”。
143→- **对比 DOM**：DOM 树包含所有琐碎的布局节点（如为了撑开高度的 div）；a11y 树只保留具有**语义**的节点（按钮、输入框、链接）。
144→- **AI 优势**：节点极少，自带语义（直接告诉 AI 这是个 Button），节省上下文 Token。
145→
146→### 获取方式
147→- **CDP (Chrome DevTools Protocol)**：AI 通过该协议连接浏览器，直接读取内存中的 a11y 树结构数据，100% 准确。
148→- **截图 (Screenshot)**：作为辅助，给多模态模型提供视觉参考，确认元素是否可见或被遮挡。
149→
150→### 上下文压力
151→操作浏览器是一个“重体力活”，a11y 树的文本和截图数据量巨大。为了不挤占主对话（写代码）的空间，通常将其交给 **Sub-Agent** 处理，主对话只接收处理结果。

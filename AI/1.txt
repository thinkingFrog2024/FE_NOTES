一、中后台系统架构现状对比分析
旨在对两个中后台前端项目，运营平台和内容安全平台进行工程化对比分析，并基于分析结果，为 AI Coding 设计一套标准化的工程架构与实施方案。

AI Coding 需要建设在标准化的工程架构上，因此高度复用和标准化方案是必备条件。

2.3 状态管理
建议保留并存，EdenX Global State 适合复杂的全局状态，Zustand 轻量灵活，适合页面级或领域级状态。工程标准应支持两种模式。相对而言，op-admin 仅通过 globalState 管理状态略显单薄，在复杂后台交互场景数据状态管理会混乱。
如果并存就必须划清边界，只有某些数据使用 globalState，组件内闭环的使用 useState，跨业务复用的一律使用 zustand。划定明确的标准，比告诉 AI "A 也可以，B 也可以" 可以降低出错的概率。

5. CSS 样式管理方案
目前 op-admin 使用 CSS Modules 方案，而 safe-web 使用 TailwindCSS + CSS Modules 文件的方案。 CSS 样式推荐使用 AI Friendly 的 tailwind 方案；这是目前 AI 最擅长的领域，优势在"Token 概率密度" 和 "上下文注意力"。
其他方案对比：
CSS Modules 和 StyleX，两者方案都实现了关注点分离，CSS 样式规则更自由。但是这样也会带来几个问题：
1. 跨文件 Coding 信息不同步的问题，AI 在生成 .tsx 文件后，往往会忘记在 css 文件中同步，或者出现错误的变量名样式名，这种“幻觉”在多文件编辑场景下发生概率更高。
2. LLM 写 <div className={styles.card}> 时，它的注意力回溯去寻找 .card 在 CSS 文件中的定义。一旦 Context Window 变长，LLM 极易产生幻觉（比如捏造一个不存在的 styles.cardHeader）
3. 样式规则更自由，AI 为了实现需求功能往往更倾向于自由发挥和定制，可维护性会受到影响。

6. 页面创建方法比较
两者都是用 op-admin 采用的 Edenx Route，其中 op-admin 页面由 loader 控制数据，safe-web 由页面内 useQuery hooks 控制数据。从路由模式上两者一致，只有数据控制方式不同，可以统一。

- Router Page 和 Page Data 不隔离，按局部引用原则，AI 更易理解。

7. 综合评估
7.1 相似点
- 基础技术栈完全一致：React 18 + TypeScript
- 使用 EdenX 框架， Edenx 路由和 Edenx GlobalState
- Rspack 构建
7.2 主要差异
所有差异点基于"渐进式"迭代的方式改造，即"增量需求对齐标准，存量逐步迁移"的方式设计，逐步将统一标准落到所有后台系统中。
1. 状态管理：EdenX 全局状态 vs Zustand
2. UI 组件库：单一 Arco vs 混合 Semi/Arco
3. 数据请求：自定义请求 vs @tanstack/react-query
4. 创建页面：Edenx Loader + Page vs 单一 Page
4. 权限控制：组件控制 vs 权限包装 Wrapper

8. 总结：后台系统标准化方案
除了以上系统现状，方案默认应该向趋同发展之外，AI Coding 更容易在配置化系统中落地，常见的方案诸如 formily，json-render，A2UI。因此需要在后台系统现状的基础上，往配置化方向迭代发展，根据 豆包 - 运营平台技术规划的设计，还应该补充表单、列表等场景化解决方案：

二、AI Coding 工程化建设

1. 组件知识
中后台系统开发中，UI 组件的复用率极高。AI Coding 最常见的问题是“幻觉” —— 例如捏造不存在的组件属性，或者混用不同版本的组件库 API（锁定一个版本组件库并建设专门的知识库很重要）。
组件知识工程的核心思路是将组件知识文档转化为"AI 可索引的知识"，通过 RAG 技术，在生成代码前强制 AI 查阅组件文档再生成，确保生成的 <Table> 或 <Form> 等通用组件符合我们工程的规范。
1.1 组件知识库
1.1.1 工程背景
传统的组件库文档索引基于关键词匹配，比如用户搜索"部门选择器"，"组织筛选"等关键词，匹配代码注释或者靠翻译查询，往往无法匹配到结果，因为注释或者代码过于技术化。那么，一方面可以通过人为 prompt 引导组件名或者文件路径，强调技术细节；一方面可以在代码注释或者函数组件变量上规范化，加强匹配度。
- 【通用组件库】Op-Admin 搭建 Arco Design 通用组件知识库后，可以先行启动 AI Coding 的试点。内容安全平台由于还在使用 semi 组件库，优先启动组件迁移的规范拉齐，UI 组件层对齐标准拉齐之后再落地 AI Coding。
- 【业务组件库】业务组件知识完成 markdown 文档编写对齐，再整理录入知识库。当前已完成业务组件库站点的搭建，已基本包含常用属性、API 和代码示例的整理。组件文档站点：[INTERNAL_DOCS_URL]。目前已实现组件更新自动更新文档，无需人工维护，新增 MR 即可创建 aime 自动任务，并推送更新。

2.1.1 解决思路：RAG + 意图增强
一句话总结：搭建一个轻量级 MCP Server，作为 IDE 或者 agent 的工具。 AI 规划代码时，可以通过 MCP 工具 recall_arco_design_knowledge 获取组件知识。
利用 RAG 服务，可以将代码组件信息向量化并注入业务信息，并人为利用 AI 为组件提供业务元数据，比如探索过程可以构建"意图描述层"。构建一个结构化的 JSON schema，用于描述组件的元信息，通过元信息向量更准确召回对应的组件，尤其是应对 LLM 对业务组件知识的缺失。

2.2 代码索引
背景：传统的文本搜索和关键词匹配，无法理解业务和代码，以及代码间的引用关系。AI 需要知道"什么叫策略工具，什么叫 trace 日志"，以及 "修改这个函数会影响哪些页面"。CodeGraph 是基于 AST 构建的代码关系图谱，结合 typescript-lsp 它能回答“谁调用了我”和“我调用了谁” 的问题。
Q: 什么是 TypeScript-LSP ？
TypeScript-LSP 指的是一组 Language Server 的实现，使用标准化的语言服务器协议（LSP，Language Server Protocol）为 TypeScript  提供丰富的代码智能。这使得除 VS Code （使用内部协议）之外的编辑器也能利用同样强大的 Language Services。
思路：没有银弹，需要调研自建或者魔改

Coding 工程是将 PE 和代码工程系统化、文件化。我们需要在代码仓库中内置一套 AI 能够理解的代码工程标准，而不是依赖每个开发者自己去写 Prompt。这部分通过标准化的 md 文件维护在仓库根目录。

写在前面
Cursor 内部分享了他们在 AI Coding 的 5 条核心经验，可以作为参考。无论是否用 AI Agent 还是 AI IDE，都应该按照这样的 workflow 开发。
1. 永远先进入 plan mode：plan mode 要求人类和 AI 必须先对齐需求，强调需求必须被澄清才能开发
2. 频繁开启新会话：新建对话可以避免多轮对话带来的无关历史对上下文的污染
3. 代码审查交给 AI：建立 bug 反馈闭环，AI 审查代码，将反馈沉淀，让 AI 基于历史反馈持续审查和反馈
4. 规划同步进行，执行异步交给云端：执行任务交给云端可以避免过重的 context 负担和资源占用
5. 建立专用的验证环境：减少人类反复手动的验证，让 AI 自测闭环

3.4 Context Engineering
3.4.1 工程背景
LLM 的上下文窗口有限，一次性把整个项目塞进去会影响推理准确率和性能。适当通过上下文整理，压缩的方案，减少上下文占用空间。利用 subagent 和 skills 动态载入 context 的特性，只在恰当的时机载入 context，避免初期撑满 context。
3.4.2 建设思路
1. 利用 CodeGraph 仅召回与当前任务直接相关的文件（如：关联的父组件、使用的 Hooks）。
2. 对于复杂的需求，前后端设计文档以及需求文档和 spec 都可以作为子任务隔离执行，分别启动不同的 Session 进行处理，最后再 summarize 合并关键信息。
3. 建设和打磨任务规划，什么任务用什么 agent，强调任务顺序，避免上下文注意力丢失等问题。

四、Coding Agent 探索
集团内部可用且覆盖较广的，值得推荐面向 Coding 场景的 Agent 有：coco， tmates， aime ... 所有探索基于这几项 agent 展开，并确定一个作为后续 AI Coding 的工程载体。
为什么是他们？
外部：

- Claude Code 和 Codex 是头部 Hardness Agent，推荐使用。

内部：
- Coco 是 Codebase 团队提供的 Code Agent 产品。其 CLI 功能对标 Claude Code / OpenAI Codex / Gemini CLI，因为是 CLI 工具，所以适用性广，几乎可以在任何环境运行，本地，开发机，ci 环境等。Coco 对于字节环境做了深度优化，可以联动 Codebase MR/DiffPreview 等功能，并无缝集成 Codebase/Lark MCP。
- TMates 是抖音研发团队产品，全称 TeamMates，是一个适合字节体质的前端研发交付平台，真实多角色产研协同合作，无缝衔接字节系生产工具，一站式完成「需求→开发→测试→发布」的全链路可靠代码交付，精准高效帮你完成任意规模前端仓库的需求承接。TMates 重点覆盖小 pd 场景准确率较高（3pd 以下准确率 80%），内置一系列衔接字节内部 MCP 的生产工具，覆盖“需求→开发→测试→发布”全链路。
- Aime 是面向公司内部使用的通用 Agent，旨在提升字节内部员工的工作效率。集成了公司内部的常用工具，应用场景包括：自动生成文档、自动化代码规划和编写、以及撰写测试用例，支持云编码环境。

内部几个 Agent 的共性：
- 支持 Skills，便于自建子任务工作流
- 支持 Agents.md 规范，便于沉淀工程规范
- 支持自定义 MCP 工具，可以承载团队自建的 MCP 服务和 RAG 服务
三者都支持了业界共识的 AI Coding 的标准工具链，基建相对全面。在公司内部社区群聊都相对活跃，迭代更新频繁，oncall 解决问题相对高效，技术咨询有保障。coco aime 背靠 codebase，集成了公司代码工程的基建，开发生态关联紧密。tmates 背靠抖音和 tt，有业务支撑。

5. CSS 样式管理方案
目前 op-admin 使用 CSS Modules 方案，而 safe-web 使用 TailwindCSS + CSS Modules 文件的方案。 CSS 样式推荐使用 AI Friendly 的 tailwind 方案；这是目前 AI 最擅长的领域，优势在"Token 概率密度" 和 "上下文注意力"。
其他方案对比：
CSS Modules 和 StyleX，两者方案都实现了关注点分离，CSS 样式规则更自由。但是这样也会带来几个问题：
1. 跨文件 Coding 信息不同步的问题，AI 在生成 .tsx 文件后，往往会忘记在 css 文件中同步，或者出现错误的变量名样式名，这种“幻觉”在多文件编辑场景下发生概率更高。
2. LLM 写 <div className={styles.card}> 时，它的注意力回溯去寻找 .card 在 CSS 文件中的定义。一旦 Context Window 变长，LLM 极易产生幻觉（比如捏造一个不存在的 styles.cardHeader）
3. 样式规则更自由，AI 为了实现需求功能往往更倾向于自由发挥和定制，可维护性会受到影响。

1. 组件知识
中后台系统开发中，UI 组件的复用率极高。AI Coding 最常见的问题是“幻觉” —— 例如捏造不存在的组件属性，或者混用不同版本的组件库 API（锁定一个版本组件库并建设专门的知识库很重要）。
组件知识工程的核心思路是将组件知识文档转化为"AI 可索引的知识"，通过 RAG 技术，在生成代码前强制 AI 查阅组件文档再生成，确保生成的 <Table> 或 <Form> 等通用组件符合我们工程的规范。
1.1 组件知识库
1.1.1 工程背景

2.2 代码索引

Q: 什么是 TypeScript-LSP ？
TypeScript-LSP 指的是一组 Language Server 的实现，使用标准化的语言服务器协议（LSP，Language Server Protocol）为 TypeScript  提供丰富的代码智能。这使得除 VS Code （使用内部协议）之外的编辑器也能利用同样强大的 Language Services。
思路：没有银弹，需要调研自建或者魔改
- 优先调研市场方案比如 BloopAI/bloop ，code-graph-rag
- 调研：基于 AST-Grep 或 Tree-sitter 自研实现

2.3 SDD( Plan First)

3. Agent 工程
Coding 工程是将 PE 和代码工程系统化、文件化。我们需要在代码仓库中内置一套 AI 能够理解的代码工程标准，而不是依赖每个开发者自己去写 Prompt。这部分通过标准化的 md 文件维护在仓库根目录。
同时我们应该强调工作流，
写在前面
Cursor 内部分享了他们在 AI Coding 的 5 条核心经验，可以作为参考。无论是否用 AI Agent 还是 AI IDE，都应该按照这样的 workflow 开发。
1. 永远先进入 plan mode：plan mode 要求人类和 AI 必须先对齐需求，强调需求必须被澄清才能开发
2. 频繁开启新会话：新建对话可以避免多轮对话带来的无关历史对上下文的污染
3. 代码审查交给 AI：建立 bug 反馈闭环，AI 审查代码，将反馈沉淀，让 AI 基于历史反馈持续审查和反馈
4. 规划同步进行，执行异步交给云端：执行任务交给云端可以避免过重的 context 负担和资源占用
5. 建立专用的验证环境：减少人类反复手动的验证，让 AI 自测闭环

3.1.1 工程背景
README.md 是给人类开发者看的，agents.md 是给 AI 开发者看的。agents.md 可以明确指出代码目录结构（如 src/routes 是页面，src/components 是组件），也可以明确代码规范（基于 lint 要求），可以在 agents.md 中提供标准页面或者模块作为最佳实践，让 AI Coding 时作为参考。

3.3 Prompt Engineering
3.3.1 工程背景
在工程化场景下，我们不能依赖开发者临时编写的 Prompt，而必须提供一套经过验证的、结构化的、版本管理的提示词模板库。
3.3.2 Actions
- Spec Prompt：调用 Arco Design MCP 和 CodeGraph（optional），基于 PRD 和 API 文档生成 Spec
- API Prompt：基于后端 API 变更生成 API 文档
- Coding Prompt：强调引用 agents.md 和使用 skills，生成代码、解决构建错误和提交代码

3.4 Context Engineering
3.4.1 工程背景
LLM 的上下文窗口有限，一次性把整个项目塞进去会影响推理准确率和性能。适当通过上下文整理，压缩的方案，减少上下文占用空间。利用 subagent 和 skills 动态载入 context 的特性，只在恰当的时机载入 context，避免初期撑满 context。
3.4.2 建设思路
1. 利用 CodeGraph 仅召回与当前任务直接相关的文件（如：关联的父组件、使用的 Hooks）。
2. 对于复杂的需求，前后端设计文档以及需求文档和 spec 都可以作为子任务隔离执行，分别启动不同的 Session 进行处理，最后再 summarize 合并关键信息。
3. 建设和打磨任务规划，什么任务用什么 agent，强调任务顺序，避免上下文注意力丢失等问题。

四、Coding Agent 探索
集团内部可用且覆盖较广的，值得推荐面向 Coding 场景的 Agent 有：coco， tmates， aime ... 所有探索基于这几项 agent 展开，并确定一个作为后续 AI Coding 的工程载体。
为什么是他们？
外部：
- Claude Code 和 Codex 是头部 Hardness Agent，推荐使用。

内部：
- Coco 是 Codebase 团队提供的 Code Agent 产品。其 CLI 功能对标 Claude Code / OpenAI Codex / Gemini CLI，因为是 CLI 工具，所以适用性广，几乎可以在任何环境运行，本地，开发机，ci 环境等。Coco 对于字节环境做了深度优化，可以联动 Codebase MR/DiffPreview 等功能，并无缝集成 Codebase/Lark MCP。
- TMates 是抖音研发团队产品，全称 TeamMates，是一个适合字节体质的前端研发交付平台，真实多角色产研协同合作，无缝衔接字节系生产工具，一站式完成「需求→开发→测试→发布」的全链路可靠代码交付，精准高效帮你完成任意规模前端仓库的需求承接。TMates 重点覆盖小 pd 场景准确率较高（3pd 以下准确率 80%），内置一系列衔接字节内部 MCP 的生产工具，覆盖“需求→开发→测试→发布”全链路。
- Aime 是面向公司内部使用的通用 Agent，旨在提升字节内部员工的工作效率。集成了公司内部的常用工具，应用场景包括：自动生成文档、自动化代码规划和编写、以及撰写测试用例，支持云编码环境。

内部几个 Agent 的共性：
- 支持 Skills，便于自建子任务工作流
- 支持 Agents.md 规范，便于沉淀工程规范
- 支持自定义 MCP 工具，可以承载团队自建的 MCP 服务和 RAG 服务
三者都支持了业界共识的 AI Coding 的标准工具链，基建相对全面。在公司内部社区群聊都相对活跃，迭代更新频繁，oncall 解决问题相对高效，技术咨询有保障。coco aime 背靠 codebase，集成了公司代码工程的基建，开发生态关联紧密。tmates 背靠抖音和 tt，有业务支撑。

# 核心执行流程 (必须严格遵守顺序)

## 阶段一：架构上下文感知 (阅读 AGENTS.md)
在构思任何技术方案之前，你必须先建立对当前工程架构的全局和局部认知：
1. **读取全局规范**：主动读取工程目录下的 `AGENTS.md` 文件，理解项目的全局架构约定、状态管理模式、路由规范以及公共工具方法。
2. **读取局部约束**：主动扫描并读取本次需求涉及到的子目录（如特定的业务模块或组件目录）下的 `AGENTS.md` 文件，掌握该子域的特定依赖关系、业务规则和开发要求。
> 注意：在未确认已充分理解上述架构上下文之前，禁止开始编写方案。

## 阶段二：技术方案推演与设计 (`design.md`)
基于需求文档和阶段一获取的架构上下文，按照 OpenSpec 规范编写 `design.md`。

## 阶段三：开发技能匹配
根据需求内容，自动识别是否调用以下技能。在design.md里各技能适配的场景，主动强调要使用的技能。

### 技能触发规则

|----------|------|------|
| 需创建新模块/Tab/顶级菜单 | `create-op-admin-tab` | 在 routes.tsx 注册新 Tab |

| 涉及 API 请求/数据获取 | `api-request-generating` | 规范 useRequest 写法 |

5. 人类验证
由于 AI Coding 的结果 emo start 基本可以零报错启动，所以人类目前要做的就是本地启动服务，观察网页和交互与产品功能 match，做微调代码和提交代码 MR 即可。

根据团队要求和数据统计的必要性，需要按照以下 checklist 完成人类验证：
[] 提交 AI Coding 的代码，记录基准代码行数（AI 生成的代码行数）
[] emo start 启动服务和网页验证
[] 微调和 bugfix，确认与产品功能 match
[] 提交人类修正的代码
[] 记录 MR 的代码行数变更，统计代码采纳率
[] 记录 AI Coding 过程到文档 使用记录

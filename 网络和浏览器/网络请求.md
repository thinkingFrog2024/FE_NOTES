# 网络请求与通信

## 1. 协议基础

### HTTP 与 HTTPS

#### 基本概念
- **HTTP (HyperText Transfer Protocol)**：超文本传输协议，互联网应用最广泛的协议。设计目的是发布和接收 HTML 页面。
- **HTTPS (HyperText Transfer Protocol Secure)**：安全版 HTTP。在 HTTP 下加入 SSL/TLS 层，提供加密传输、完整性校验和身份验证。

#### 核心区别
| 特性 | HTTP | HTTPS |
| :--- | :--- | :--- |
| **安全性** | 明文传输，易被窃听、篡改。 | **加密传输**，基于 SSL/TLS，具备信息加密、完整性校验、身份验证。 |
| **端口** | 80 | 443 |
| **证书** | 不需要。 | 需要 CA 证书（通常收费）。 |
| **性能** | 较快（仅 TCP 握手）。 | 稍慢（需 TCP + SSL/TLS 握手，消耗更多资源）。 |
| **SEO** | 权重较低。 | 搜索引擎（如 Google）优先索引，利于 SEO。 |

#### HTTPS 工作原理（简化）
1.  **客户端请求**：连接服务端 443 端口。
2.  **服务端下发证书**：包含公钥。
3.  **客户端验证证书**：验证合法性，生成随机值（预主密钥），用公钥加密发送给服务端。
4.  **服务端解密**：用私钥解密得到随机值。
5.  **生成会话密钥**：双方利用随机值生成相同的对称密钥。
6.  **加密通信**：后续数据传输均使用该对称密钥加密。

## 2. 前端请求方式

### DOM 标签触发
浏览器根据 HTML 标签自动发起请求，无需 JS 配置。
- `<a>`：点击跳转。
- `<img>`：自动请求图片资源。
- `<link>`：加载 CSS 等外部文件。
- `<script>`：加载 JS 脚本。
- `<iframe>`：嵌入 HTML 页面。

**JS 监听加载状态：**
```javascript
const img = new Image();
img.src = 'path/to/image.jpg';
img.onload = () => console.log('加载成功');
img.onerror = () => console.log('加载失败');
```

### AJAX (XHR)
**Asynchronous JavaScript and XML**，在不刷新页面的情况下与服务器交换数据并更新部分网页。


#### 优缺点
- **优点**：无刷新体验好，按需加载减少服务器压力。
- **缺点**：无浏览历史（无法回退），存在跨域问题，对 SEO 不友好。

首先我们明确 javascript本身 完全不具有发起网络请求的能力。
当我们的代码在宿主环境里面运行的时候，我们的运行上下文 或者说词法环境 会被注入进很多东西 在浏览器里面就有fetch XML，在node里面就是http https模块。
这是通过window对象注入 ，就比如fetch，其实是在window上面加了一个fetch，于是在代码调用的时候 就会向上层的词法环境访问 直到找到这个window.fetch。
正是这种特性 允许我们通过修改window.fetch 来实现一些自定义的功能 比如 全局拦截 全局修改请求头 全局修改响应体 等等。
但是这也可能导致xss注入 或者库冲突这样的问题。
polifill也是这样实现的 在一些不支持fetch的旧浏览器 我们可以自己写一个window.fetch 来实现类似的功能。

这提醒我们，在使用window对象的灵活性的同时也要保持洁净 


#### 原生操作步骤
```javascript
const btn = document.querySelector('button');
btn.onclick = function() {
    // 1. 创建对象
    const xhr = new XMLHttpRequest();
    
    // 2. 初始化 (Method, URL)
    xhr.open('GET', 'http://127.0.0.1:8000/server?a=100&b=200');
    
    // 3. 设置请求头 (可选，POST 请求通常需要)
    // xhr.setRequestHeader('Content-Type', 'application/json');
    
    // 4. 发送请求
    xhr.send(); 
    
    // 5. 监听状态变化
    xhr.onreadystatechange = function() {
        if (xhr.readyState === 4) { // 4: 完成
            if (xhr.status >= 200 && xhr.status < 300) {
                // 成功
                console.log('状态码:', xhr.status);
                console.log('响应头:', xhr.getAllResponseHeaders());
                console.log('响应体:', xhr.response);
                
                // 自动解析 JSON (如果设置了 responseType)
                // const data = JSON.parse(xhr.response);
            } else {
                // 失败
                console.error('请求失败', xhr.status);
            }
        }
    }
    
    // 其他常用事件
    xhr.timeout = 2000; // 超时设置
    xhr.ontimeout = () => console.log('请求超时');
    xhr.onerror = () => console.log('网络错误');
}
```

#### 常见问题解决
- **取消请求**：`xhr.abort()`。
- **重复请求处理**：发送新请求前检查并取消上一个未完成的请求。

### Fetch API
现代浏览器提供的原生替代方案，基于 Promise，语法更简洁。

```javascript
fetch('http://api.example.com/data', {
    method: 'POST',
    headers: {
        'Content-Type': 'application/json'
    },
    body: JSON.stringify({ name: 'test' })
})
.then(response => {
    if (!response.ok) throw new Error('Network response was not ok');
    return response.json(); // 解析 JSON
})
.then(data => console.log(data))
.catch(error => console.error('Fetch error:', error));
```

从上古的xhr 演变到fetch ，两者的思想是截然不同的：xhr诞生的时间还没有promise async await 这个时候为了实现异步的机制 只能通过事件驱动的形式完成，而fetch 则是和promise async await 一起诞生的 基于promise链式调用的思想。
但是 fetch并不能实现xhr的上传进度监控，只能搞定下载进度 上传进度却没啥办法，所以现在的大文件上传仍然是基于xhr。

当然 两者并不只是思想能力的差异 底层也是截然不同的 他可以原生支持流式传输 对大文件下载是非常重用的。
### Service Worker 与 Cache API 实现离线应用

`Fetch` 只是发请求的工具，真正实现离线能力的是 **Service Worker** (拦截器) 和 **Cache API** (仓库)。

#### 1. 核心理论：拦截与代理
Service Worker 本质上是一个**驻留在浏览器后台的脚本**，它独立于网页运行，就像是一个**客户端代理服务器**。
- **拦截能力**：当网页发起 Fetch 请求时，Service Worker 可以通过 `fetch` 事件拦截这个请求。
- **决策权**：它可以决定是放行请求去网络，还是直接从 Cache API 拿出缓存返回，甚至可以凭空构造一个 Response 返回。
- **生命周期**：它有独立的生命周期（Install -> Activate -> Fetch），即使网页关闭了，它依然可以在后台运行（比如做离线缓存更新或推送通知）。

#### 2. Cache API：请求/响应对象的持久化仓库
与 localStorage 只能存字符串不同，Cache API 是专门为存储 **Request** 和 **Response** 对象设计的。
- **键值对存储**：Key 是 Request 对象（通常是 URL），Value 是 Response 对象。
- **完整性**：它存储的是完整的 HTTP 响应，包括状态码、Headers 和 Body（二进制流）。
- **异步操作**：所有操作都是基于 Promise 的，不会阻塞主线程。

#### 3. 离线工作的原理
1.  **首次访问（在线）**：
    - 网页发起请求。
    - Service Worker 拦截请求，去网络获取数据。
    - 获取成功后，**克隆**一份 Response 存入 Cache API，另一份返回给网页。
2.  **再次访问（离线/弱网）**：
    - 网页发起请求。
    - Service Worker 拦截请求。
    - 发现网络不通或策略设置为“Cache First”。
    - 直接从 Cache API 中取出之前存好的 Response 返回给网页。
    - **结果**：网页以为自己发起了网络请求并拿到了数据，但实际上根本没有流量产生。

#### 4. 关键点：流（Stream）的处理
Fetch 的 Response Body 是一个 ReadableStream。
- **单次消费**：流只能被读取一次。一旦被浏览器渲染读取了，流就空了。
- **克隆必要性**：如果要同时把响应 **显示在页面上** 和 **存入缓存**，必须在读取前调用 `.clone()` 方法，分出两个流来分别处理。



### Axios
基于 Promise 的 HTTP 客户端，用于浏览器和 Node.js。

> **Axios 和 AJAX 的关系**
> 
> 可以把它们理解为**工具库 (Axios)** 和 **底层技术 (AJAX)** 的关系。
> 
> *   **AJAX (XHR)**：是浏览器提供的一种**原生技术能力**（像是一个裸露的发动机），允许网页在不刷新的情况下与服务器通信。直接使用原生 XHR 比较繁琐（需要处理状态码、手动解析数据等）。
> *   **Axios**：是基于原生 XHR 封装好的**高级工具库**（像是一辆组装好的汽车）。它帮我们把繁琐的底层操作（如设置请求头、转换数据、处理 Promise）都封装好了，让我们用起来更简单、舒服。
> 
> **总结**：Axios 在浏览器端**底层就是用的 AJAX (XMLHttpRequest)** 实现的。

> **HTTP 客户端是什么？**
> 简单来说，就是**发起请求的一方**。
> - **服务端 (Server)**：提供资源（数据），监听端口，等待被访问。
> - **客户端 (Client)**：需要资源，主动向服务器发起请求。
> 
> 常见的客户端有浏览器、Postman、curl 等。Axios 就是一个让你的代码具备“客户端能力”（发起请求、接收响应）的工具库。

#### 核心特性
- 浏览器端发起 XMLHttpRequests。
- Node.js 端发起 http 请求。
- 支持 Promise API。
- **拦截器**：请求/响应拦截。
- **自动转换 JSON**。
- 客户端防御 CSRF。

#### 基本使用
```javascript
// GET
axios.get('/user', { params: { id: 123 } })
  .then(res => console.log(res.data))
  .catch(err => console.error(err));

// POST
axios.post('/user', { firstName: 'Fred' })
  .then(res => console.log(res))
  .catch(err => console.error(err));
```

#### 封装示例 (带拦截器)
```javascript
const service = axios.create({
    baseURL: 'https://api.example.com',
    timeout: 5000
});

// 请求拦截器
service.interceptors.request.use(
    config => {
        const token = localStorage.getItem('token');
        if (token) {
            config.headers['Authorization'] = 'Bearer ' + token;
        }
        return config;
    },
    error => Promise.reject(error)
);

// 响应拦截器
service.interceptors.response.use(
    response => response.data,
    error => {
        if (error.response && error.response.status === 401) {
            // 处理未登录
        }
        return Promise.reject(error);
    }
);
```

### axios在真实项目里面的使用

一般创建一个全局唯一的axios实例，并且添加请求拦截器，响应拦截器，在请求拦截器中添加特定的请求头，或者是添加前端埋点。在响应拦截器中会处理http状态码 code，验证请求是否成功，若成功的话可以对数据做一定的处理，若不成功可以抛出错误，在全局的错误捕捉里面捕获错误，从而避免网络请求失败的逻辑散落在代码中 。

另外 如果请求的发起直接由前端传递参数进行，如果参数有地方写错了，那么请求就可能出现问题，并且这种方案也很麻烦，我们需要编写大量相似的代码，所以一般采用：根据后端的api定义生成前端代码的方案，把每个api的调用封装成一个函数 从而避免了手动编写大量代码。

### Beacon API 与 Fetch Keepalive

在页面卸载（关闭、刷新、跳转）时发送数据是一个经典难题。
- **同步 XHR**：会阻塞页面关闭，用户体验极差（浏览器已逐步禁用）。
- **普通异步请求**：页面关闭时，浏览器会直接**取消**未完成的请求，导致数据丢失。

#### 1. Beacon API (`navigator.sendBeacon`)
专门用于解决“页面临终遗言”问题的 API。
- **异步非阻塞**：将数据放入浏览器队列，即使页面关闭，浏览器也会在后台完成发送。
- **可靠性高**：比普通 XHR/Fetch 更可靠，不会因为页面卸载而中断。

```javascript
// 监听 visibilitychange 而不是 unload (unload 不可靠)
document.addEventListener('visibilitychange', () => {
    if (document.visibilityState === 'hidden') {
        const data = JSON.stringify({ event: 'page_hide', time: Date.now() });
        // 只能 POST，且无法自定义 Header (除了 Content-Type)
        navigator.sendBeacon('/log', data);
    }
});
```

#### 2. Fetch Keepalive (推荐)
`Fetch API` 的 `keepalive` 选项是 `sendBeacon` 的现代替代品。
- **功能更强**：支持自定义 Headers、各种 HTTP 方法（GET/PUT 等），比 `sendBeacon` 灵活。
- **共用配额**：它和 `sendBeacon` 共享 64KB 的数据大小限制。

```javascript
const logData = { event: 'page_hide', time: Date.now() };

fetch('/log', {
    method: 'POST',
    body: JSON.stringify(logData),
    headers: {
        'Content-Type': 'application/json',
        'Authorization': 'Bearer token' // sendBeacon 做不到这个
    },
    keepalive: true // 关键！开启后拥有“不死之身”
});
```

#### 3. 最佳实践
- **不要用 `unload` / `beforeunload`**：这两个事件在移动端极不可靠，且会破坏 BFCache（往返缓存）。
- **请用 `visibilitychange`**：当 `visibilityState` 变为 `hidden` 时发送数据，这是最可靠的时机。
- **数据量限制**：无论是 `sendBeacon` 还是 `keepalive`，请求体都不能超过 **64KB**。

## 3. 浏览器跨域 (CORS)

### 原因
**同源策略 (Same-Origin Policy)**：协议、域名、端口必须完全相同。

### 解决方案
1.  **CORS (Cross-Origin Resource Sharing)**：最正统的方案。后端设置响应头 `Access-Control-Allow-Origin` 等。
2.  **JSONP**：利用 `<script>` 标签不受同源策略限制。
    *   仅支持 GET。
    *   存在 XSS 风险。
3.  **代理 (Proxy)**：
    *   **开发环境**：Webpack/Vite 代理 (`devServer.proxy`)。
    *   **生产环境**：Nginx 反向代理。

## 4. 实时通信

### 轮询 (Polling)
- **短轮询**：客户端定时发送请求（`setInterval`）。
- **长轮询**：服务端挂起请求，直到有数据更新才返回。

### SSE (Server-Sent Events)
服务端向客户端单向推送流式数据。基于 HTTP 协议。 像chatGPT的发送消息 其实就是sse
- **本质**：HTTP 长连接 + 特定的文本格式 (`text/event-stream`)。
- **能力来源**：浏览器宿主环境提供的 `EventSource` 接口。
- **特点**：
    - **轻量级**：不像 WebSocket 那样需要握手和复杂的协议升级。
    - **自动重连**：浏览器发现连接断开会自动尝试重连（带上 `Last-Event-ID`）。
    - **单向通信**：只能服务器推给客户端，客户端不能通过这个连接发消息。

```javascript
// 1. 建立连接 (浏览器自动处理握手)
const sse = new EventSource('/events');

// 2. 监听消息
sse.onmessage = (event) => {
    // data 只能是文本
    console.log('收到消息:', event.data);
};

// 3. 监听自定义事件 (后端发 event: ping)
sse.addEventListener('ping', (e) => {
    console.log('收到自定义事件:', e.data);
});

sse.onerror = (err) => {
    console.error('SSE 错误:', err);
    // 可以在这里决定是否关闭，否则浏览器会一直重连
    // sse.close();
};
```

#### SSE vs WebSocket
| 特性 | SSE | WebSocket |
| :--- | :--- | :--- |
| **方向** | **单向** (Server -> Client) | **双向** (全双工) |
| **协议** | 标准 HTTP | 自定义 ws 协议 (需升级) |
| **数据格式** | **只能文本** (UTF-8) | 文本 + 二进制 (ArrayBuffer/Blob) |
| **重连机制** | **浏览器内置自动重连** | 需要手动写代码实现 (心跳/重连) |
| **适用场景** | AI 打字机、消息通知、股票行情 | 聊天室、多人游戏、协同编辑 |


### WebSocket
全双工通信协议。
- **特点**：实时性强，双向通信，支持二进制。
- **库**：`socket.io` (推荐)


对于实时的流媒体 支持二进制这个特性是非常重要的
如果不能传递二进制数据 只能传递文本的话 那么如果传递图片  那就只能把图片转成base64字符串 这会导致体积增加 并且先编码 再解码 也很消耗资源

但是支持二进制的话 直接传递原始的字节流就行 体积也不变大 也不用编码再解码 

就比如直播
视频数据都是一帧一帧的 每帧其实就是压缩之后的二进制数据 收到这些二进制数据之后 直接传递给vedio标签就行 

或者游戏的话 通常是定义一个二进制的协议 用几个字节表示一个执行 但是如果不使用二进制数据 而是一个json对象 就会消耗好几倍的字节 


```javascript
// 原生 WebSocket
const ws = new WebSocket('ws://localhost:8080');

ws.onopen = () => {
    ws.send('Hello Server');
};

ws.onmessage = (event) => {
    console.log('收到:', event.data);
};

// 心跳检测
setInterval(() => {
    if (ws.readyState === WebSocket.OPEN) ws.send('ping');
}, 30000);
```

## 5. 网络安全

### XSS (Cross-Site Scripting) 跨站脚本攻击
攻击者在网页中注入恶意脚本 浏览器就会执行这些脚本 
- **反射型**：恶意脚本在 URL 中。就比如<div>您搜索的是：<script>alert(1)</script></div> 浏览器在渲染的时候 就会执行这个script 
- **存储型**：恶意脚本存入数据库（最危险）。就比如我觉得很赞！<script>偷你cookie.js</script> 。 这个如果被存储到了数据库 那么每个看见这个数据的用户的cookie都会泄漏
- **DOM 型**：利用 DOM API 修改页面结构。就比如前端使用url参数：document.getElementById('box').innerHTML = location.hash.slice(1); 但是黑客手动添加了这样的参数http://你的网站.com#<img src=x onerror=alert(1)> 于是网站的dom结构就被修改了

**防御**：
1.  **输入过滤**。
2.  **输出转义**（HTML Encoding）。
3.  **CSP (Content Security Policy)**。告诉浏览器，只许加载我域名的 JS，其他域名的 JS 一律不准执行！内联脚本 ( <script>... ) 也不准执行！
4.  **HttpOnly Cookie**：防止 JS 读取 Cookie。后端给浏览器“发证”（Set-Cookie）的时候，必须加上HttpOnly

### CSRF (Cross-Site Request Forgery) 跨站请求伪造
诱导用户访问恶意网站，利用用户的登录状态发起请求。
想象一下，你刚登录了网银网站 bank.com ，你的浏览器里存了网银的 Cookie（身份凭证）。
这时候，你手贱点开了一个色情网站 evil.com 。
这个网站上有一个恶意脚本，它会在用户不知情的情况下，利用你浏览器里的 Cookie 发起一个转账请求到 bank.com 。
因为 bank.com 要求用户登录才能转账，所以这个请求就会成功，并且你会认为自己成功转账了。

**防御**：
1.  **Token 验证**。服务端会在下发数据的时候 给前端下发一个token 这个token可以放在responseHeader里面 ，在前端以后请求的时候 就需要在请求头里面携带这个token 否则服务端将会拒绝请求
2.  **Referer 验证**。服务器会检查请求头里面的Refer字段 这个字段指明了请求发起的地址 但是这个请求头也是可以伪造的
3.  **SameSite Cookie**。Cookie有一个sameSite属性 可以设置成严格模式 这样第三方完全禁止携带cookie进行请求 只有在当前域名下才会携带cookie，但是这样用户的体验就不好了。这里的第三方 指的是：请求的发起者 和目标资源的域名不同。举个例子，你在某个网站登录过了，之后你退出这个网站，又从百度进入这个网站 你会发现需要重新登陆，因为这个时候你是从百度的域名 请求目标资源 百度就是第三方 不能携带你的cookie，为了平衡安全和体验 我们可以使用Lax模式：只有使用顶级导航的时候才会携带cookie 
顶级导航指的是：会修改浏览器地址栏url的操作 比如点击链接 在地址栏输入url 点击书签 表单提交 ，至于图片请求 AJAX 这样的就不算顶级请求 ，
这是因为顶级导航的话 是我们要换到某个网站 浏览器就为我们方便登陆带上了cookie

### Session (会话)
传统的身份验证方式，**有状态**。
- **原理**：
    1.  用户登录成功，服务端在内存（或 Redis）里生成一个唯一的 `session_id`，并关联用户信息。
    2.  服务端通过 `Set-Cookie` 把这个 `session_id` 发给浏览器。
    3.  浏览器以后每次请求都会自动带上这个 Cookie。
    4.  服务端拿着 Cookie 里的 `session_id` 去内存里找：“哦，这是张三，还没过期，放行。”
- **缺点**：
    - **服务器压力大**：用户越多，内存占用越大。
    - **扩展性差**：多台服务器集群时，必须搞 Session 共享（如存 Redis），否则用户在这个服务器登录了，下次请求打到另一台服务器就不认识了。

### Session vs Cookie vs JWT
不要混淆这三个概念：
- **Session**：服务器端的**数据**（会员档案）。
- **Session ID**：客户端持有的**唯一标识**（会员卡号）。
- **Cookie**：浏览器端的**存储容器**（钱包），用来装 Session ID（或者 JWT）。
- **JWT**：自带信息的**加密票据**（上面刻着名字和有效期的万能卡）。

**比喻：**
- **Session 模式**：你拿着会员卡（Cookie 里的 Session ID）去健身房。前台（服务器）刷一下卡，查电脑里的档案（Session），确认你是会员，放行。
- **JWT 模式**：你拿着一张签了名的纸条（JWT）去健身房。纸条上写着“我是张三，有效期到明天”。前台（服务器）看一眼签名是真的，直接放行，**根本不用查电脑档案**。

**注意：**
JWT 既可以放在 HTTP Header (`Authorization: Bearer ...`) 里，也可以放在 Cookie 里。
- 放 Header：前端需要手写代码取出来带上，容易被 XSS 偷走（如果存 LocalStorage）。
- 放 Cookie：配合 `HttpOnly`，浏览器自动带上，JS 读不到，**防 XSS**，但要小心 CSRF。

## 6. 网络状态 API
需要使用web api中的Bom 用于操作浏览器窗口 ，
navigator就是典型的bom api ，它提供了很多属性和方法 ，比如判断浏览器是否支持某个api ，获取浏览器的信息 ，操作浏览器的窗口 ，操作浏览器的历史记录 ，操作浏览器的cookie 等。

```javascript
// 判断在线状态
console.log(navigator.onLine);

window.addEventListener('online', () => console.log('网络已连接'));
window.addEventListener('offline', () => console.log('网络断开'));

// 获取详细连接信息 (部分浏览器支持)
const connection = navigator.connection || navigator.mozConnection || navigator.webkitConnection;
if (connection) {
    console.log('下行速度:', connection.downlink);
    console.log('网络类型:', connection.effectiveType); // '4g', '3g', etc.
    console.log('RTT:', connection.rtt);
}
```

## 7. HTTP 缓存策略

前端性能优化的核心。缓存对了，用户秒开；缓存错了，发版后用户还是旧页面（灾难）。

### 强缓存 (Strong Cache)
浏览器直接判断本地缓存是否过期。**如果没过期，根本不发请求给服务器**。
- **状态码**：`200 OK` (Size 显示 `memory cache` 或 `disk cache`)。
- **控制字段**（响应头）：
    1.  **Cache-Control** (HTTP/1.1，优先级高)：
        - `max-age=3600`：缓存 1 小时。
        - `no-cache`：**不要强缓存**，每次都要去服务器验证（协商缓存）。
        - `no-store`：**绝对禁止缓存**，每次都下载新的（用于隐私数据）。
        - `public` / `private`：中间代理（CDN）是否可以缓存。
    2.  **Expires** (HTTP/1.0，优先级低)：绝对时间，依赖客户端系统时间，不靠谱。

### 协商缓存 (Negotiation Cache)
强缓存失效（过期）后，浏览器携带缓存标识向服务器发起请求，询问资源是否更新。
- **状态码**：
    - **304 Not Modified**：没变，继续用本地的。
    - **200 OK**：变了，服务器发新的给你。
- **控制字段**（成对出现）：
    1.  **ETag / If-None-Match** (优先级高)：
        - 服务器生成文件的唯一指纹（Hash）。
        - 只要文件内容变了一个字节，Hash 就会变。**最精准**。
    2.  **Last-Modified / If-Modified-Since** (优先级低)：
        - 文件的最后修改时间。
        - 精度只到秒。如果 1 秒内改了多次，或者只改了时间没改内容，它就失效了。

### 最佳实践
1.  **HTML 文件**：`Cache-Control: no-cache`。永远走协商缓存，确保用户第一时间拿到最新的引用（JS/CSS 文件名）。
2.  **JS/CSS/图片 (带 Hash)**：`Cache-Control: max-age=31536000` (一年)。文件名带 Hash (如 `main.a1b2c.js`)，内容变了文件名就变了，直接当新文件下载；没变就一直用缓存。

## 8. HTTP 版本演进

面试必问，性能优化的基石。

### HTTP/1.1 (队头阻塞的噩梦)
- **长连接 (Keep-Alive)**：默认开启。同一个 TCP 连接可以发送多个 HTTP 请求，避免了每次请求都重新握手。
- **缺点**：
    1.  **队头阻塞 (Head-of-Line Blocking)**：
        - 这里的阻塞发生在**应用层**。
        - 一个 TCP 连接**同一时间只能处理一个请求**。如果第一个请求（比如一个大文件下载）没结束，后面的请求（JS/CSS）即使很小，也必须排队等着。
        - 想象一下：单车道公路，前面有辆大卡车坏了，后面所有的法拉利都得堵着。
    2.  **Header 臃肿**：
        - HTTP/1.1 是纯文本协议。
        - 每次请求都会带上一堆重复的 Header（`User-Agent`, `Cookie`, `Accept` 等），可能有几百字节甚至几 KB，严重浪费带宽。
    3.  **并发限制**：
        - 浏览器限制同一个域名下只能开 **6 个** TCP 连接。
        - **优化黑科技（已过时）**：
            - **雪碧图 (Sprite)**：把几十张小图标拼成一张大图，减少请求数。
            - **域名分片 (Domain Sharding)**：把静态资源分散到 `a.img.com`, `b.img.com`，突破 6 个连接的限制。

### HTTP/2 (多路复用的救星)
- **二进制分帧 (Binary Framing)**：
    - 不再是纯文本，而是把数据切成一个个小的**二进制帧 (Frame)**。
    - 机器解析二进制比解析文本快得多，也更健壮。
- **多路复用 (Multiplexing)**：
    - **核心**：一个域名**只用一个 TCP 连接**！
    - **原理**：所有的请求（JS, CSS, 图片）被打散成无数个小帧，在同一个连接里**乱序并发传输**。
    - **Stream ID**：每个帧都带有一个流 ID，浏览器收到后根据 ID 把它们重新组装成完整的文件。
    - **后果**：
        - **雪碧图不需要了**！因为请求多也不怕阻塞。
        - **小图合并反而有害**：因为合并后的大图如果改了一个像素，整个缓存都失效了。
- **头部压缩 (HPACK)**：
    - 客户端和服务器维护一张**动态表**，记录之前发过的 Header。
    - 后面再发相同的 Header（如 `User-Agent`），只需要发一个索引号（比如 `index: 1`），几百字节瞬间变几字节。
- **服务端推送 (Server Push)**：
    - 浏览器请求 `index.html`，服务器猜到你肯定要用 `style.css`，于是**主动**把 `style.css` 推给你，不用等你来要把。
    - *注：实际落地很难，很多浏览器已废弃此功能。*

### HTTP/3 (基于 UDP 的颠覆)
- **背景：HTTP/2 的软肋**：
    - HTTP/2 解决了应用层的队头阻塞，但**TCP 协议本身有队头阻塞**。
    - **TCP 丢包问题**：TCP 是可靠传输，必须保证顺序。如果第 1 个包丢了，操作系统会把收到的第 2-100 个包全部**卡住**，直到第 1 个包重传成功。
    - 在 HTTP/2 里，因为所有请求都挤在一个 TCP 连接里，一个包丢了，**所有请求（所有流）都会被卡住**。这比 HTTP/1.1 还惨（1.1 至少还有 6 个连接，坏了一个还有 5 个能用）。
- **QUIC 协议**：抛弃 TCP，改用 **UDP**！
    - **UDP 不靠谱？** QUIC 在应用层自己实现了可靠传输机制，不用操作系统管。
    - **无队头阻塞**：
        - QUIC 的流（Stream）之间是**真正独立**的。
        - Stream A 丢包了，只会阻塞 Stream A，Stream B 照样跑，完全不受影响。
    - **0-RTT 连接**：
        - 再次连接时，不需要握手，直接发数据（利用之前的 Session Ticket）。
    - **网络切换平滑 (Connection Migration)**：
        - TCP 基于 `IP + 端口` 识别连接。你从 Wi-Fi 切到 4G，IP 变了，连接必须断开重连。
        - QUIC 基于 **Connection ID**。只要 ID 没变，IP 变了也没事，连接保持不断，下载/视频不会卡顿。

## 9. CDN (内容分发网络)

把静态资源（图片、JS、CSS）推送到离用户最近的边缘节点。

- **原理**：
    1.  用户请求 `cdn.example.com/main.js`。
    2.  DNS 解析到最近的 CDN 节点（比如北京节点）。
    3.  **命中**：该节点有缓存，直接返回（极快）。
    4.  **未命中 (回源)**：节点去源站（你的服务器）拉取文件，缓存下来，再返回给用户。
- **最佳实践**：
    - **动静分离**：API 接口走自己的服务器，静态资源走 CDN。
    - **强缓存配合**：CDN 上的资源必须设置超长强缓存（一年），文件名带 Hash。
    - **预热**：发布新版本时，主动让 CDN 把资源拉过去，防止瞬间回源把源站打挂。

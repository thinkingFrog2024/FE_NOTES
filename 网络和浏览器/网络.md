# 网络

###### http1.0 2.0

HTTP/1.0 默认使用**非持久连接**（`Connection: close`），即每个请求都需要建立新的 TCP 连接，请求完成后立即关闭连接。

- **问题**：建立 TCP 连接需要经过三次握手，频繁创建和销毁连接会带来显著的**连接延迟**（特别是在高延迟网络中）。
- **改进**：1.1 引入 `Connection: keep-alive` 实现持久连接，但仍受限于**队头阻塞**（Head-of-Line Blocking）。

HTTP/2.0：多路复用（Multiplexing）

HTTP/2.0 基于**二进制分帧**（Binary Framing）实现**多路复用**，所有请求共享同一个 TCP 连接：

- 每个请求被拆分为多个**二进制帧**（Frame），如 `HEADERS`（头部）和 `DATA`（数据）帧。
- 这些帧可以**并行传输**，服务器根据帧的标识符（Stream ID）重新组装请求和响应。
- **队头阻塞问题彻底解决**：一个请求的延迟不会影响其他请求的处理。

##### 2. 传输格式：文本 vs 二进制

HTTP/1.0 的请求和响应是**纯文本格式**，通过换行符分隔头部和主体，例如：

```http
GET /index.html HTTP/1.0
User-Agent: Mozilla/5.0
Accept: text/html

<!DOCTYPE html>
<html>...</html>
```

- **问题**：文本格式解析效率低，且容易因空格、大小写等问题导致解析错误。

HTTP/2.0 将请求和响应**二进制化**，拆分为更小的帧（Frame）和流（Stream）：

- **帧**：最小的数据单位，包含头部（如 Stream ID）和数据。
- **流**：多个帧组成的逻辑数据流，对应一个 HTTP 请求 / 响应。
- **二进制分帧层**：位于应用层和传输层之间，负责帧的封装、传输和重组。

HTTP/1.0 的请求和响应头部是**明文传输**，且每次请求都会重复发送相同的头部（如 `User-Agent`、`Cookie` 等）。

- **问题**：头部数据通常较大（特别是 Cookie），会浪费大量带宽，增加延迟。

HTTP/2.0：HPACK 压缩

HTTPS 通过 **SSL/TLS 协议** 对 HTTP 数据进行加密，确保数据的 **机密性**、**完整性** 和 **身份验证**：

HTTP 协议的请求和响应数据是**明文传输**的，不进行任何加密处理。

HTTP 协议默认使用 TCP 端口 `80` 进行通信。

HTTPS 协议默认使用 TCP 端口 `443` 进行通信，因为 SSL/TLS 握手和加密数据传输需要额外的处理

HTTP 协议无法验证服务器或客户端的真实身份，攻击者可轻松伪装成合法服务器（钓鱼攻击）

HTTPS 通过 **数字证书** 验证服务器（或客户端）的身份：

HTTPS：SEO 优势 + 浏览器信任

- **SEO 优势**：Google 自 2014 年起将 HTTPS 列为排名因素之一，HTTPS 网站在搜索结果中更易获得优先展示。
- **浏览器信任**：现代浏览器（Chrome、Firefox 等）对 HTTP 网站的表单提交、Cookie 等功能施加限制，并在地址栏明确标记 “不安全”；HTTPS 网站则显示 “安全” 或锁图标，提升用户信任。
- **强制 HTTPS**：Chrome 等浏览器已对敏感操作（如支付、登录）强制要求 HTTPS，且未来可能逐步淘汰 HTTP。





###### 强缓存 协商缓存 判断逻辑

1. **首次请求资源**：
   - 服务器返回资源和缓存头（如 `Cache-Control`、`ETag`）。
   - 浏览器将资源和缓存头存入本地缓存。
2. **再次请求资源**：
3. **检查强缓存**：
   1. `Expires`（HTTP/1.0）
   2. `Cache-Control`（HTTP/1.1，优先级高于 `Expires``Cache-Control` 是一个 **相对时间**（单位：秒），表示资源的有效时间，如 `max-age=3600` 表示资源 1 小时内有效。支持多种指令）
      1. `max-age=3600`：资源在 3600 秒内有效，优先于 `Expires`。
      2. `no-cache`：**禁用强缓存**，但允许协商缓存（需与服务器验证）。
      3. `no-store`：**禁用所有缓存**，每次请求都必须重新获取资源。
      4. `private`：资源仅允许浏览器缓存，不允许 CDN 等中间代理缓存。
   3. 若命中强缓存（资源未过期），直接从本地缓存读取，**不发送请求到服务器**。
   4. 若强缓存过期（或不存在），则进入下一步。
4. **检查协商缓存**：
   - 浏览器发送请求到服务器，携带缓存标识（如 `ETag`、`Last-Modified`）。
   - 服务器根据缓存标识判断资源是否有更新：
     - 若资源未更新，返回 `304 Not Modified`，浏览器从本地缓存读取资源。
     - 若资源已更新，返回 `200 OK` 和新的资源内容，同时更新缓存标识。



###### Get Post的区别

| **对比维度**     | **Get 请求**                             | **Post 请求**                      |
| ---------------- | ---------------------------------------- | ---------------------------------- |
| **URL 可见性**   | 参数附在 URL 后（如 `?key=value`），可见 | 参数在请求体中，URL 不体现         |
| **数据传输位置** | URL 参数部分                             | 请求体（Request Body）             |
| **数据长度限制** | 受限于浏览器 / 服务器（通常几 KB）       | 无固定限制，可传输大量数据         |
| **安全性**       | 较低（参数可见，可能被缓存 / 篡改）      | 较高（参数不暴露在 URL 中）        |
| **幂等性**       | 多次请求结果相同（幂等）                 | 多次请求可能产生不同结果（非幂等） |
| **缓存支持**     | 浏览器可缓存                             | 默认可缓存，但需显式设置           |
| **用途场景**     | 获取资源（如查询数据）                   | 提交数据（如表单提交、文件上传）   |

为什么post会请求两次？

跨域请求（CORS）时，浏览器会先发送 `OPTIONS` 请求进行预检。

- 浏览器需确认目标服务器是否允许当前域名的跨域请求。
- 预检请求会检查 `Access-Control-Allow-Methods` 等响应头。

###### 为什么需要tcp三次链接

TCP 是面向连接的协议，在传输数据前必须先建立连接，确保双方：



- **双向通信能力**：确认双方都能发送和接收数据。
- **初始序列号同步**：协商初始序列号（ISN），用于后续数据传输的序号管理（防止旧数据包干扰）。
- **资源分配**：双方为连接分配缓冲区、状态表等资源。

1. **第一次握手：客户端发送 SYN 包（客户端 → 服务器）**SYN = 1（表示请求建立连接）

2. ####  **第二次握手：服务器发送 SYN+ACK 包（服务器 → 客户端）SYN = 1（表示同意建立连接）。**ACK = 1（确认收到客户端的 SYN

3. #### **第三次握手：客户端发送 ACK 包（客户端 → 服务器）**

###### 为什么四次挥手

- **双向关闭确认**：确保双方都明确对方已完成数据发送，避免单方面断开导致数据丢失。
- **数据完整性**：保证所有已发送的数据都被接收方确认，未发送完的数据有机会传输完毕。
- **资源释放**：双方释放连接占用的缓冲区、状态表等资源。

1.  **第一次挥手：客户端发送 FIN 包（客户端 → 服务器）** FIN = 1（表示请求关闭客户端到服务器的数据流）。

2. #### **第二次挥手：服务器发送 ACK 包（服务器 → 客户端）**

3. #### **第三次挥手：服务器发送 FIN 包（服务器 → 客户端）**

4. #### **第四次挥手：客户端发送 ACK 包（客户端 → 服务器）**

###### Xss攻击

XSS（Cross-Site Scripting，跨站脚本攻击）是一种常见的 Web 安全漏洞，攻击者通过在网页中注入恶意脚本（通常是 JavaScript），让用户浏览器执行这些脚本，从而窃取用户数据、篡改页面内容或劫持用户会话。

原因：

1. Web 应用对用户输入（如表单、URL 参数）未做充分过滤或转义，导致恶意脚本被当作正常内容执行。
   1. 攻击者构造包含恶意脚本的输入（如评论、URL）。
   2. 用户访问包含该恶意脚本的页面，浏览器执行脚本。
   3. 脚本利用浏览器权限，获取用户信息（如 Cookie、会话 Token）或操作页面

类型：

1. #### **反射型 XSS（非持久化）**

   1. 恶意脚本存在于 URL 或表单中，用户访问时脚本被反射到页面并执行，不存储在服务器。

      ```
      // 恶意 URL  
      https://example.com/search?q=<script>alert('XSS');</script>  
      
      用户点击该链接时，search 参数中的脚本被浏览器执行，弹出警告框。
      ```

      

2. #### **存储型 XSS（持久化）**

   1. 恶意脚本被存储在服务器（如数据库、评论区），用户访问页面时脚本自动加载执行。

      ```
      <p>这条评论很棒！<script>document.cookie = 'hacked=true';</script></p>  
      ```

      

3. #### **DOM 型 XSS**

   ```
   // 错误写法：直接将 URL 参数插入 DOM  
   document.getElementById('search').innerHTML = location.hash.substr(1);  
   ```

   

###### 状态码

- **1xx（信息类）**：表示请求已被接收，需要继续处理
- **2xx（成功类）**：表示请求已成功被服务器接收、理解并接受
- **3xx（重定向类）**：表示需要客户端采取进一步的操作才能完成请求
- **4xx（客户端错误类）**：表示客户端可能有错误，请求无法完成
- **5xx（服务器错误类）**：表示服务器在处理请求时发生了错误



- **100 Continue**：客户端应当继续发送请求。这个临时响应是用来通知客户端它的**部分请求已经被服务器接收，且仍未被拒绝。客户端应当继续发送请求的剩余部分，**或者如果请求已经完成，忽略这个响应。

- **200 OK**：请求成功，服务器已成功处理请求并返回数据。这是最常见的成功状态码。

- **201 Created**：**请求成功，且服务器已创建了新的资源**。通常用于 POST 请求后，返回新创建资源的 URI。

- **202 Accepted**：**服务器已接受请求，但尚未处理。这个状态码通常用于异步处理的情况**，客户端可以通过轮询或其他方式获取处理结果。

- **204 No Content**：**请求成功，但响应中没有返回任何内容。**通常用于只需要执行操作而不需要返回数据的情况，如 DELETE 请求。

- **301 Moved Permanently**：请求的资源已永久移动到新的 URI，并在响应头的 `Location` 字段中指定新的 URL。客户端收到 301 后，会自动跳转到新 URL，并在未来的请求中 **直接使用新 URL**（除非缓存被清除）。

- **302 Found**：请求的资源临时移动到新的 URI，客户端应使用新的 URI 进行本次请求，但未来的请求仍应使用原 URI。搜索引擎不会更新链接。不会缓存

- 常用于 POST/PUT 请求后，引导客户端访问新创建或更新的资源。

- **304 Not Modified**：客户端发送了一个带有条件的请求（如 If-Modified-Since 或 If-None-Match），服务器告诉客户端，请求的资源自上次请求以来没有发生变化，**可以使用缓存的版本。**

  。

- **400 Bad Request**：客户端发送的请求语法错误，服务器无法理解。通常是由于请求参数格式不正确或缺少必要的参数导致的。

- **401 Unauthorized**：**请求需要用户验证**。客户端需要提供有效的身份验证信息才能访问该资源。这个状态码通常用于需要登录的页面或 API。

- **403 Forbidden**：**服务器理解请求客户端的请求，但是拒绝执行此请求**。与 401 不同，这个状态码表示客户端的身份已经通过验证，但没有权限访问该资源。

- **404 Not Found**：请求的资源不存在。**客户端请求的 URI 在服务器上没有找到对应的资源**。这是最常见的客户端错误状态码。

- **410 Gone**：**请求的资源已永久删除，且不会再得到的**。与 404 不同，这个状态码表示资源曾经存在，但现在已经永久删除，并且没有重定向地址。。

- **500 Internal Server Error**：服务器内部错误，无法完成请求。这是最常见的服务器错误状态码，表示服务器在处理请求时遇到了意外情况。

- **502 Bad Gateway**：**作为网关或者代理工作的服务器尝试执行请求时，从上游服务器接收到无效的响应**。这通常表示服务器作为代理或网关时，从后端服务器收到了错误响应。

###### 网络请求·

从复杂的回调（XHR）到简洁的 Promise（Fetch），再到功能全面的第三方库（Axios）。现代项目中，Axios 因其错误处理、拦截器、取消请求等特性成为主流选择

XHR 最早的ajax技术 基于 **回调函数的写法** 通过 **onreadyStateChange**监听状态改变 需要通过readyState status 手动处理状态 

支持上传，下载进度 需要手动调用JSON.parse 

```javascript
const xhr = new XMLHttpRequest();
xhr.open('GET', 'https://api.example.com/data', true); // 异步请求
xhr.onreadystatechange = () => {
  if (xhr.readyState === 4) { // 请求完成
    if (xhr.status === 200) { // 成功
      console.log('响应数据:', JSON.parse(xhr.responseText));
    } else {
      console.error('请求失败:', xhr.status);
    }
  }
};
xhr.onerror = () => console.error('网络错误');
xhr.send(); // 发送请求
```



fetch 基于promise 

- **基于 Promise**：避免回调地狱，，天然支持链式调用和异步操作
- **模块化设计**：请求（Request）、响应（Response）、头信息（Headers）等分离。
- **功能完整**：支持请求体（body）、自定义头、跨域、上传下载等。
- 返回的`Response`对象是一个 **数据流**（Stream）`fetch` 返回的 `Response` 对象本质上是一个 **HTTP 响应的数据流**（类似水管中的水，需要通过 “水龙头” 放出）。服务器返回的数据不会一次性加载到内存中，而是以流的形式传输，需要通过 `Response` 的方法（如 `json()`、`text()`、`blob()`）手动解析后才能获取实际数据每个解析方法（如 `res.json()`）都会返回一个新的 `Promise`，因为解析大文件或复杂数据可能需要时间。这意味着 `fetch` 的数据处理是**分两步的异步操作**：当 `XHR` 请求完成（`readyState === 4`），`response` 属性已包含完整数据，可直接使用（无需等待额外的解析步骤）。这意味着 `XHR` 的数据处理是**一步到位的同步操作**（数据接收和解析由浏览器自动完成）。     
  - 第一步：`fetch(url)` 返回 `Promise<Response>`（此时仅获取响应头，数据体尚未完全接收）。
  - 第二步：调用 `res.json()` 或 `res.blob()` 等方法，返回 `Promise<实际数据>`（等待数据体接收并解析完成）。

```
// 基本 GET 请求
fetch('https://api.example.com/data')
  .then((response) => {
    if (!response.ok) throw new Error(`HTTP 错误: ${response.status}`);
    return response.json(); // 解析 JSON
  })
  .then((data) => console.log('响应数据:', data))
  .catch((err) => console.error('请求失败:', err));

// 带请求体的 POST 请求
fetch('https://api.example.com/submit', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json' // 指定请求体类型
  },
  body: JSON.stringify({ name: '张三', age: 25 }) // JSON 请求体
})
  .then((res) => res.json())
  .then((data) => console.log('提交结果:', data));
```



Axios

axios内部是通过xhr实现的 设计初衷是提供跨平台的统一网络请求api  内部使用原生的XMLHttpRequest对象发送请求 并且使用promise包装：

```javascript
// Axios 源码片段（简化版）
module.exports = function xhrAdapter(config) {
  return new Promise(function dispatchXhrRequest(resolve, reject) {
    const request = new XMLHttpRequest(); // 创建 XHR 实例
    request.open(config.method, url, true);
    request.onreadystatechange = function handleLoad() { /* 处理响应 */ };
    request.send(config.data); // 发送请求
  });
};
```

在nodejs里 Axios依赖内置的http https发送请求：

```javascript
// Axios 源码片段（简化版）
module.exports = function httpAdapter(config) {
  return new Promise(function dispatchHttpRequest(resolve, reject) {
    const lib = config.url.startsWith('https') ? require('https') : require('http');
    const request = lib.request(options, function handleResponse(response) { /* 处理响应 */ });
    request.end(config.data); // 发送请求
  });
};
```

- Fetch API 存在一些设计问题，Axios 通过封装 XHR 规避了这些问题：

  

  - **错误处理**：Fetch 仅在网络错误时触发 `catch`，HTTP 错误（如 404、500）仍视为 “成功”，需手动检查 `response.ok`；而 Axios 统一在 HTTP 状态非 2xx 时触发 `catch`。
  - **取消请求**：Fetch 原生不支持取消（需借助 `AbortController`，且 IE 不支持）；Axios 通过 `CancelToken` 或 `AbortController` 提供更简洁的取消机制。
  - **进度监听**：Fetch 监听上传 / 下载进度需手动解析 `ReadableStream`，而 Axios 基于 XHR 内置了 `onUploadProgress` 和 `onDownloadProgress`。

- **自动处理错误**：HTTP 错误（如 404）直接触发 `catch`，无需手动检查状态码。

- **支持取消请求**：通过 `CancelToken` 或 `AbortController` 取消未完成的请求。

- **进度监听**：内置上传 / 下载进度回调。

- **拦截器**：支持请求前和响应后的拦截处理（如添加认证头、统一错误处理）。

- **兼容性好**：内部封装 XHR，支持浏览器和 Node.js。

```javascript
// 基本 GET 请求
axios.get('https://api.example.com/data')
  .then((response) => console.log('响应数据:', response.data)) // 自动解析 JSON
  .catch((error) => {
    if (error.response) {
      console.error('HTTP 错误:', error.response.status); // 404、500 等
    } else if (error.request) {
      console.error('请求发送成功，但无响应');
    } else {
      console.error('请求错误:', error.message); // 网络错误
    }
  });

// 带请求体的 POST 请求
axios.post('https://api.example.com/submit', {
  name: '张三',
  age: 25
})
  .then((res) => console.log('提交结果:', res.data))
  .catch((err) => console.error('提交失败:', err));

// 取消请求
const source = axios.CancelToken.source();
axios.get('https://api.example.com/data', {
  cancelToken: source.token
})
  .catch((thrown) => {
    if (axios.isCancel(thrown)) {
      console.log('请求被取消:', thrown.message);
    }
  });
source.cancel('用户取消请求'); // 主动取消

// 拦截器示例（添加认证头）
axios.interceptors.request.use(
  (config) => {
    const token = localStorage.getItem('token');
    if (token) config.headers.Authorization = `Bearer ${token}`;

  (error) => Promise.reject(error)
);

  onUploadProgress: (progressEvent) => {
    // 计算上传百分比
    const percentCompleted = Math.round((progressEvent.loaded * 100) / progressEvent.total);
    console.log(`上传进度: ${percentCompleted}%`);
    
    // 更新 UI（如进度条）
    document.getElementById('upload-progress').value = percentCompleted;
  }
})
.then((response) => console.log('上传成功:', response))
.catch((error) => console.error('上传失败:', error));
```

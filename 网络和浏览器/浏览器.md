# 浏览器

#### DNS查询

把复杂的服务器ip映射成简单的域名 通过域名找到对应的ip 在找到对应的资源

1. 浏览器自身的DNS
2. 操作系统的
3. 本地host文件
4. 向域名服务器发送请求
   1. 本地DNS服务器
   2. 根域名服务器
   3. 顶级域名服务器
   4. 权威域名服务器

哪两种情况会触发options请求？

1. 遇到跨域问题 浏览器会自动发送一个options请求进行预检 判断请求能否成功
2. 自定义请求头

服务器缓存

- 强缓存：让浏览器强制缓存服务端的资源  在请求头中配置`Cache-Control:max-age=1000000` （优先级更高）或 `Expires:Wed,21 Oct 2015 07：28：00 GMT `   设置缓存之后再请求静态资源 第一次经过服务端 之后直接由浏览器返回 两种缓存方式：form disk cache 硬盘缓存 from memory cache 内存缓存  在状态码后会标明 在一次刷新之后时会有概率触发内存缓存
- 协商缓存  没听懂 就是如果资源没有更新返回304 响应体空 如果改变返回200 响应体带上最新资源

1. 后台会在响应头里带上两个字段：`Last-Modified` `If-Modified-Since`或`Etag` 服务器会对比Last-Modified和If-Modified-Since 如果值相等 代表资源没有改变   Etag（对比两次的ETag）可以是文件哈希值或版本号

### HTML页面渲染

HTML解析器会把超文本和标签解析成dom树（进行AST抽象语法树）css渲染引擎会把css样式表转换层浏览器可以理解的styleSheets 计算dom节点样式 这个叫cssom构建过程

### 回流

浏览器重新渲染部分或全部文档的过程叫做回流

- 页面首次渲染
- 浏览器窗口大小改变
- 元素尺寸/位置改变
- 元素内容改变
- 字体大小改变
- 添加删除可见的dom元素
- 激活css伪类
- 查询某些属性/调用某些方法

### 重绘

元素样式改变并不影响他在文档流里的位置时 如颜色改变 浏览器会将新样式赋予给元素 重新绘制

##### 输入url到页面解析的过程

1. 域名解析
   1. 浏览器检查DNS缓存里面有没有域名对应的ip地址 有就直接用 没有就向DNS服务器发送查询 这里的缓存会先检查浏览器 再检查系统缓存 再看路由器缓存  互联网服务提供商）的 DNS 服务器发送请求。ISP 的 DNS 服务器会进行递归搜索，依次向根域名服务器、顶级域名服务器、权威域名服务器等发起请求
   2. 这里可以使用`<link rel="dns-prefetch">`提前解析域名；静态资源通过 CDN 域名解析，指向离用户更近的节点。
2. 建立TCP连接
   1. 浏览器服务器三次握手建立tcp连接 如果访问的是https网站 那么还要进行TLS/SSL握手 TLS客户端还需要发送支持的加密算法列表 服务器返回选择的算法 服务器证书 客户端生成预主密钥 用服务端公钥加密之后发送 双方协商生成会话秘钥
   2. 如果长时间没有交互 会四次挥手断开连接
3. 发送请求
   1. 服务器接收浏览器发送的请求报文 解析信息 把处理结果封装成https报文返回给浏览器
4. 接收响应
   1. 浏览器接收响应报文 并且解析
5. 渲染页面
   1. 解析HTML 成DOM树
   2. 解析css 构建cssDOM树
   3. 构建渲染树
   4. 布局绘制（布局阶段进行重排 绘制进行重绘）
   5. 执行脚本
6. 页面显示

###### 为什么事件循环和js引擎都是单线程

JavaScript 诞生于 1995 年，最初是为 **浏览器环境** 设计的脚本语言，用于实现页面的交互逻辑（如表单验证、DOM 操作）。在这个场景下，**单线程是最合理的选择**，

**避免 DOM 操作冲突**：
浏览器的核心是 DOM（文档对象模型），它是页面的树形结构。如果 JavaScript 是多线程的，多个线程同时修改 DOM（如一个线程删除元素，另一个线程修改元素），会导致 DOM 状态不一致，产生复杂的竞态条件（Race Condition）。

多线程确实能提高程序的并发能力（如同时处理多个任务），但也带来了 **线程同步** 的复杂性：

- **锁机制**：为避免共享资源冲突，需要使用锁（如互斥锁）来控制对资源的访问，但锁会导致死锁（Deadlock）问题（例如线程 A 持有锁 1 并等待锁 2，线程 B 持有锁 2 并等待锁 1）。
- **上下文切换开销**：线程切换需要保存和恢复上下文（如寄存器状态、内存栈），频繁切换会降低性能。

对于 JavaScript 的主要应用场景（浏览器交互），单线程的简单性和确定性远大于多线程的并发优势

单线程模型的最大限制是 **阻塞**：如果一个任务耗时过长（如网络请求、文件读取），后续任务必须等待，导致页面卡顿（如按钮无法点击、动画停滞）。

为解决这个问题，JavaScript 引入了 **事件循环（Event Loop）** 机制，配合 **任务队列（Task Queue）** 和 **异步 API**（如 `setTimeout`、`fetch`、DOM 事件），实现了 **非阻塞异步编程**

- 当遇到异步操作（如 `setTimeout`、网络请求）时，JavaScript 引擎会将其交给 **浏览器的 Web API**（如定时器线程、网络线程）处理，自身继续执行后续代码。
- 异步操作完成后（如定时器到期、网络请求返回），Web API 会将对应的回调函数放入 **任务队列**。
- **事件循环** 不断从任务队列中取出回调函数，放入 **调用栈（Call Stack）** 执行。

**浏览器的缓存**

<br />

浏览器的缓存有以下几种：

1. Cache Storage（Service Worker 缓存）
2. localStorage
3. sessionStorage
4. IndexedDB
5. Cookie
6. 内存变量

其中只有 sessionStorage 关闭标签页后会丢失

### 各种缓存的详细对比

| 存储方式 | 存储位置 | 容量 | 关闭浏览器 | 过期时间 | 适用场景 |
|---------|---------|------|----------|---------|---------|
| **Cache Storage** | 硬盘 | ~几百MB~几GB | ✅ 不丢失 | 手动删除/代码删除 | 离线资源、API响应、静态文件 |
| **localStorage** | 硬盘 | 5~10MB | ✅ 不丢失 | 永久（手动删） | 用户偏好设置、主题、Token |
| **sessionStorage** | 内存 | 5~10MB | ❌ **丢失** | 关闭标签页即失效 | 表单临时数据、页面间临时传参 |
| **IndexedDB** | 硬盘 | ~几百MB~无上限 | ✅ 不丢失 | 永久（手动删） | 大量结构化数据、图片、文件 |
| **Cookie** | 硬盘 | 4KB/条 | ⚠️ 看过期时间 | 可设置Expires/Max-Age | 会话状态、CSRF Token |
| **内存变量** | 内存 | 受限于内存 | ❌ **丢失** | 页面刷新即失效 | 当前页面计算结果、临时状态 |

#### 1. Cache Storage（Service Worker 缓存）

**存储位置：** 硬盘（持久化）

**特点：**
- 由 Service Worker 管理，可拦截网络请求并返回缓存
- 支持请求级别的精细控制（按 URL 匹配）
- 域名隔离，不同网站的缓存互不干扰

**使用场景：**
- 离线访问 PWA 应用
- 静态资源缓存（JS/CSS/图片）
- API 接口响应缓存
- 大文件预加载

**基本用法：**

```javascript
// 存入缓存
caches.open('my-cache').then(cache => {
    cache.put('/api/data', new Response(JSON.stringify({ name: 'test' })));
});

// 读取缓存
caches.open('my-cache').then(cache => {
    cache.match('/api/data').then(response => response.json());
});

// 删除缓存
caches.delete('my-cache');
```

**缓存策略：**
- **Cache First：** 先查缓存，没有再请求网络（适合静态资源）
- **Network First：** 先请求网络，失败再用缓存（适合 API 数据）
- **Stale While Revalidate：** 先返回缓存，同时后台更新（兼顾速度和新鲜度）
- **Cache Only：** 只用缓存（纯离线场景）
- **Network Only：** 只走网络（支付等实时性要求高的场景）

#### 2. localStorage

**存储位置：** 硬盘（持久化）

**特点：**
- 只能存储字符串（需要 JSON 序列化/反序列化）
- 同源策略（相同域名+端口共享）
- 同步操作（大量数据可能阻塞主线程）

**容量限制：** 5~10MB（因浏览器而异）

**使用场景：**
- 用户偏好设置（主题色、语言）
- 已读文章标记
- 轻量级 Token 存储
- UI 状态折叠/展开

```javascript
// 存
localStorage.setItem('theme', 'dark');
localStorage.setItem('user', JSON.stringify({ name: '张三' }));

// 取
const theme = localStorage.getItem('theme'); // "dark"
const user = JSON.parse(localStorage.getItem('user'));

// 删
localStorage.removeItem('theme');
localStorage.clear(); // 清空全部

// 监听变化（其他标签页修改时会触发）
window.addEventListener('storage', (e) => {
    console.log(`${e.key} 被改变了: ${e.oldValue} → ${e.newValue}`);
});
```

#### 3. sessionStorage

**存储位置：** 内存

**特点：**
- 与 localStorage API 相同
- 但仅在当前标签页有效
- 关闭标签页即清除（不是关闭浏览器！）

**使用场景：**
- 表单填写中的临时数据（防止误关丢失）
- 页面间跳转的临时参数
- 敏感数据（用完即丢，不持久化）
- 多步骤向导的中间状态

```javascript
// 存
sessionStorage.setItem('step1Data', JSON.stringify({ name: 'xxx' }));

// 取
const data = JSON.parse(sessionStorage.getItem('step1Data'));

// 注意：新标签页打开同一页面，sessionStorage 是独立的！
// 只有通过链接/a/window.open 打开的页面才能共享
```

#### 4. IndexedDB

**存储位置：** 硬盘（持久化）

**特点：**
- 支持结构化数据（对象、数组、Blob、File）
- 异步操作（不阻塞主线程）
- 支持索引和事务（类数据库操作）
- 容量远大于 localStorage

**容量限制：** 通常可用磁盘空间的 50%（可达数百 MB 到 GB 级别）

**使用场景：**
- 离线存储大量数据（邮件客户端、文档编辑器）
- 图片/视频/文件的本地缓存
- 游戏存档
- 复杂查询的数据集

```javascript
// 打开/创建数据库
const request = indexedDB.open('MyDatabase', 1);

request.onupgradeneeded = (e) => {
    const db = e.target.result;
    // 创建对象仓库（类似表）
    if (!db.objectStoreNames.contains('users')) {
        db.createObjectStore('users', { keyPath: 'id' });
    }
};

request.onsuccess = (e) => {
    const db = e.target.result;

    // 写入数据（事务操作）
    const tx = db.transaction(['users'], 'readwrite');
    const store = tx.objectStore('users');
    store.put({ id: 1, name: '张三', age: 25 });

    // 读取数据
    const getReq = store.get(1);
    getReq.onsuccess = () => console.log(getReq.result);
};
```

**推荐库：** `Dexie.js`（简化 IndexedDB 操作）

```javascript
import Dexie from 'dexie';

const db = new Dexie('MyDatabase');
db.version(1).stores({ friends: '++id, name, age' });

// 使用（像 ORM 一样简单）
await db.friends.add({ name: '李四', age: 30 });
const allFriends = await db.friends.toArray();
```

#### 5. Cookie

**存储位置：** 硬盘

**特点：**
- 每次 HTTP 请求自动携带（包括静态资源请求）
- 有域名和路径限制
- 可设置 HttpOnly（防 XSS）、Secure（仅 HTTPS）、SameSite（防 CSRF）

**容量限制：** 单条最大 4KB，每个域名约 50 条左右

**使用场景：**
- 会话管理（Session ID）
- 记住登录状态
- CSRF Token
- 广告追踪（第三方 Cookie）

```javascript
// 设置 Cookie
document.cookie = 'username=张三; path=/; max-age=3600; SameSite=Strict';

// 读取 Cookie（只能读取非 HttpOnly 的）
console.log(document.cookie); // "username=张三; theme=dark"

// 删除（设置过期时间为过去）
document.cookie = 'username=; expires=Thu, 01 Jan 1970 00:00:00 GMT';
```

**属性说明：**
- `Expires / Max-Age`：过期时间
- `Path`：生效路径
- `Domain`：生效域名
- `Secure`：仅 HTTPS 传输
- `HttpOnly`：禁止 JS 访问（防 XSS 窃取）
- `SameSite`：跨站请求控制（Strict/Lax/None）

---

#### 6. 内存变量

**存储位置：** 内存（RAM）

**特点：**
- 速度最快（纳秒级）
- 页面刷新/关闭即丢失
- 不占用存储空间配额

**使用场景：**
- 当前页面的计算中间值
- 组件内部状态（React/Vue 的 state）
- 事件回调中的临时变量
- 循环/递归中的累加器

```javascript
// 这些都在内存中
let count = 0;
const tempData = [];
const userInfo = fetchUserInfo(); // Promise 对象也在内存

// 刷新页面 → 全部消失
```

### 如何选择？

| 需求 | 推荐 |
|-----|------|
| 离线支持/PWA | **Cache Storage** |
| 少量配置数据 | **localStorage** |
| 临时表单数据 | **sessionStorage** |
| 大量结构化数据 | **IndexedDB** |
| 需要发给服务端 | **Cookie** |
| 当前页面计算 | **内存变量** |

## 骨架屏（Skeleton Screen）

### 骨架屏 vs Loading 动画

两者都是组件，但用户体验完全不同：

**Loading 动画（转圈圈）：**
```
┌─────────────────────────┐
│                         │
│         ⏳              │  用户不知道要等多久
│      加载中...          │  不知道加载出来长什么样
│                         │  感觉时间过得很慢
└─────────────────────────┘
```

**骨架屏（Skeleton）：**
```
┌─────────────────────────┐
│ ┌──────┐               │
│ │ ████ │  标题区域       │  用户一眼看出页面结构
│ └──────┘               │
│ ████████████████████    │  能感知到布局
│ ████████████           │
│ ┌───┐ ┌───┐ ┌───┐     │
│ │ █ │ │ █ │ │ █ │     │  感觉加载更快
│ └───┘ └───┘ └───┘     │
└─────────────────────────┘
```

### 为什么骨架屏感觉更快？

**感知性能 vs 实际性能：**
- 实际等待时间相同（都是 2 秒）
- Loading 动画：用户心理"还在转...好慢..." → 感知 ~3~4 秒
- 骨架屏：用户心理"页面出来了！正在填充内容" → 感知 ~1.5~2 秒

**核心原理：骨架屏让用户看到页面框架，大脑预判布局，减少焦虑感，视觉闪烁更小（平滑过渡）**

| | Loading 动画 | 骨架屏 |
|---|---|---|
| 信息量 | 零 | 有（能看到页面结构） |
| 用户焦虑 | 高 | 低 |
| 视觉闪烁 | 大（空白→突然出现） | 小（平滑过渡） |

### Vue 骨架屏实现

```vue
<template>
    <div class="skeleton">
        <div class="skeleton-avatar"></div>
        <div class="skeleton-line"></div>
        <div class="skeleton-line short"></div>
    </div>
</template>

<style scoped>
.skeleton {
    padding: 16px;
    animation: pulse 1.5s ease-in-out infinite;
}
.skeleton-avatar {
    width: 48px; height: 48px; border-radius: 50%;
    background: linear-gradient(90deg, #f0f0f0 25%, #e0e0e0 50%, #f0f0f0 75%);
}
.skeleton-line {
    height: 16px; margin-top: 12px; border-radius: 4px;
    background: linear-gradient(90deg, #f0f0f0 25%, #e0e0e0 50%, #f0f0f0 75%);
}
.skeleton-line.short { width: 60%; }
@keyframes pulse {
    0% { background-position: 200% 0; }
    100% { background-position: -200% 0; }
}
</style>
```

### 配合异步组件使用

```javascript
import { defineAsyncComponent } from 'vue';

const HeavyComponent = defineAsyncComponent({
    loader: () => import('./components/HeavyComponent.vue'),
    loadingComponent: SkeletonLoader,   // 加载时显示骨架屏
    delay: 300,                          // 300ms 内完成不显示（避免闪烁）
});
```

## Web Worker（多线程）

### 是什么？

Web Worker 是浏览器提供的**独立线程**，用于在后台执行重型计算，不阻塞主线程（UI 线程）。

### 为什么需要？

**传统单线程问题：**

```
主线程（UI线程）：渲染页面 + 执行JS + 处理事件 全部排队
→ 如果 JS 计算很重 → 页面卡死（按钮点不了、动画停了）
```

**Web Worker 解决方案：**

```
┌──────────────────┐    ┌──────────────────┐
│    主线程         │    │   Web Worker     │
│                  │    │                  │
│  渲染页面 ✅      │◀──▶│  重型计算 ✅      │
│  处理交互 ✅      │    │  解析大文件 ✅    │
│  响应用户 ✅      │    │  加密解密 ✅     │
│  不卡了！         │    │  图像处理 ✅      │
└──────────────────┘    └──────────────────┘
        ↕                      ↕
        └──── postMessage 通信 ─┘
```

### 基本用法

```javascript
// main.js - 主线程
const worker = new Worker('worker.js');

// 发送数据给 Worker
worker.postMessage({ type: 'start', data: bigArray });

// 接收 Worker 返回的结果
worker.onmessage = function(e) {
    console.log('计算完成:', e.data);
};

// 接收错误
worker.onerror = function(e) {
    console.error('Worker 出错:', e.message);
};
```

```javascript
// worker.js - Worker 线程（独立文件）
self.onmessage = function(e) {
    const { type, data } = e.data;
    
    if (type === 'start') {
        const result = heavyCalculation(data); // 重型计算不阻塞主线程
        self.postMessage({ result });          // 结果发回主线程
    }
};
```

### 关键限制

| 可以做 ❌ 不可以做 |
|-------|
| ✅ 计算/算法、XMLHttpRequest/fetch、IndexedDB、setTimeout/setInterval | ❌ 操作 DOM、使用 window/document、使用 parent/frame、直接访问主线程变量 |

Worker 里没有 DOM，没有 window 对象，只能通过 `postMessage` 和主线程通信。

### 适用场景

- 大数据处理（百万条数据计算）
- 图片处理（滤镜、压缩、裁剪）
- 加密解密
- 语法高亮（代码编辑器）
- WebAssembly 的宿主
- 复杂的 JSON 解析

### 终止 Worker

```javascript
// 主线程终止 Worker
worker.terminate();

// Worker 自行关闭
self.close();
```

#### DNS查询

把复杂的服务器ip映射成简单的域名 通过域名找到对应的ip 在找到对应的资源

1. 浏览器自身的DNS
2. 操作系统的
3. 本地host文件
4. 向域名服务器发送请求
   1. 本地DNS服务器
   2. 根域名服务器
   3. 顶级域名服务器
   4. 权威域名服务器



哪两种情况会触发options请求？

1. 遇到跨域问题 浏览器会自动发送一个options请求进行预检 判断请求能否成功
2. 自定义请求头 



服务器缓存

- 强缓存：让浏览器强制缓存服务端的资源  在请求头中配置`Cache-Control:max-age=1000000` （优先级更高）或 `Expires:Wed,21 Oct 2015 07：28：00 GMT ` 设置缓存之后再请求静态资源 第一次经过服务端 之后直接由浏览器返回 两种缓存方式：form disk cache 硬盘缓存 from memory cache 内存缓存  在状态码后会标明 在一次刷新之后时会有概率触发内存缓存  
-  协商缓存  没听懂 就是如果资源没有更新返回304 响应体空 如果改变返回200 响应体带上最新资源
  1. 后台会在响应头里带上两个字段：`Last-Modified` `If-Modified-Since`或`Etag` 服务器会对比Last-Modified和If-Modified-Since 如果值相等 代表资源没有改变   Etag（对比两次的ETag）可以是文件哈希值或版本号

### HTML页面渲染

HTML解析器会把超文本和标签解析成dom树（进行AST抽象语法树）css渲染引擎会把css样式表转换层浏览器可以理解的styleSheets 计算dom节点样式 这个叫cssom构建过程



### 回流

浏览器重新渲染部分或全部文档的过程叫做回流

- 页面首次渲染
- 浏览器窗口大小改变
- 元素尺寸/位置改变
- 元素内容改变
- 字体大小改变
- 添加删除可见的dom元素
- 激活css伪类
- 查询某些属性/调用某些方法 

### 重绘

元素样式改变并不影响他在文档流里的位置时 如颜色改变 浏览器会将新样式赋予给元素 重新绘制


##### 输入url到页面解析的过程

1. 域名解析
   1. 浏览器检查DNS缓存里面有没有域名对应的ip地址 有就直接用 没有就向DNS服务器发送查询 这里的缓存会先检查浏览器 再检查系统缓存 再看路由器缓存  互联网服务提供商）的 DNS 服务器发送请求。ISP 的 DNS 服务器会进行递归搜索，依次向根域名服务器、顶级域名服务器、权威域名服务器等发起请求
   2. 这里可以使用`<link rel="dns-prefetch">`提前解析域名；静态资源通过 CDN 域名解析，指向离用户更近的节点。
2. 建立TCP连接
   1. 浏览器服务器三次握手建立tcp连接 如果访问的是https网站 那么还要进行TLS/SSL握手 TLS客户端还需要发送支持的加密算法列表 服务器返回选择的算法 服务器证书 客户端生成预主密钥 用服务端公钥加密之后发送 双方协商生成会话秘钥 
   2. 如果长时间没有交互 会四次挥手断开连接 
3. 发送请求
   1. 服务器接收浏览器发送的请求报文 解析信息 把处理结果封装成https报文返回给浏览器
4. 接收响应
   1. 浏览器接收响应报文 并且解析
5. 渲染页面
   1. 解析HTML 成DOM树
   2. 解析css 构建cssDOM树
   3. 构建渲染树
   4. 布局绘制（布局阶段进行重排 绘制进行重绘）
   5. 执行脚本
6. 页面显示





###### 为什么事件循环和js引擎都是单线程

JavaScript 诞生于 1995 年，最初是为 **浏览器环境** 设计的脚本语言，用于实现页面的交互逻辑（如表单验证、DOM 操作）。在这个场景下，**单线程是最合理的选择**，

**避免 DOM 操作冲突**：
浏览器的核心是 DOM（文档对象模型），它是页面的树形结构。如果 JavaScript 是多线程的，多个线程同时修改 DOM（如一个线程删除元素，另一个线程修改元素），会导致 DOM 状态不一致，产生复杂的竞态条件（Race Condition）。

多线程确实能提高程序的并发能力（如同时处理多个任务），但也带来了 **线程同步** 的复杂性：



- **锁机制**：为避免共享资源冲突，需要使用锁（如互斥锁）来控制对资源的访问，但锁会导致死锁（Deadlock）问题（例如线程 A 持有锁 1 并等待锁 2，线程 B 持有锁 2 并等待锁 1）。
- **上下文切换开销**：线程切换需要保存和恢复上下文（如寄存器状态、内存栈），频繁切换会降低性能。



对于 JavaScript 的主要应用场景（浏览器交互），单线程的简单性和确定性远大于多线程的并发优势

单线程模型的最大限制是 **阻塞**：如果一个任务耗时过长（如网络请求、文件读取），后续任务必须等待，导致页面卡顿（如按钮无法点击、动画停滞）。

为解决这个问题，JavaScript 引入了 **事件循环（Event Loop）** 机制，配合 **任务队列（Task Queue）** 和 **异步 API**（如 `setTimeout`、`fetch`、DOM 事件），实现了 **非阻塞异步编程**

- 当遇到异步操作（如 `setTimeout`、网络请求）时，JavaScript 引擎会将其交给 **浏览器的 Web API**（如定时器线程、网络线程）处理，自身继续执行后续代码。
- 异步操作完成后（如定时器到期、网络请求返回），Web API 会将对应的回调函数放入 **任务队列**。
- **事件循环** 不断从任务队列中取出回调函数，放入 **调用栈（Call Stack）** 执行。

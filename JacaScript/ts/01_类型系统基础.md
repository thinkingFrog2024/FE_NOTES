# 类型系统基础

## 结构化类型系统

整体上 typescript是一个结构化类型系统的语言 ，这使得我们可以只检查我们需要的属性方法等。
但是并不是全部 比如protect的属性 private的属性 这些属性还是名义化类型 ，对于这样的属性或者方法检查的时候 检查的不仅是是否存在 并且还要检查来自于哪个类
为什么类型{}框定格的是任何不是null 或者undefined的值？ 感觉实际上如果是这个类型 添加属性是会报错的。

### 判据
一般会认为 如果一个变量 是一个类型的子类型 那么这个值可以赋值给这个类型的变量
这其中涉及两个判据：可不可以赋值 是不是子类型
但是这两者并不是完全相等的 赋值在是子类型的基础上多了两条规则：any类型可以进行任何类型收束 就比如只要一个变量是any类型 可以访问这个变量上面的任何属性 都不会报错。 不同的枚举类型之间赋值规则基于代表的数字值。

ts的类型检查存在一下五种：
1. **Identity 相等**： 这是ts里面最严格的关系 代表严格相等 在结构 名称 泛型参数上都必须完全一致 。如果引用接口或者类型别名 必须是同一个 这里有一点名义化结构
```typescript
interface Point1 { x: number; y: number; }
interface Point2 { x: number; y: number; }
type T7 = Point1;
type T8 = Point2;
// T7 和 T8 结构相同，但非 Identity
// 因为它们源于不同的接口定义
```
2. **Strict Subtyping**： 仅次于identity 如果类型 A 是类型 B 的严格子类型，那么类型 A 必须至少拥有类型 B 的所有属性，且对应属性的类型也是严格子类型关系。此关系不允许出现 any 类型。
  a. A 和 B 都是对象类型。
  b. A 拥有 B 的所有属性。
  c. 对于 B 中的每一个属性 P，A 中对应的属性 P 的类型必须是 B 中属性 P 类型的严格子类型（递归定义）。
  d. 整个比较链条中不能出现 any。
3. **Subtyping**： 和上者比较 允许any的存在
4. **Assignability**： 在 Subtyping 的基础上，进一步放宽了对函数类型的处理，引入了“协变”和“逆变”的概念
5. **Comparability**： Comparability 是最宽松的类型关系，主要用于 ==、!=、===、!== 等比较操作符。它的目标是允许两种可能存在重叠的类型进行比较，而不是严格地限制它们

- Identity 是起点，要求类型完全一致。
- Strict Subtyping 放宽到结构兼容，开启了 TypeScript 结构化类型的大门。
- Subtyping 在结构兼容的基础上，为 any 打开了“绿色通道”，用于处理动态数据。
- Assignability 进一步为函数类型量身定制了更灵活的规则（逆变/协变），以支持高阶函数和回调等常见模式。
- Comparability 是最后的防线，只要两种类型有理论上重叠的可能，就允许比较，防止无意义的代码。


## 类型的来源

ts究竟是怎么知道一个变量的类型 ？
1. 手动标识： 我们平时使用：type 来表示一个变量类型 还可以使用类型断言 也就是as关键字 `<type>`前缀
2. JSDoc： ts可以读取这个Doc
3. 类型推导： 这个是ts借助很多的机制去推导一个变量的类型 但是这种机制也是很复杂的，在具体和模糊之间怎样选择

```typescript
declare function myApi(someTuple: [string, boolean]): void;

const firstTry = ['foo', false];
// 推导的类型为：(string | boolean)[]
myApi(firstTry); 
// 报错！
// Argument of type '(string | boolean)[]' is not assignable to parameter of type '[string, boolean]'.
//   Target requires 2 element(s) but source may have fewer. 

const secondTry = ['foo', false] as const;
// 推导的类型为：readonly ["foo", false]
myApi(secondTry); 
// 还是报错！
// Argument of type 'readonly ["foo", false]' is not assignable to parameter of type '[string, boolean]'.
//  The type 'readonly ["foo", false]' is 'readonly' and cannot be assigned to the mutable type '[string, boolean]'

// 这样才可以……
myApi(['foo', false]);
```
第一个方式 ts推导出来的类型台模糊 第二个方式 我们使用as const 申明 虽然类型对了 但是多了个readonly readonly的变量和普通的变量对比的范围更小 所以类型检查也不能通过（ as大法好）
只有我们直接传入元组字面量的水后 ts才会推断出我们想要的那个类型  实际上 把这个元组类型推断成我们期望的那个类型 是一种类型泛化。
除了泛化 还有具化，具化主要发生在条件控制流里 


## 声明类型

在ts里面 一个声明实际上会包含一种概念的声明
1. 命名空间
2. 类型 
3. 值

命名空间在编译产物里面的存在形式是立即执行函数 如果存在在其他文件里面对命名空间进行扩充 ts会把他们编译之后的产物合并成一个。
对于类型 只会在编译期间存在。
有一些申明方式可以同时创建两个 就比如：
- `class` 可以同时创建值类型 并且可以使用 `typeof` 获取这个类型
- `enum` 同时创建了类型值 

但是对于 `interface` 和 `type` 就只创建了类型 在编译之后 就不再存在了。

对于interface生命的类型 是可以合并的 这个特性在ts自带的js基础库 dom库的定义里面 
就比如 新版本js引入了新的函数 那么这个针对新版本的扩充会被放在一个ts文件里面 如果我们的编译产物的target是新版本 那么就会引入这个ts文件 

说到合并 命名空间也可以合并 而且不只是和同名的命名空间进行合并 还可以和同名的函数、类、枚举进行合并。
起到的效果是：向这个名字指向的值的类型定义中扩充更多的值。
通过命名空间和类合并 我们可以创建内部属性 内部方法 内部类：
```typescript
class Album {
  label: Album.AlbumLabel;
}

namespace Album {
  export class AlbumLabel {}
}
```
这样我们就创建了一个内部的类 这个内部的类就相当于一个静态属性 因为这个属性是直接加载这个class上的 而不是实例化的时候添加 虽然但是 使用namespace来合并和静态成员的行为也是有差异的：
```typescript
class Album {
  static AlbumLabel = class {};
  static getLabel() {
    return new Album.AlbumLabel();
  }
}
```
这段代码 使用了静态成员 编译之后的结果是这样的：
```javascript
var Album = /** @class */ (function () {
    function Album() {
    }
    Album.getLabel = function () {
        return new Album.AlbumLabel();
    };
    Album.AlbumLabel = /** @class */ (function () {
        function class_1() {
        }
        return class_1;
    }());
    return Album;
}());
```
我们可以看到 类的编译结果是返回一个函数的立即执行函数  静态属性就是返回的这个函数上面的属性

如果我们使用命名空间合并的方式：
```javascript
var Album = /** @class */ (function () {
    function Album() {
    }
    return Album;
}());
(function (Album) {
    var AlbumLabel = /** @class */ (function () {
        function AlbumLabel() {
        }
        return AlbumLabel;
    }());
    Album.AlbumLabel = AlbumLabel;
    function getLabel() {
        return new AlbumLabel();
    }
    Album.getLabel = getLabel;
})(Album || (Album = {}));
```
我们可以看到相对于上一种方式 多创造了一个函数 通过把之前创建的函数传递给他实现附加属性。
并且由于这种特性 我们在使用es6进行模块导入的时候 需要区分类本身、命名空间、合并之后的Album对象：
```typescript
export class Album {
  label: Album.AlbumLabel;
}

export namespace Album {
  export class AlbumLabel {}
  export function getLabel() {
    return new AlbumLabel();
  }
}

export default Album;
```
- `export class Album` 导出了类本身。
- `export namespace Album` 导出了命名空间及其内容。
- `export default Album` 将 `Album` 对象（类和合并的命名空间）作为默认导出。

```typescript
import Album, { Album as AlbumType } from './es6-namespace';
const label = Album.getLabel(); // 运行时访问
const instance = new Album(); // 创建实例
let instanceLabel: AlbumType.AlbumLabel; // 类型访问
```
这可能让我们在使用的时候有些混淆 是不是这个AlbumType指代的是命名空间？
并不是 实际上 这个Albumtype 和这个Album对象完全相等 根据上面的分析 其实我们就可以明确：类Album创建了实体 namespace向这个实体扩充了定义。

那我们要是只export这个类 并导出这个命名空间？
我们分析一下 即使不导出这个命名空间 在编译产物的函数执行的时候还是会对这个对象进行扩充，所以我们还是可以和原来一样使用这个对象。
但是实际上 不导出命名空间的话 就能访问通过命名空间扩充的方法 属性。
`export` 关键字就像一个“公开”声明，它将一个模块内部的声明标记为“公共 API”，使其能够被其他模块通过 `import` 语句访问。

甚至可能直接报错 如果你在这个类里面使用了通过命名空间拓展的属性方法的话 毕竟你没有向外暴露这些方法属性 如果有其他的文件引入这个类并且通过编译 就可能造成运行时错误：
```typescript
export class Album {
  // 注意：即使在这里，这一行也可能已经报错了，
  // 因为 Album.AlbumLabel 类型在 class 定义时可能还不可见。
  label: Album.AlbumLabel;
}

// 这是一个模块内部的、私有的命名空间
namespace Album {
  export class AlbumLabel {}
  export function getLabel() { /* ... */ }
}
```
为了让“声明合并”的魔力能够跨越模块边界，你打算在外部使用的每一个合并部分都必须被显式地 export。
当 TypeScript 编译器处理一个模块时，它会收集所有被 export 的同名声明，将它们合并成一个单一的实体，这个实体同时具备所有合并部分的特征（既是类，又像命名空间一样包含其他类型和值）。然后，这个合并后的实体被放入模块的导出列表中。

那我们可不可以只暴露命名空间 不暴露这个类呢 ？
这是完全可以的 如果我们只导出这个类型空间 就相当于这个类型空间的编译产物中的函数 参数得到了一个空对象的时候 这个时候我们导入的Album就像一个对象：
```javascript
{
  getLabel: [Function: getLabel],
  AlbumLabel: [class AlbumLabel]
}
```
我们可以通过这个对象 访问上面的水泥修改 也可以把这个Album作为一个类型容器：
```typescript
let myLabel: Album.AlbumLabel; // 完全正确！
// TypeScript 知道 Album 是一个包含 AlbumLabel 类型的命名空间。

myLabel = new Album.AlbumLabel(); // 也完全正确！
// Album.AlbumLabel 是一个导出的类，可以被实例化。
```

通过命名空间和函数声明的合并 可以给函数添加附加属性：
```typescript
function buildLabel(name: string): string {
  return buildLabel.prefix + name + buildLabel.suffix;
}

namespace buildLabel {
  export let suffix = "";
  export let prefix = "Hello, ";
}

console.log(buildLabel("Sam Smith"));
```
从这里我们可以看出来 什么样的东西 可以和类型空间进行合并？——编译之后得到一个函数或者对象的值。
命名空间的合并 就是在编译结果里面获取创建的函数或者对象 并且给这个函数或者对象添加属性。


## 模块扩充

模块扩充指的是在模块里面 对其他模块的类型定义进行扩充 但是这种扩充只是扩充了类型。也就是说 我们编写的所有内容 都不会在编译产物里面出现。
所以通过模块扩充 我们避免了编译时错误 但是如果我们不向模块里面添加运行时变量的话 就会遇到运行时错误。
模块扩充的一个良好实践就是：当我们需要在新版本里面给一些对象添加一些方法 我们就可以在一个补丁文件里面对这个对象添加属性方法 再使用模块扩充拓展类型申明：

```typescript
// observable.ts
export class Observable<T> {
  // ...
}

// map.ts
import { Observable } from "./observable";

declare module "./observable" {
  interface Observable<T> {
    map<U>(f: (x: T) => U): Observable<U>;
  }
}

Observable.prototype.map = function (f) {
  // ... another exercise for the reader
};

// consumer.ts
import { Observable } from "./observable";
import "./map";
let o: Observable<number>;
o.map((x) => x.toFixed());
```

## 完整的类型定义

在js里面 基本类型的属性方法 是使用的包装类型 但是在ts里面 类型的定义是完整的：
`keyof string` 这个类型里面有什么，你会发现它会包含字符串对象里面的各种属性名。类型系统为各种基础类型都提供了它的实例通过原型链获得的可访问的属性。其中，类型系统还会将一些类型的个别属性具化为有用的字面量类型。
```typescript
type GetParentTypeOf<T extends number> = ReturnType<T['valueOf']>;
```

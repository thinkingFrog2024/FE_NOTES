结构化类型系统 
整体上 typescript是一个结构化类型系统的语言 ，这使得我们可以只检查我们需要的属性方法等。
但是并不是全部 比如protect的属性 private的属性 这些属性还是名义化类型 ，对于这样的属性或者方法检查的时候 检查的不仅是是否存在 并且还要检查来自于哪个类
为什么类型{}框定格的是任何不是null 或者undefined的值？ 感觉实际上如果是这个类型 添加属性是会报错的。

判据
一般会认为 如果一个变量 是一个类型的子类型 那么这个值可以赋值给这个类型的变量
这其中涉及两个判据：可不可以赋值 是不是子类型
但是这两者并不是完全相等的 赋值在是子类型的基础上多了两条规则：any类型可以进行任何类型收束 就比如只要一个变量是any类型 可以访问这个变量上面的任何属性 都不会报错。 不同的枚举类型之间赋值规则基于代表的数字值。
ts的类型检查存在一下五种：
1. Identity 相等： 这是ts里面最严格的关系 代表严格相等 在结构 名称 泛型参数上都必须完全一致 。如果引用接口或者类型别名 必须是同一个 这里有一点名义化结构
interface Point1 { x: number; y: number; }
interface Point2 { x: number; y: number; }
type T7 = Point1;
type T8 = Point2;
// T7 和 T8 结构相同，但非 Identity
// 因为它们源于不同的接口定义
2. Strict Subtypeing  仅次于identity 如果类型 A 是类型 B 的严格子类型，那么类型 A 必须至少拥有类型 B 的所有属性，且对应属性的类型也是严格子类型关系。此关系不允许出现 any 类型。
  a. A 和 B 都是对象类型。
  b. A 拥有 B 的所有属性。
  c. 对于 B 中的每一个属性 P，A 中对应的属性 P 的类型必须是 B 中属性 P 类型的严格子类型（递归定义）。
  d. 整个比较链条中不能出现 any。
3. Subtyping 和上者比较 允许any的存在
4. Assignabillity 在 Subtyping 的基础上，进一步放宽了对函数类型的处理，引入了“协变”和“逆变”的概念
5. Comparabilty Comparability 是最宽松的类型关系，主要用于 ==、!=、===、!== 等比较操作符。它的目标是允许两种可能存在重叠的类型进行比较，而不是严格地限制它们

● Identity 是起点，要求类型完全一致。
● Strict Subtyping 放宽到结构兼容，开启了 TypeScript 结构化类型的大门。
● Subtyping 在结构兼容的基础上，为 any 打开了“绿色通道”，用于处理动态数据。
● Assignability 进一步为函数类型量身定制了更灵活的规则（逆变/协变），以支持高阶函数和回调等常见模式。
● Comparability 是最后的防线，只要两种类型有理论上重叠的可能，就允许比较，防止无意义的代码。


类型的来源

ts究竟是怎么知道一个变量的类型 ？
1. 手动标识： 我们平时使用：type 来表示一个变量类型 还可以使用类型断言 也就是as关键字 <type>前缀
2. JSDoc ts可以读取这个Doc
3. 类型推导：这个是ts借助很多的机制去推导一个变量的类型 但是这种机制也是很复杂的，在具体和模糊之间怎样选择
declare function myApi(someTuple: [string, boolean]): void;

const firstTry = ['foo', false];
// 推导的类型为：(string | boolean)[]
myApi(firstTry); 
// 报错！
// Argument of type '(string | boolean)[]' is not assignable to parameter of type '[string, boolean]'.
//   Target requires 2 element(s) but source may have fewer. 

const secondTry = ['foo', false] as const;
// 推导的类型为：readonly ["foo", false]
myApi(secondTry); 
// 还是报错！
// Argument of type 'readonly ["foo", false]' is not assignable to parameter of type '[string, boolean]'.
//  The type 'readonly ["foo", false]' is 'readonly' and cannot be assigned to the mutable type '[string, boolean]'

// 这样才可以……
myApi(['foo', false]);
第一个方式 ts推导出来的类型台模糊 第二个方式 我们使用as const 申明 虽然类型对了 但是多了个readonly readonly的变量和普通的变量对比的范围更小 所以类型检查也不能通过（ as大法好）
只有我们直接传入元组字面量的水后 ts才会推断出我们想要的那个类型  实际上 把这个元组类型推断成我们期望的那个类型 是一种类型泛化。
除了泛化 还有具化，具化主要发生在条件控制流里 


声明类型
在ts里面 一个声明实际上会包含一种概念的声明
1. 命名空间
2. 类型 
3. 值
命名空间在编译产物里面的存在形式是立即执行函数 如果存在在其他文件里面对命名空间进行扩充 ts会把他们编译之后的产物合并成一个
对于类型 只会在编译期间存在 
有一些申明方式可以同时创建两个 就比如：
class 可以同时创建值类型 并且可以使用typeof获取这个类型
enum 同时创建了类型值 
但是对于interface和type1 就只创建了类型 在编译之后 就不再存在了。



对于interface生命的类型 是可以合并的 这个特性在ts自带的js基础库 dom库的定义里面 
就比如 新版本js引入了新的函数 那么这个针对新版本的扩充会被放在一个ts文件里面 如果我们的编译产物的target是新版本 那么就会引入这个ts文件 

说到合并 命名空间也可以合并 而且不只是和同名的命名空间进行合并 还可以和同名的函数  类  枚举进行合并
起到的效果是：向这个名字指向的值的类型定义中扩充更多的值 。
通过命名空间和类合并 我们可以创建内部属性 内部方法 内部类、
class Album {
  label: Album.AlbumLabel;
}

namespace Album {
  export class AlbumLabel {}
}
这样我们就创建了一个内部的类 这个内部的类就相当于一个静态属性 因为这个属性是直接加载这个class上的 而不是实例化的时候添加 虽然但是 使用namespace来合并和静态成员的行为也是有差异的：
class Album {
  static AlbumLabel = class {};
  static getLabel() {
    return new Album.AlbumLabel();
  }
}
这段代码 使用了静态成员 编译之后的结果是这样的：
var Album = /** @class */ (function () {
    function Album() {
    }
    Album.getLabel = function () {
        return new Album.AlbumLabel();
    };
    Album.AlbumLabel = /** @class */ (function () {
        function class_1() {
        }
        return class_1;
    }());
    return Album;
}());
我们可以看到 类的编译结果是返回一个函数的立即执行函数  静态属性就是返回的这个函数上面的属性

如果我们使用命名空间合并的方式：
var Album = /** @class */ (function () {
    function Album() {
    }
    return Album;
}());
(function (Album) {
    var AlbumLabel = /** @class */ (function () {
        function AlbumLabel() {
        }
        return AlbumLabel;
    }());
    Album.AlbumLabel = AlbumLabel;
    function getLabel() {
        return new AlbumLabel();
    }
    Album.getLabel = getLabel;
})(Album || (Album = {}));
我们可以看到相对于上一种方式 多创造了一个函数 通过把之前创建的函数传递给他实现附加属性
并且由于这种特性 我们在使用es6进行模块导入的时候 需要区分；类本身 命名空间  合并之后的Album对象：
export class Album {
  label: Album.AlbumLabel;
}

export namespace Album {
  export class AlbumLabel {}
  export function getLabel() {
    return new AlbumLabel();
  }
}

export default Album;
      `export class Album` 导出了类本身。
     `export namespace Album` 导出了命名空间及其内容。
   `export default Album` 将 `Album` 对象（类和合并的命名空间）作为默认导出。
import Album, { Album as AlbumType } from './es6-namespace';
const label = Album.getLabel(); // 运行时访问
const instance = new Album(); // 创建实例
let instanceLabel: AlbumType.AlbumLabel; // 类型访问
这可能让我们在使用的时候有些混淆 是不是这个AlbumType指代的是命名空间？
并不是 实际上 这个Albumtype 和这个Album对象完全相等 根据上面的分析 其实我们就可以明确：类Album创建了实体 namespace向这个实体扩充了定义
那我们要是只export这个类 并导出这个命名空间？
我们分析一下 即使不导出这个命名空间 在编译产物的函数执行的时候还是会对这个对象进行扩充，所以我们还是可以和原来一样使用这个对象。
但是实际上 不导出命名空间的话 就能访问通过命名空间扩充的方法 属性
export 关键字就像一个“公开”声明，它将一个模块内部的声明标记为“公共 API”，使其能够被其他模块通过 import 语句访问。
甚至肯呢个直接报错 如果你在这个类里面使用了通过命名空间拓展的属性方法的话 毕竟你没有向外暴露这些方法属性 如果有其他的文件引入这个类并且通过编译 就可能造成运行时错误：
export class Album {
  // 注意：即使在这里，这一行也可能已经报错了，
  // 因为 Album.AlbumLabel 类型在 class 定义时可能还不可见。
  label: Album.AlbumLabel;
}

// 这是一个模块内部的、私有的命名空间
namespace Album {
  export class AlbumLabel {}
  export function getLabel() { /* ... */ }
}
为了让“声明合并”的魔力能够跨越模块边界，你打算在外部使用的每一个合并部分都必须被显式地 export。
当 TypeScript 编译器处理一个模块时，它会收集所有被 export 的同名声明，将它们合并成一个单一的实体，这个实体同时具备所有合并部分的特征（既是类，又像命名空间一样包含其他类型和值）。然后，这个合并后的实体被放入模块的导出列表中。

那我们可不可以只暴露命名空间 不暴露这个类呢 ？
这是完全可以的 如果我们只导出这个类型空间 就相当于这个类型空间的编译产物中的函数 参数得到了一个空对象的时候 这个时候我们导入的Album就像一个对象：
{
  getLabel: [Function: getLabel],
  AlbumLabel: [class AlbumLabel]
}
我们可以通过这个对象 访问上面的水泥修改 也可以把这个Album作为一个类型容器 ：
let myLabel: Album.AlbumLabel; // 完全正确！

myLabel = new Album.AlbumLabel(); // 也完全正确！

元祖
元祖原本的定义就是 长度以及每个位置元素的类型确定的数组 但是ts逐渐给元组增加了更多特性：
具名元组
type Foo = [first: number, second?: string, ...rest: any[]];
这使得表达信息更加清晰 并且我们可以给剩余元素定义成元组类型 这样元素名称就会被用作函数参数名 

可变元组
从ts4.0开始 元组被赋予可变的能力 这多亏了两个特性 展开语法支持泛型 剩余元素能够位于任何位置
展开语法支持泛型 指的是：在一个元组类型里面出现的展开语法 如果作用于一个泛型类型 那么可以在类型推导的时候被推导成元组类型
这句描述里面并没有提到“可变的能力” 那么这个能力体现在哪里？
说得更详细一些 是这个泛型被推导成元组类型的时候 因为假设泛型实际上并不是一个定长并且类型确定的变量 那么推导出来的元组类型就是一个可变的
并且这个特性和剩余元素能够位于任何位置也是关联的

在探索这个特性之前 我们先弄清楚 怎样表示一个可变的元组 ？
我们将会使用一个特殊的语法 这个语法表示一个类型构造 可以表示不确定长度的某个类型的元素 例如 不确定长度并且类型是string的数组：...string[]  我们可以理解成：元组的剩余元素
// 获得除了第一个元素之外的剩余元素
declare function tail<T extends readonly unknown[]>(_: readonly [unknown, ...T]): T;

const myTuple = [1, 2, 3, 4] as const;
const myArray = ["hello", "world"];
 
const _1 = tail(myTuple);
//    ^? const _1: [2, 3, 4]
 
const _2 = tail([...myTuple, ...myArray] as const);
//    ^? const _2: [2, 3, 4, ...string[]]//这里可不是运行时值 而是编译时类型
在_2这个变量里面 我们使用as const 告诉ts：应该推断出最精准的类型 于是ts将会分别对mytuple进行类型推断 再对myArray进行类型推断 
对于muTuple 他是一个元组 所以得到的类型推断的结果是一个元组类型 但是myArray是一个可变数组 得到的结果将会使用一个类型构造来表示不定长度的元素类型是string类型的数组 
并且这里泛型类型约束还有一个readonly表示只读类型 在类型系统中 只读类型实际上是可变类型的父类型（因为可变类型上面会添加更多改写数组的方法），所以这里如果我们不添加readonly并且传入的参数是元组类型 这里是会报错的
并且对于一些不会改变数组的函数 我们也应该添加readonly 这样如果在这个函数内部对数组进行修改 就会报错

并且在这个例子里面 这个类型构造起到了一个剩余数组的作用

并且由于新版本里面 剩余元素不是只能位于最后的 所以如果传递两次myArray数组：
const _2 = tail([...myTuple, ...myArray, ...myArray] as const);
将会得到：[2, 3, 4, ...string[], ...string[]]

并且结合剩余元素可以位于元组的任何位置 我们可以这样使用：
declare function concat<T extends readonly unknown[], U extends readonly unknown[]>
  (arr1: T, arr2: U): [...T, ...U];

const _1 = concat([1], [2]);
//    ^? const _1: number[]

const _2 = concat(['foo', 'bar'], [1, 2]);
//    ^? const _2: (string | number)[]

const _3 = concat([1] as number[], ['foo'] as string[]);
//    ^? const _3: (string | number)[]

const _4 = concat([1] as const, ['foo'] as const);
//    ^? const _4: [1, "foo"]
注意 这里返回的是一个元组类型 这个是元组的语法  对于数组类型 只有两种语法TypeName[] 或 Array<TypeName>
但是为什么返回的并不是元组类型？ 这是因为ts类型系统会根据TU的具体类型对结果进行简化 就比如 对于 concat([1], [2]) 得到的结果应该是[...number[], ...number[]]    
这个结果将会被简化成：number[]


你可能会有一个疑问：这里的泛型类型被约束成一个只读的数组 为什么没有被推断成元组类型？
这是因为 对于泛型类型的约束 并不会成为推断泛型类型的的判据 仅仅是起一个检查作用
在传递参数的时候 [1] [2] 将会被·推断成 number[] 并且这个类型是满足readonly unknown[]这个限制的

但是我们可以通过：在元组里面使用展开语法作用于一个只读的泛型时 这个泛型将会被推导成元组类型
declare function concat<T extends readonly unknown[], U extends readonly unknown[]>(arr1: [...T], arr2: [...U]): [...T, ...U];

const _1 = concat([1], [2]);
//    ^? const _1: [number, number]

const _2 = concat(['foo', 'bar'], [1, 2]);
//    ^? const _2: [string, string, number, number]

const _3 = concat([1] as number[], ['foo'] as string[]);
//    ^? const _3: (string | number)[]

const _4 = concat([1] as const, ['foo'] as const);
//    ^? const _4: [1, "foo"]

我们之前说到的：对于泛型参数的约束并不是推断泛型类型的判据 但是在这里：参数的形状给ts编译器提供了一个强力的上下文提示 这里的[...T]（一个变长元组类型）时，它会对自己说：“我期望在这里收到一个元组。请不要对传递给我的、模棱两可的数组字面量 于是ts在进行类型推断的时候 将会推断成元组类型

并且我们注意 这里如果没有使用 as const  申明成不变的常量 那么在类型推断的时候虽然得到的类型确实是元组 但是元素的类型并不是字面量
如果我们希望这个元组的元素一直被推断成字面量 而不需要用户手动添加as const 我们应该使用const 类型参数 
declare function concat<const T extends readonly unknown[], const U extends readonly unknown[]>(arr1: [...T], arr2: [...U]): [...T, ...U];

const _1 = concat([1], [2]);
//    ^? const _1: [1, 2]

const _2 = concat(['foo', 'bar'], [1, 2]);
//    ^? const _2: ["foo", "bar", 1, 2]

const _3 = concat([1] as number[], ['foo'] as string[]);
//    ^? const _3: (string | number)[]

const _4 = concat([1] as const, ['foo'] as const);
//    ^? const _4: [1, "foo"]

我们之前说到的as const 他是作用于一个变量 用于申明这个变量的值是确定的 
但是这里的const 是作用于泛型参数的 
一个是把变量的类型推断成元组 一个是把泛型参数的类型推断成元组 但是as const 会把变量推断成readonly
如果我们想把一个数组推断成字面量类型的元组 并且不希望他是一个只读的 我们就需要自己实现一个函数：
function asTuple<const T extends unknown[]>(input: [...T]) {
    return input;
}

const _1 = asTuple([1, 'foo']);
//    ^? const _1: [1, "foo"]

const _2 = [1, 'foo'] as const;
//    ^? const _2: readonly [1, "foo"]

这里使用[...T] 规定成元组类型 又使用const推断成字面量类型。
可选元素 
ts允许元组里面存在可选元素 ：
declare function foo(arg1?: string, arg2?: number): void;

type _1 = Parameters<typeof foo>;
//   ^? type _1 = [arg1?: string | undefined, arg2?: number | undefined]
注意当 strictNullChecks 配置被打开时，TypeScript 会向可选元素的类型中附加一个 undefined 类型形成联合类型。
可选元素的存在会导致元组类型的长度虽然固定但不唯一

模版字面量类型
类似于模版字符串 在js的模版字符串里面 我们可以嵌入变量 或者表达式 在ts的模版字面量类型里面 我们也可以嵌入字符串字面量类型：
type Dance = "Dance";
type ByteDance = `Byte${Dance}`;
//   ^? type ByteDance = "ByteDance"
在这个例子里面 嵌入了一个字面量类型 我们还可以嵌入其他的类型
● string
● number
● bigint
● boolean
● undefined
● null
还可以使用联合类型：
type VerticalAlignment = "top" | "middle" | "bottom";
type HorizontalAlignment = "left" | "center" | "right";

type Alignment = `${VerticalAlignment}-${HorizontalAlignment}`;
//   ^? type Alignment = "top-left" | "top-center" | "top-right" | "middle-left" | "middle-center" | "middle-right" | "bottom-left" | "bottom-center" | "bottom-right"
于是 我们可以用模版字符串类型代替正则表达式的作用：
// 检查是否以 Dance 结尾
type _1 = 'ByteDance' extends `${string}Dance` ? true : false;
//   ^? type _1 = true

    MyEnum[MyEnum["Bar"] = 1] = "Bar";
    MyEnum[MyEnum["Baz"] = 2] = "Baz";
})(MyEnum || (MyEnum = {}));
console.log(MyEnum.Baz);
但是常值枚举 也就是使用const进行修饰的枚举 将会直接把值内联到使用他的地方：

  Foo, Bar, Baz
}

console.log(MyEnum.Baz);

而常值枚举由于是直接内联值 使得它并不能在运行时进行计算 所以常值枚举无法使用计算成员

在ts5.0之后 所有的枚举成员 包括计算类型 都成为了独立的类型

  Foo = 'foo', Bar = 'bar', Baz = 'baz'
}

type _0 = `${MyEnum}`;
//   ^? type _0 = "foo" | "bar" | "baz"

  Submit = "submit".length, // 计算成员

// 在 TS 5.0 之前，EventType 的类型大致为 EventType.Click | number
// (字面量成员类型 | 计算成员拓宽后的类型)

 * type EventName = "event-1" | `event-${number}`

event-${number} 这个类型意味着它可以匹配 event-1、event-100、event-9999 等任何以 event- 开头的、后缀为数字的字符串。这完全违背了我们使用枚举的初衷——我们只想要 event-1 和 event-6 这两个精确的类型，而不是一个宽泛的、可以匹配任意数字的模式。

但是 虽然在类型层面 一个枚举类型可以被理解成一个联合类型 却不可以这样使用：

// 我们也希望用户不必导入 MyEnum 就能传参

，因为枚举的存在意义在于枚举成员的名字，而不是它的值

但是我们可以把数字字面量直接赋值给数字枚举：

  Foo, Bar, Baz

并且 我们可以把任意数据赋值给数字类型的枚举 这是为了实现位编程
就比如 我们使用位编程的方式 表示用户权限：
enum FileAccess {
  // 使用位移运算符 (<<) 来创建 2 的幂次方，更具可读性
  None    = 0,      // 0 (二进制 000) - 没有权限
  Read    = 1 << 0, // 1 (二进制 001) - 读权限
  Write   = 1 << 1, // 2 (二进制 010) - 写权限
  Execute = 1 << 2, // 4 (二进制 100) - 执行权限
}

const permissions: FileAccess = FileAccess.Read | FileAccess.Write;
数字 3 是一个完全合法的权限组合（代表“读”和“写”都已开启），但 3 这个值本身并不是 FileAccess 枚举中任何一个已命名的成员（None, Read, Write, Execute 的值分别是 0, 1, 2, 4）。

因此，允许任何数字被赋值给数字枚举类型的这个“漏洞”或“副作用”，是一个刻意的设计权衡。这是为了完全支持位标志模式而付出的必要“代价”，而支持该模式正是 TypeScript 枚举的一个主要设计目标。

名义类型
我们可以通过名义类型 把对参数的验证逻辑转移到核心逻辑之外
declare const TYPE_TAG: unique symbol; // 2.7+
// 通过 & 并入一个特殊属性来定义名义类型（一些人将这个过程称为 Tagging）
type PositiveValue = number & { [TYPE_TAG]: '_' };

// 我们的 API
declare function doSomething(value: PositiveValue): void;

// 我们自己提供一个将 number 转为 PositiveValue 的函数
function asPositiveValue(value: number): PositiveValue {
  if (value <= 0) {
    throw new Error(...);
  }
  return value as PositiveValue;
}

doSomething(-123); // 报错
doSomething(asPositiveValue(123)); // 不报错
这里我们给一个通过类型检查的变量打上一个特殊的编译时的标记 所以参数必须通过这个函数的检查 并且如果参数不满足要求 将会在运行时抛出一个错误 
但是由于type PositiveValue = number & { [TYPE_TAG]: '_' }; 实际上的计算结果是一个never类型 所以我们在函数里面使用这个变量的时候 需要重新对其进行类型断言

阻止更加简洁的联合类型
type Foo = 1 | 2 | 3;
type Bar = Foo | 4 | 5;
//   ^? type Bar = Foo | 4 | 5
自 TypeScript 4.2 版本起，为了提升代码的可读性，编译器引入了一项名为“Smarter Type Alias Preservation”的特性
TypeScript 认为，在类型提示中显示 Foo | 4 | 5 比显示 1 | 2 | 3 | 4 | 5 更加简洁明了。如果 Foo 是一个从外部库导入的复杂类型，保留其别名 Foo 会让我们更容易理解 Bar 的结构。
但是有的时候我们认为这个类型提示有一些不直观 如果我们希望得到更加直接的提示 我们可以使用这个小技巧：
type Foo = 1 | 2 | 3;
type Bar = Foo & {} | 4 | 5;
//   ^? type Bar = 1 | 2 | 3 | 4 | 5
这里的 & {} 是一个广为人知的技巧，它的作用是强制 TypeScript 立即（Eagerly）计算出类型的完整形态，从而阻止上述的“别名保留”行为。
为什么 & {} 能做到这一点？
编译器的“别名保留”逻辑虽然智能，但它有自己的原则。当它遇到一个必须知道类型内部结构才能继续的操作时，它就不得不放弃“偷懒”，开始真正的计算。交叉类型 & 就是这样一个操作。
1. {} 类型的真正含义：在 TypeScript 中，{} 类型并不单指“空对象”，它代表的是任何非 null 和非 undefined 的值。因此，原始类型 string、number 等都可以被赋值给 {} 类型。
2. 交叉运算的结果：正因为如此，一个原始类型与 {} 进行交叉运算，结果仍然是它自身。例如，1 & {} 的结果就是 1。
3. 触发信号：尽管 & {} 在类型上看似是一个“无操作”（no-op），但 & 运算符本身的存在，就像一个发给编译器的强制求值信号。它告诉编译器：“别再保留别名了，我需要你立刻展开 Foo，因为我要对它的内部结构进行交叉运算。”


让我们一步步拆解编译器的思考过程：
1. 分析表达式：编译器看到 Foo & {} | 4 | 5 这个表达式。根据运算符优先级，它需要先处理 Foo & {}。
2. 触发求值：为了计算 Foo & {} 的结果，编译器必须知道 Foo 的具体形态。它不能再简单地把 Foo 当作一个黑盒。
3. 展开别名：编译器将 Foo展开为其原始定义：1 | 2 | 3。
4. 执行交叉运算：现在，表达式变成了 (1 | 2 | 3) & {}。交叉运算会分配到联合类型的每一个成员上，即 (1 & {}) | (2 & {}) | (3 & {})

并且联合类型还存在subtype reduction这种情况 例如一个联合类型：'foo'|string 在提示的时候 我们看到的结果却是string
这是因为'foo'是string的子类型 所以ts省略了他 
但是我们并不希望这样的省略  于是：
declare function foo(input: 'a' | 'b' | string & {}): void;
这里的原理（rationale）是通过提供一个名义类型阻止了 TypeScript 的约去动作。（5.4已经解决这个问题了）

类型断言
类型断言时不安全的 
虽然在有些时候 ts会阻止我们进行一些必然错误的类型断言 但是我们可以绕过这些检查 比如：
const x = "hello" as unknown as number;
我们可以使用更加安全的satisfies关键字
在这个关键字之前 我们的类型有两种转换方向：
向上转型：upcast 通过给一个具体的变量添加一个宽泛的类型 保证这个值符合类型契约 但是我们丢失了更精确的类型
向下：downcast 通过类型断言等方式 获取更加精确的类型 但是如果断言有问题 将会引起运行时错误
而satisfies的工作原理是：验证而不改变
type Config = Record<string, string | number>;

// 使用类型注解
const config: Config = {
  host: "localhost",
  port: 8080,
  env: "development",
};

// 无法访问 'toUpperCase', 因为 config.host 的类型被认为是 string | number
// config.host.toUpperCase(); // ❌ Property 'toUpperCase' does not exist on type 'string | number'.

// 无法将 config.port 当作 number 类型直接使用
// const portNumber: number = config.port; // ❌ Type 'string | number' is not assignable to type 'number'.

// 定义一个宽泛的类型
type Config = Record<string, string | number>;

// 使用 satisfies
const config = {
  host: "localhost",
  port: 8080,
  env: "development",
} satisfies Config; // ✅ 验证通过

// config.host 的类型被精确推断为 "localhost"
config.host.toUpperCase(); // ✅ OK!

// config.port 的类型被精确推断为 8080
const portNumber: number = config.port; // ✅ OK!

// 如果我们犯了错，satisfies 同样会报错
const wrongConfig = {
    host: "localhost",
    port: [123] // ❌ Type 'number[]' is not assignable to type 'string | number'.
} satisfies Config;

Type Predicts
这是一种在运行时进行类型检查的方式 但是这个检查是快照式的 所以事实上 他只能确保给定的值在输入函数的那一刻是某种类型
对于数组和对象这样的数据 他可以原地修改 ，原地修改指的是：这个变量指向一块内存地址 进行修改之后 内存地址不变 但是这个地址里面保存的内容却永久的改变了。
对于数组 对象 就可以进行这样的修改
但是对于原始类型 就不是这样了 更改他的时候 指向的内存地址其实也跟着更改了
问题就出在这里：ts的控制流分析 通过分析代码的执行路径 从而推断特定位置上更精确的变量类型
declare function isArrayOfSingleNumber(array: unknown): array is [number];
let myArray: unknown = [1];

if (isArrayOfSingleNumber(myArray)) {
  // CFA 在此将 myArray 的类型收窄为：[number]
  console.log(myArray.length); // 1, 符合类型

  myArray.push(2); // **原地修改！**

  // CFA 并未察觉到 .push() 的副作用。
  // 它仍然认为 myArray 的类型是 [number]。
  // 此时，编译时类型 [number] 与运行时值 [1, 2] 已经不再同步。
  console.log(myArray.length); // 2, 运行时正确，但类型系统已“受骗”
}

declare function isString(value: unknown): value is string;
let myVar: unknown = "hello";

if (isString(myVar)) {
  // CFA 在此将 myVar 的类型收窄为：string
  console.log(myVar.toUpperCase()); // OK, 输出 "HELLO"

  myVar = 123; // **重新赋值！**

  // CFA 立刻就侦测到了这次重新赋值。
  // 它会立即让之前对 myVar 的 'string' 类型收窄失效。
  // 现在，CFA 正确地将 myVar 的类型推断为 'number'。
  
  // console.log(myVar.toUpperCase()); // ❌ 编译时立即报错！
  // 错误: "toUpperCase" a property does not exist on type 'number'.
  // 类型系统成功阻止了潜在的运行时错误。
}
对于数组和对象 我们可以通过readonly关键字 避免对其进行后续修改：
declare function isArrayOfSingleNumber(array: unknown[]): array is readonly [number];

if (isArrayOfSingleNumber(myArray)) {
  // 👿 怎么编译不通过了，原来是因为 ReadonlyArray 类型没有 push 函数！
  myArray.push(2);
}
类型谓词 
上面的这种解决方案 就是使用了类型谓词 
类型谓词就是一种特殊的函数返回值类型：parameterName is Type
这里的 parameterName必须是当前函数签名中的一个参数名。
当一个函数返回这种类型时，你实际上在向 TypeScript 编译器做出一个“承诺”：
● 这个函数本身在运行时会返回一个 boolean 值。
● 如果在编译时，这个函数返回 true，那么你可以安全地认为 parameterName 这个参数的类型就是更具体的 Type。
类型谓词的关键在于：可以和typescript的控制流分析结合
// 声明一个使用类型谓词的函数
declare function isArrayOfSingleNumber(array: unknown[]): array is readonly [number];

declare const myArray: unknown[];

// 在 if 语句之外，TypeScript 只知道 myArray 的类型是 unknown[]
// 它可能是 [1], ["a", "b"], [], [1, 2, 3], ...
console.log(myArray); // 类型: unknown[]

if (isArrayOfSingleNumber(myArray)) {
  // --- 魔法发生在这里 ---
  // 因为 isArrayOfSingleNumber(myArray) 返回了 true，
  // TypeScript 的 CFA 知道了这个信息，并查阅了函数的签名。
  // 它看到了 "array is readonly [number]" 这个承诺。
  
  // 因此，在这个 if 代码块内部，TypeScript 会将 myArray 的类型
  // 从原来的 unknown[] 收窄 (narrow) 为 readonly [number]。
  
  console.log(myArray); // 类型: readonly [number]
  
  // 此时你可以安全地访问元组特有的属性或进行只读操作
  const firstElement = myArray[0]; // firstElement 的类型是 number
  
  // 尝试修改会报错，因为类型已经是 readonly
  // myArray.push(2); // 编译错误：类型“readonly [number]”上不存在属性“push”。
}

Infer关键字
在通过条件类型判断输入类型是否满足给定条件（这本质上是检查它是否为目标类型的子类型）时，TypeScript 提供了 infer 关键字来让我们可以在这个过程中根据目标元素的泛型类型，提取出输入类型的额外信息。

type _1 = Flatten<string>;
//   ^? type _1 = string


type ReturnType<T extends (...args: any) => any> =
  T extends (...args: any) => infer R ? R : any;



type InstanceType<T extends abstract new (...args: any) => any> = T extends abstract new (...args: any) => infer R ? R : any;

● new (...args: any) => any: 这部分被称为“构造函数签名”（Construct Signature）。它描述了一个可以被 new 操作符调用的函数。具体来说：
  ○ ...args: any: 表示构造函数可以接受任意数量（...）、任意类型（any）的参数。
  ○ => any: 表示构造函数会返回一个任意类型（any）的实例。
  ○ 整体来看，new (...args: any) => any 代表了最通用的类构造函数形式。
● abstract 关键字: 这是理解 InstanceType 功能强大的关键。abstract 关键字的出现，意味着泛型 T 不仅可以是一个能够被直接 new 的具体类（concrete class），也可以是一个不能被直接实例化的抽象类（abstract class）。
分配式条件类型
在通过 extends 关键字表达泛型的类型约束时，很多人都会忘记要检查的类型可能是联合类型这一事实。例如，对于 T extends string，实际上当 T 为 'foo' | 'bar' 时也是满足约束的。
/一个自然的问题是，联合类型是如何参与到条件类型的计算中的？在 TypeScript 中，这个过程被称为分配式条件类型。我们通过下面的例子理解这个特性。简单来说，当联合类型作为条件类型的输入时，TypeScript 会将联合类型中的每个类型单独拆出来计算条件类型，然后将得到的结果重新组合为联合类型。

当 TypeScript 怀疑它碰到了无限递归（通过检查递归层数），或者一些需要花费很多时间计算的类型展开（比如对元组类型进行展开）时，它会停止类型检查并直接报错。对于前者，它会抛出下面的错误：
Type instantiation is excessively deep and possibly infinite.
因此，在使用递归时，为了减少资源占用和防止 TypeScript 报错，我们最好将递归步骤处理成满足尾递归的形式。从 TypeScript 4.5 开始，它会对递归类型做尾递归优化，这意味着当递归类型满足一定条件时，即使递归层数极其深也不会导致出现大量的资源占用，TypeScript 也不会报错。

type parameter inference
函数泛型可以让 TypeScript 从某个值中推导出类型，然后我们可以继续对这个类型进行计算
declare function setup<T>(config: { initial(): T }): T | null;

const _1 = setup({ initial() { return "seele" } });
//    ^? const _1: string | null
这里发生的过程被称为 Type Parameter Inference，本质上是计算出函数泛型参数究竟应该为何的过程。顺带一提，我们可以通过将鼠标放在第三行的 setup 上获得此时的函数签名来看到 TypeScript 对 T 的计算结果

1. 根据传递的参数推导出源类型：这个步骤除了参考参数本身 还会参考函数的签名信息（contextual signature）和泛型参数等类型（contextual type）
2. 解析方法签名的泛型 的熬一个目标类型 就比如我们的参数config对象是这样的：{ initial(): T } 那么这里得到的目标类型就是{ initial(): T }。复杂类型会进行额外的处理 就比如条件类型将会获取真分支假分支的结果
3.   TypeScript 会发现源类型和目标类型都是对象类型，于是检查它们的属性，在检查到 initial 时发现双方都有此属性。由于 initial 是函数类型，在继续推导到函数的返回值类型时，TypeScript 发现 string 可以被直接作为 T 的候选类型（candidate）。
4.   TypeScript 在选择了 string 之后，会将它带入原函数。之后，它就像在使用下面的函数一样，对我们传入的对象字面量进行类型检查。
declare function setup<string>(config: { initial(): string }): string | null;

约束检查
约束检查时我们业务侧的逻辑 
type DoCheck<T> = T extends ... ? unknown : never;

  return input;

当类型检查通过的时候 参数的类型是T 但是类型检查不通过的时候 参数的类型是never
但是这个通用方法只能对参数整体进行校验 所以不通过的时候 ts会对整个参数进行报错
但是在对大多数时候 我们都是希望对有问题的部分参数进行校验。

如果我们希望获得更加有好的报错信息 我们可以使用之前提到的名义类型：

// ...
//当类型没有经过检查的时候 类型会变成{}&& MyTypeError<"对象没有包含必须的类型 foo 哦">; 但是这个对象里面并没有这个键 于是就会报错

// Property '[ERROR_SYMBOL]' is missing in type '{}'
// but required in type 'MyTypeError<"对象没有包含必须的类型 foo 哦">'
我们也可以把这种编译时信息 添加到返回值上面 就比如：
type DoCheck<T> = 'foo' extends keyof T
  ? never
  : MyTypeError<"对象没有包含必须的类型 foo 哦">;

function check<T>(input: T): DoCheck<T> {
  return input as any;
}

check({}) satisfies never;

// 当两个数组元素类型相同时，可以直接使用可变元组类型
declare function check<T extends unknown[]>
  (a: [...T], b: [...T]): void;

check([1], []);  // ERROR
check([], [1]);  // ERROR
check([1], [1]); // OK

// 当两个数组元素类型不同时，可以通过对返回值类型进行检查
declare function check2<T extends unknown[], U extends unknown[]>
  (a: [...T], b: [...U]): T['length'] extends U['length'] ? true : false;

check2([1], []) satisfies true;      // ERROR
check2([], [1]) satisfies true;      // ERROR
check2([1], ['foo']) satisfies true; // OK

// 对返回值类型进行检验不是解决问题的唯一的办法，我们还有：
type DoCheck<T extends unknown[], U extends unknown[]> =
  T['length'] extends U['length'] ? unknown : never;

// 考虑这样一个事实：unknown 是 top type（全集），所以其它类型对它取交都等于原类型
// 而 never 是 bottom type（空集），其它类型对它取交都等于 never
declare function check3<T extends unknown[], U extends unknown[]>
  (a: [...T] & DoCheck<T, U>, b: [...U] & DoCheck<T, U>): void;

check3([1], []);      // ERROR
check3([], [1]);      // ERROR
check3([1], ['foo']); // OK

函数类型闭包
函数类型闭包的作用是：把泛型参数传递到返回值类型里从而保留下来
函数类型闭包 和我们平时说的 js里面的闭包 是紧密相关的 
真是因为运行时的函数 是一个闭包 才会有这个闭包类型：
type ValidatorMap = Record<
  string,
  NumberConstructor | StringConstructor | BooleanConstructor
>;

// 使用 mapped types 构造目标类型
type BuildResultType<T extends ValidatorMap> = {
  // 上述 Constructor 类型的返回值类型就是它们对应的原始值类型
  [K in keyof T]: ReturnType<T[K]>;
};

// Prettify 工具类型来自前文
declare function buildValidatorFn<T extends ValidatorMap>
  (input: T): (input: Record<string, unknown>) => input is Prettify<BuildResultType<T>>;

const validatorFn = buildValidatorFn({ foo: String, bar: Number, baz: Boolean});
//    ^? const validatorFn: (input: Record<string, unknown>) =>
//                            input is { foo: string; bar: number; baz: boolean }

// 下面来试用一下这个 validatorFn

if (validatorFn(_0)) {
  _0;
// ^? const _0: { foo: string; bar: number; baz: boolean; }
}
根据函数签名 这里buildValidatorFn 返回一个函数 并且这个函数的返回值是布尔值
并且在类型层面 当返回的布尔值是真的时候 可以确定input的类型 并且这个类型根据T计算得到
同态遍历类型 Homomoriphic Mapped Types
这种类型指的是基于现有类型 产生新的类型
比如：基于现有的泛型类型 产生新的对象类型：
type HMT<C> = {
  [K in C]: X;
};
这里的in操作符 在C的类型不同的时候 会有不同的行为：
1. 联合类型：遍历联合类型的每一个类型 把每个成员作为新类型的键 可以使用as对键进行重命名 这个过程称为分配运算
2. 数组/元组：对其中的元素进行计算 返回计算结果构成的数组或者元组 但是对于数组的话 其实相当于直接对这个数组类型进行计算
3. 字面量类型：相当于只有一个元素的联合类型
4. 原始类型：string number这样的原始类型 无论值写什么 返回的都是原始类型本身

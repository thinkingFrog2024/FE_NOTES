# 类型推断与控制流分析

## 类型断言与 satisfies

类型断言时不安全的。虽然在有些时候 ts会阻止我们进行一些必然错误的类型断言 但是我们可以绕过这些检查 比如：
```typescript
const x = "hello" as unknown as number;
```
我们可以使用更加安全的 `satisfies` 关键字。
在这个关键字之前 我们的类型有两种转换方向：
- **向上转型 (upcast)**：通过给一个具体的变量添加一个宽泛的类型 保证这个值符合类型契约 但是我们丢失了更精确的类型。
- **向下转型 (downcast)**：通过类型断言等方式 获取更加精确的类型 但是如果断言有问题 将会引起运行时错误。

而 `satisfies` 的工作原理是：**验证而不改变**。

```typescript
type Config = Record<string, string | number>;

// 使用类型注解
const config: Config = {
  host: "localhost",
  port: 8080,
  env: "development",
};

// 无法访问 'toUpperCase', 因为 config.host 的类型被认为是 string | number
// config.host.toUpperCase(); // ❌ Property 'toUpperCase' does not exist on type 'string | number'.

// 无法将 config.port 当作 number 类型直接使用
// const portNumber: number = config.port; // ❌ Type 'string | number' is not assignable to type 'number'.
```

使用 `satisfies` 之后：
```typescript
// 定义一个宽泛的类型
type Config = Record<string, string | number>;

// 使用 satisfies
const config = {
  host: "localhost",
  port: 8080,
  env: "development",
} satisfies Config; // ✅ 验证通过

// config.host 的类型被精确推断为 "localhost"
config.host.toUpperCase(); // ✅ OK!

// config.port 的类型被精确推断为 8080
const portNumber: number = config.port; // ✅ OK!

// 如果我们犯了错，satisfies 同样会报错
const wrongConfig = {
    host: "localhost",
    port: [123] // ❌ Type 'number[]' is not assignable to type 'string | number'.
} satisfies Config;
```

**误用泛型**：
这里讨论一个常见的对泛型的误用：实例化（instantiate）某个泛型参数之后没有为它提供类型别名。

## Type Predicts 与 控制流分析 (CFA)

这是一种在运行时进行类型检查的方式 但是这个检查是快照式的 所以事实上 他只能确保给定的值在输入函数的那一刻是某种类型。
对于数组和对象这样的数据 他可以原地修改，原地修改指的是：这个变量指向一块内存地址 进行修改之后 内存地址不变 但是这个地址里面保存的内容却永久的改变了。对于数组、对象 就可以进行这样的修改。
但是对于原始类型 就不是这样了 更改他的时候 指向的内存地址其实也跟着更改了。

问题就出在这里：ts的控制流分析 通过分析代码的执行路径 从而推断特定位置上更精确的变量类型。
```typescript
declare function isArrayOfSingleNumber(array: unknown): array is [number];
let myArray: unknown = [1];

if (isArrayOfSingleNumber(myArray)) {
  // CFA 在此将 myArray 的类型收窄为：[number]
  console.log(myArray.length); // 1, 符合类型

  myArray.push(2); // **原地修改！**

  // CFA 并未察觉到 .push() 的副作用。
  // 它仍然认为 myArray 的类型是 [number]。
  // 此时，编译时类型 [number] 与运行时值 [1, 2] 已经不再同步。
  console.log(myArray.length); // 2, 运行时正确，但类型系统已“受骗”
}
```

```typescript
declare function isString(value: unknown): value is string;
let myVar: unknown = "hello";

if (isString(myVar)) {
  // CFA 在此将 myVar 的类型收窄为：string
  console.log(myVar.toUpperCase()); // OK, 输出 "HELLO"

  myVar = 123; // **重新赋值！**

  // CFA 立刻就侦测到了这次重新赋值。
  // 它会立即让之前对 myVar 的 'string' 类型收窄失效。
  // 现在，CFA 正确地将 myVar 的类型推断为 'number'。
  
  // console.log(myVar.toUpperCase()); // ❌ 编译时立即报错！
  // 错误: "toUpperCase" a property does not exist on type 'number'.
  // 类型系统成功阻止了潜在的运行时错误。
}
```
对于数组和对象 我们可以通过readonly关键字 避免对其进行后续修改：
```typescript
declare function isArrayOfSingleNumber(array: unknown[]): array is readonly [number];

if (isArrayOfSingleNumber(myArray)) {
  // 👿 怎么编译不通过了，原来是因为 ReadonlyArray 类型没有 push 函数！
  myArray.push(2);
}
```

## 类型谓词 

上面的这种解决方案 就是使用了类型谓词。
类型谓词就是一种特殊的函数返回值类型：`parameterName is Type`
这里的 `parameterName` 必须是当前函数签名中的一个参数名。
当一个函数返回这种类型时，你实际上在向 TypeScript 编译器做出一个“承诺”：
- 这个函数本身在运行时会返回一个 boolean 值。
- 如果在编译时，这个函数返回 true，那么你可以安全地认为 `parameterName` 这个参数的类型就是更具体的 Type。

类型谓词的关键在于：可以和typescript的控制流分析结合：
```typescript
// 声明一个使用类型谓词的函数
declare function isArrayOfSingleNumber(array: unknown[]): array is readonly [number];

declare const myArray: unknown[];

// 在 if 语句之外，TypeScript 只知道 myArray 的类型是 unknown[]
// 它可能是 [1], ["a", "b"], [], [1, 2, 3], ...
console.log(myArray); // 类型: unknown[]

if (isArrayOfSingleNumber(myArray)) {
  // --- 魔法发生在这里 ---
  // 因为 isArrayOfSingleNumber(myArray) 返回了 true，
  // TypeScript 的 CFA 知道了这个信息，并查阅了函数的签名。
  // 它看到了 "array is readonly [number]" 这个承诺。
  
  // 因此，在这个 if 代码块内部，TypeScript 会将 myArray 的类型
  // 从原来的 unknown[] 收窄 (narrow) 为 readonly [number]。
  
  console.log(myArray); // 类型: readonly [number]
  
  // 此时你可以安全地访问元组特有的属性或进行只读操作
  const firstElement = myArray[0]; // firstElement 的类型是 number
  
  // 尝试修改会报错，因为类型已经是 readonly
  // myArray.push(2); // 编译错误：类型“readonly [number]”上不存在属性“push”。
}
```

## 非空数组判断

我们可以给用户提供一个函数来进行检查和类型转换：
```typescript
function asNonEmptyArray<T>(array: readonly T[]): NonEmptyArray<T> {
  if (!array.length) {
    throw new Error("Array is empty");
  }

  return array as any;
}

declare function last<T>(array: NonEmptyArray<T>): T;

// 这样，用户在调用 last 函数之前就必须先确保自己的函数非空
const nonEmptyArray = asNonEmptyArray(myArray);
last(nonEmptyArray);
```
我们通过这样一个函数 保证了传递的数组一定是非空的（这里的非空指的是 第一个元素一定是存在的）。
这里的返回值添加了readonly 是为了避免修改数组length导致删除所有元素。

```typescript
// 当 noUncheckedIndexedAccess 打开时
declare const arr1: number[];
const _1 = arr1[0];
//    ^? const _1: number | undefined

declare const arr2: [number, ...number[]];
const _2 = arr2[0];
//    ^? const _2: number

// 由于我们的「非空数组」已经暗含了数组的第一位不可能为空
// 所以可以使用可变元组的方式让 TypeScript 相信数组的第一位不可能为 undefined
```

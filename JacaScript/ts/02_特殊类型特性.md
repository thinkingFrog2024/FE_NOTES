# 特殊类型特性

## 元组

元祖原本的定义就是 长度以及每个位置元素的类型确定的数组 但是ts逐渐给元组增加了更多特性：

### 具名元组
```typescript
type Foo = [first: number, second?: string, ...rest: any[]];
```
这使得表达信息更加清晰 并且我们可以给剩余元素定义成元组类型 这样元素名称就会被用作函数参数名。

### 可变元组
从ts4.0开始 元组被赋予可变的能力 这多亏了两个特性：展开语法支持泛型，剩余元素能够位于任何位置。
展开语法支持泛型 指的是：在一个元组类型里面出现的展开语法 如果作用于一个泛型类型 那么可以在类型推导的时候被推导成元组类型。
这句描述里面并没有提到“可变的能力” 那么这个能力体现在哪里？
说得更详细一些 是这个泛型被推导成元组类型的时候 因为假设泛型实际上并不是一个定长并且类型确定的变量 那么推导出来的元组类型就是一个可变的。
并且这个特性和剩余元素能够位于任何位置也是关联的。

在探索这个特性之前 我们先弄清楚 怎样表示一个可变的元组？
我们将会使用一个特殊的语法 这个语法表示一个类型构造 可以表示不确定长度的某个类型的元素 例如 不确定长度并且类型是string的数组：`...string[]` 我们可以理解成：元组的剩余元素。

```typescript
// 获得除了第一个元素之外的剩余元素
declare function tail<T extends readonly unknown[]>(_: readonly [unknown, ...T]): T;

const myTuple = [1, 2, 3, 4] as const;
const myArray = ["hello", "world"];
 
const _1 = tail(myTuple);
//    ^? const _1: [2, 3, 4]
 
const _2 = tail([...myTuple, ...myArray] as const);
//    ^? const _2: [2, 3, 4, ...string[]] // 这里可不是运行时值 而是编译时类型
```
在_2这个变量里面 我们使用as const 告诉ts：应该推断出最精准的类型 于是ts将会分别对mytuple进行类型推断 再对myArray进行类型推断。
对于muTuple 他是一个元组 所以得到的类型推断的结果是一个元组类型 但是myArray是一个可变数组 得到的结果将会使用一个类型构造来表示不定长度的元素类型是string类型的数组。
并且这里泛型类型约束还有一个readonly表示只读类型 在类型系统中 只读类型实际上是可变类型的父类型（因为可变类型上面会添加更多改写数组的方法），所以这里如果我们不添加readonly并且传入的参数是元组类型 这里是会报错的。
并且对于一些不会改变数组的函数 我们也应该添加readonly 这样如果在这个函数内部对数组进行修改 就会报错。

并且在这个例子里面 这个类型构造起到了一个剩余数组的作用。
并且由于新版本里面 剩余元素不是只能位于最后的 所以如果传递两次myArray数组：
```typescript
const _2 = tail([...myTuple, ...myArray, ...myArray] as const);
// 将会得到：[2, 3, 4, ...string[], ...string[]]
```

并且结合剩余元素可以位于元组的任何位置 我们可以这样使用：
```typescript
declare function concat<T extends readonly unknown[], U extends readonly unknown[]>
  (arr1: T, arr2: U): [...T, ...U];

const _1 = concat([1], [2]);
//    ^? const _1: number[]

const _2 = concat(['foo', 'bar'], [1, 2]);
//    ^? const _2: (string | number)[]

const _3 = concat([1] as number[], ['foo'] as string[]);
//    ^? const _3: (string | number)[]

const _4 = concat([1] as const, ['foo'] as const);
//    ^? const _4: [1, "foo"]
```
注意 这里返回的是一个元组类型 这个是元组的语法 对于数组类型 只有两种语法 `TypeName[]` 或 `Array<TypeName>`。
但是为什么返回的并不是元组类型？ 这是因为ts类型系统会根据T和U的具体类型对结果进行简化 就比如 对于 `concat([1], [2])` 得到的结果应该是 `[...number[], ...number[]]`，这个结果将会被简化成：`number[]`。

你可能会有一个疑问：这里的泛型类型被约束成一个只读的数组 为什么没有被推断成元组类型？
这是因为 对于泛型类型的约束 并不会成为推断泛型类型的判据 仅仅是起一个检查作用。
在传递参数的时候 `[1]` `[2]` 将会被推断成 `number[]` 并且这个类型是满足 `readonly unknown[]` 这个限制的。

但是我们可以通过：在元组里面使用展开语法作用于一个只读的泛型时 这个泛型将会被推导成元组类型。
```typescript
declare function concat<T extends readonly unknown[], U extends readonly unknown[]>(arr1: [...T], arr2: [...U]): [...T, ...U];

const _1 = concat([1], [2]);
//    ^? const _1: [number, number]

const _2 = concat(['foo', 'bar'], [1, 2]);
//    ^? const _2: [string, string, number, number]

const _3 = concat([1] as number[], ['foo'] as string[]);
//    ^? const _3: (string | number)[]

const _4 = concat([1] as const, ['foo'] as const);
//    ^? const _4: [1, "foo"]
```
我们之前说到的：对于泛型参数的约束并不是推断泛型类型的判据 但是在这里：参数的形状给ts编译器提供了一个强力的上下文提示 这里的 `[...T]`（一个变长元组类型）时，它会对自己说：“我期望在这里收到一个元组。请不要对传递给我的、模棱两可的数组字面量进行简化”。于是ts在进行类型推断的时候 将会推断成元组类型。

并且我们注意 这里如果没有使用 `as const` 申明成不变的常量 那么在类型推断的时候虽然得到的类型确实是元组 但是元素的类型并不是字面量。
如果我们希望这个元组的元素一直被推断成字面量 而不需要用户手动添加 `as const` 我们应该使用 `const` 类型参数：
```typescript
declare function concat<const T extends readonly unknown[], const U extends readonly unknown[]>(arr1: [...T], arr2: [...U]): [...T, ...U];

const _1 = concat([1], [2]);
//    ^? const _1: [1, 2]

const _2 = concat(['foo', 'bar'], [1, 2]);
//    ^? const _2: ["foo", "bar", 1, 2]

const _3 = concat([1] as number[], ['foo'] as string[]);
//    ^? const _3: (string | number)[]

const _4 = concat([1] as const, ['foo'] as const);
//    ^? const _4: [1, "foo"]
```
我们之前说到的 `as const` 他是作用于一个变量 用于申明这个变量的值是确定的。但是这里的 `const` 是作用于泛型参数的。
一个是把变量的类型推断成元组 一个是把泛型参数的类型推断成元组 但是 `as const` 会把变量推断成readonly。
如果我们想把一个数组推断成字面量类型的元组 并且不希望他是一个只读的 我们就需要自己实现一个函数：
```typescript
function asTuple<const T extends unknown[]>(input: [...T]) {
    return input;
}

const _1 = asTuple([1, 'foo']);
//    ^? const _1: [1, "foo"]

const _2 = [1, 'foo'] as const;
//    ^? const _2: readonly [1, "foo"]
```
这里使用 `[...T]` 规定成元组类型 又使用const推断成字面量类型。

### 可选元素 
ts允许元组里面存在可选元素：
```typescript
declare function foo(arg1?: string, arg2?: number): void;

type _1 = Parameters<typeof foo>;
//   ^? type _1 = [arg1?: string | undefined, arg2?: number | undefined]
```
注意当 `strictNullChecks` 配置被打开时，TypeScript 会向可选元素的类型中附加一个 undefined 类型形成联合类型。
可选元素的存在会导致元组类型的长度虽然固定但不唯一。

## 模版字面量类型

类似于模版字符串 在js的模版字符串里面 我们可以嵌入变量 或者表达式 在ts的模版字面量类型里面 我们也可以嵌入字符串字面量类型：
```typescript
type Dance = "Dance";
type ByteDance = `Byte${Dance}`;
//   ^? type ByteDance = "ByteDance"
```
在这个例子里面 嵌入了一个字面量类型 我们还可以嵌入其他的类型：
- `string`
- `number`
- `bigint`
- `boolean`
- `undefined`
- `null`

还可以使用联合类型：
```typescript
type VerticalAlignment = "top" | "middle" | "bottom";
type HorizontalAlignment = "left" | "center" | "right";

type Alignment = `${VerticalAlignment}-${HorizontalAlignment}`;
//   ^? type Alignment = "top-left" | "top-center" | "top-right" | "middle-left" | "middle-center" | "middle-right" | "bottom-left" | "bottom-center" | "bottom-right"
```
于是 我们可以用模版字符串类型代替正则表达式的作用：
```typescript
// 检查是否以 Dance 结尾
type _1 = 'ByteDance' extends `${string}Dance` ? true : false;
//   ^? type _1 = true
```

## 映射类型 

在映射类型里面 我们可以使用as关键字来重命名构造的属性的名称：
```typescript
type Getters<T> = {
    [K in keyof T as `get${Capitalize<string & K>}`]: () => T[K]
};
```
通过 `string & K` 把除了字面量类型的其他键名映射成never从而删除。

## 枚举 

我们之前说到了 命名空间可以和枚举进行合并。
这就是因为 枚举的编译结果就是一个对象：
```javascript
var MyEnum;
(function (MyEnum) {
    MyEnum[MyEnum["Foo"] = 0] = "Foo";
    MyEnum[MyEnum["Bar"] = 1] = "Bar";
    MyEnum[MyEnum["Baz"] = 2] = "Baz";
})(MyEnum || (MyEnum = {}));
console.log(MyEnum.Baz);
```
但是常值枚举 也就是使用const进行修饰的枚举 将会直接把值内联到使用他的地方：
```typescript
const enum MyEnum {
  Foo, Bar, Baz
}
console.log(MyEnum.Baz);
```
编译结果是：
```javascript
console.log(2 /* MyEnum.Baz */);
```
但是对于普通的枚举 我们可以使用计算属性：
```typescript
enum FileAccess {
  // constant members
  None,
  Read = 1 << 1,
  Write = 1 << 2,
  ReadWrite = Read | Write,
  // computed member
  G = "123".length,
}
```
而常值枚举由于是直接内联值 使得它并不能在运行时进行计算 所以常值枚举无法使用计算成员。

在ts5.0之后 所有的枚举成员 包括计算类型 都成为了独立的类型。
则表明一个枚举类型在类型层面上 实际是他的各个枚举成员组成的联合类型 所以我们可以把枚举类型当成一个联合类型来使用：
```typescript
const enum MyEnum {
  Foo = 'foo', Bar = 'bar', Baz = 'baz'
}

type _0 = `${MyEnum}`;
//   ^? type _0 = "foo" | "bar" | "baz"
```
在这之前 对于计算类型 没有自己独立的字面量类型 而是会被拓宽：
```typescript
enum EventType {
  Click = 1,
  Submit = "submit".length, // 计算成员
}

// 在 TS 5.0 之前，EventType 的类型大致为 EventType.Click | number
// (字面量成员类型 | 计算成员拓宽后的类型)

type EventName = `event-${EventType}`; 

/**
 * 推断结果 (TS 5.0 之前):
 * type EventName = "event-1" | `event-${number}`
 */
```
`event-${number}` 这个类型意味着它可以匹配 `event-1`、`event-100`、`event-9999` 等任何以 `event-` 开头的、后缀为数字的字符串。这完全违背了我们使用枚举的初衷——我们只想要 `event-1` 和 `event-6` 这两个精确的类型，而不是一个宽泛的、可以匹配任意数字的模式。

但是 虽然在类型层面 一个枚举类型可以被理解成一个联合类型 却不可以这样使用：
```typescript
enum MyEnum {
  Foo = 'foo', Bar = 'bar', Baz = 'baz'
}

declare function myFunction(value: MyEnum): void;

// 我们希望用户可以这样传参：
myFunction(MyEnum.Foo); // 编译通过

// 我们也希望用户不必导入 MyEnum 就能传参
myFunction('foo'); // 编译不通过 :(
```
因为枚举的存在意义在于枚举成员的名字，而不是它的值。

但是我们可以把数字字面量直接赋值给数字枚举：
```typescript
enum MyEnum {
  Foo, Bar, Baz
}

declare function myFunction(value: MyEnum): void;

myFunction(MyEnum.Foo); // OK!
myFunction(0); // OK! <-- WTF?
```
并且 我们可以把任意数据赋值给数字类型的枚举 这是为了实现位编程。
就比如 我们使用位编程的方式 表示用户权限：
```typescript
enum FileAccess {
  // 使用位移运算符 (<<) 来创建 2 的幂次方，更具可读性
  None    = 0,      // 0 (二进制 000) - 没有权限
  Read    = 1 << 0, // 1 (二进制 001) - 读权限
  Write   = 1 << 1, // 2 (二进制 010) - 写权限
  Execute = 1 << 2, // 4 (二进制 100) - 执行权限
}

const permissions: FileAccess = FileAccess.Read | FileAccess.Write;
```
数字 3 是一个完全合法的权限组合（代表“读”和“写”都已开启），但 3 这个值本身并不是 FileAccess 枚举中任何一个已命名的成员（None, Read, Write, Execute 的值分别是 0, 1, 2, 4）。
因此，允许任何数字被赋值给数字枚举类型的这个“漏洞”或“副作用”，是一个刻意的设计权衡。这是为了完全支持位标志模式而付出的必要“代价”，而支持该模式正是 TypeScript 枚举的一个主要设计目标。

## 函数this的类型

在 TypeScript 的函数类型定义中，我们实际上可以提供一个特殊的 this 参数，通过指定它的类型，可以指定在这个函数中 this 的指向。此时，用户若传入箭头函数，它的 this 仍然指向全局空间。
特别地，可以通过将 this 的类型指定为 `never` 来阻止用户在函数中使用 `this`。

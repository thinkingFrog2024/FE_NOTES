# 工程化面试

###### lock文件的作用1

`package.json` 虽然定义了项目的依赖，但它存在**版本不确定性**问题，而 `package-lock.json` 正是为解决这个问题而设计的。

`package.json` 中的依赖版本通常使用**语义化版本范围**（SemVer Range），例如：

```json
{
  "dependencies": {
    "lodash": "^4.17.21",   // ^ 表示兼容主版本号的最新版本
    "react": "~18.2.0",     // ~ 表示兼容次版本号的最新版本
    "axios": "1.3.4"         // 固定版本
  }
}
```

这些符号的含义：

- `^4.17.21`：允许安装 `4.x.x` 的最新版本（但不包括 `5.x.x`）
- `~18.2.0`：允许安装 `18.2.x` 的最新版本（但不包括 `18.3.x`）
- `1.3.4`：固定版本，始终安装 `1.3.4`

当执行 `npm install` 时，npm 会根据这些范围**动态选择最新可用版本**。例如，今天安装 `^4.17.21` 可能得到 `4.17.21`，但一周后可能得到 `4.17.23`（如果期间发布了新版本）。

除了直接依赖的版本不确定，**间接依赖**（依赖的依赖）的版本也可能不同。例如：

- 项目依赖 `A@1.0.0`，而 `A@1.0.0` 依赖 `B@2.x.x`
- 当 `B` 发布 `2.1.0` 后，新安装的项目可能得到 `B@2.1.0`，而旧项目可能仍使用 `B@2.0.0`

这种差异会导致**不同环境下的依赖树结构不一致**，从而引发潜在的兼容性问题

###### webpak和vite和esbuild

1. Webpack

- **功能全面**：支持各种模块类型（JS、CSS、图片、字体等），通过 Loader 可处理几乎所有资源。
- **代码分割**：支持动态导入（Dynamic Import）和 SplitChunksPlugin，可按需加载代码。
- **插件系统**：丰富的插件生态，可实现 HMR（热更新）、Tree Shaking、代码压缩等。

1. Vite

- **极速冷启动**：基于浏览器原生 ES 模块（ES Module），无需打包即可启动开发服务器。
- **即时热更新**：HMR 速度极快，无需等待整个应用重新编译。
- **生产环境优化**：使用 Rollup 打包，支持 Tree Shaking、代码分割等优化。
- **配置简单**：预设了合理的默认配置，上手门槛低，配置文件更简洁。
- **原生支持 TS 和 JSX**：无需额外配置即可支持 TypeScript 和 JSX。

. esbuild

- **极快的构建速度**：使用 Go 语言编写，**利用多线程并行处理**，速度比传统打包工具快 10-100 倍。
- **支持多种功能**：支持 JS/TS 转译、CSS 处理、代码压缩等。
- **API 友好**：提供 JavaScript API，易于集成到其他工具中。
- **体积小**：单个可执行文件，无需安装依赖。
- **渐进式采用**：**可作为其他打包工具的预处理步骤**（如 Vite 在生产环境使用 esbuild 压缩代码）。

1. Rollup

- **ES 模块优先**：专注于 ES 模块的打包，天然支持 Tree Shaking。
- **轻量高效**：打包产物体积小，适合库和工具的开发。
- **插件生态**：提供丰富的插件，如处理 CommonJS、JSON、CSS 等。
- **配置简洁**：配置文件简单直观，易于理解和维护。
- **与 Webpack 互补**：Webpack 适合应用打包，Rollup 更适合库打包

为什么vite启动更快：

Webpack 是 **全量打包（Bundler）** 工具，启动时会：

- **从入口文件（如** **`index.js`）开始，递归解析所有依赖（包括** **`node_modules`** **中的第三方库）。**
- **将所有模块打包为一个或多个 bundle 文件（如** **`main.js`、`vendor.js`）。**
- **启动开发服务器，将打包后的文件提供给浏览器。**

Vite 是 **基于浏览器原生 ES 模块的构建工具**，启动时会：

- **第一步：启动开发服务器**（基于 `esbuild` 或 `koa`），不进行打包。
- 第二步：预构建第三方依赖

  （仅一次）：
  - 将 CommonJS/UMD 格式的第三方库（如 `node_modules` 中的 React）转换为 ES 模块（浏览器原生支持）。
  - 缓存预构建结果（默认在 `node_modules/.vite` 目录），下次启动直接使用。
- **第三步：按需加载模块**：浏览器请求哪个模块，Vite 就实时编译并返回哪个模块（如请求 `index.js`，Vite 才解析其中的 `import './App.js'`）。。Vite 利用 **浏览器原生 ES 模块** 和 **即时编译** 的特性，将 `.vue` 文件的编译时机推迟到 **浏览器请求时**，而非构建时。

Webpack 的 HMR 依赖 **打包后产物的更新**，流程为：

- 修改代码后，Webpack 重新编译变化的模块及其依赖（可能涉及多个模块）。
- 生成新的 bundle 文件，并通过 WebSocket 通知浏览器。
- 浏览器加载新的 bundle，执行 HMR Runtime 逻辑，替换旧模块。

一个模块修改可能触发其依赖链上所有模块的重新编译

Vite 的 HMR 基于 **原生 ES 模块的动态导入**，流程为：

- 修改代码后，Vite 仅重新编译变化的模块（不涉及依赖链）。
- 通过 WebSocket 发送 HMR 更新消息（包含模块 ID 和更新后的代码）。
- 浏览器执行 HMR 逻辑，动态导入（`import()`）新模块，替换旧模块。

在开发环境中，Vite 不会提前编译所有 `.vue` 文件，而是采用 **按需编译** 策略：

1. **浏览器请求** **`.vue`** **文件**
   当浏览器加载入口 HTML 并遇到 `<script type="module">` 导入 `.vue` 文件时（如 `import App from './App.vue'`），会直接向 Vite 开发服务器请求 `App.vue` 文件。
2. **Vite 拦截请求并解析** **`.vue`** **文件**
   Vite 开发服务器接收到请求后，会使用 `@vitejs/plugin-vue` 插件解析 `.vue` 文件的三个部分：
   - `<template>`：Vue 模板（HTML 语法）。
   - `<script>`：组件逻辑（JS/TS）。
   - `<style>`：组件样式（CSS / 预处理器）。
3. **即时编译为浏览器可执行的 ES 模块**
   - **`<template>`** **编译**：使用 `@vue/compiler-dom` 将模板编译为 **渲染函数**（如 `h('div', { id: 'app' }, ...)`），并注入到 `<script>` 导出的组件中。
   - **`<script>`** **处理**：若使用 TypeScript 或 JSX，通过 `esbuild` 即时转换为浏览器支持的 JS 语法（如将 `const App = defineComponent({})` 转换为 ES 模块）。
   - **`<style>`** **提取**：将 CSS 提取为单独的模块（如 `App.vue?vue&type=style`），通过 `<link rel="stylesheet">` 或 `<style>` 标签注入页面。
4. **返回编译后的 ES 模块给浏览器**
   Vite 将编译后的 JS 代码（包含渲染函数和组件逻辑）作为 ES 模块返回给浏览器，浏览器执行该模块，完成组件加载。

#### 2. **生产环境：构建时预编译**

在生产环境中，Vite 会使用 **Rollup** 进行打包，此时 `.vue` 文件会被 **预编译** 为最终产物：

1. **解析项目依赖图**
   Rollup 从入口文件（通常是 `index.html` 引用的 JS 文件）开始，递归解析所有依赖，包括 `.vue` 文件。
2. **编译** **`.vue`** **文件**
   `@vitejs/plugin-vue` 插件在打包阶段将 `.vue` 文件编译为纯 JS 代码：
   - `<template>`：编译为渲染函数（与开发环境相同，但会进行更多优化，如静态节点提升）。
   - `<script>`：转换为最终的 JS 语法（根据 `build.target` 配置，可能降级为 ES5）。
   - `<style>`：提取并压缩为 CSS 文件（如 `assets/index.123456.css`），或通过 JS 动态注入（取决于 `build.cssCodeSplit` 配置）。
3. **Tree Shaking 优化**
   Rollup 会分析编译后的代码，移除未使用的组件和逻辑（如未引用的组件、死代码），进一步减小包体积。
4. **生成最终产物**
   打包后的 `.vue` 文件会被合并到对应的 JS chunk 中（如 `index.123456.js`），最终产物不包含 `.vue` 文件，而是纯 HTML、JS、CSS。

###### webpackLoader的作用是什么

**Webpack 本身只能处理 JavaScript 模块，loader 扩展了其能力，使其可以打包任何类型的资源**

Loader 的本质是一个 **转换器**，它接收源文件内容作为输入，经过处理后返回新的内容。

1. 将 \*\*CSS、图片、字体等文件转换为 JS 模块，\*\*使 Webpack 能够打包它们
2. 将**高级语法（如 TypeScript、JSX）转换为浏览器兼容的 JS 代码。**
3. 对资源进行**压缩、优化或转换。**
4. 在处理过程中自动注入依赖

###### 图片是怎么变成js的？

图片本质上是二进制文件（如 JPG 是经过压缩的图像数据），而 JS 是文本代码。将图片转为 JS 的过程，实际上是 **将图片的二进制数据编码为文本格式**（如 **Base64 字符串或 URL 路径），然后包装为 JS 模块导出。**

###### es6怎么转换成es5

将 ES6（ES2015）代码转换为 ES5 代码的过程称为 **代码转译（Transpilation）**，主要目的是让现代 JavaScript 代码能在不支持新特性的旧版浏览器（如 IE11）或环境中运行。这个转换依赖 **转译工具**（如 Babel、TypeScript）和 **polyfill**（垫片）来实现。

**Babel 只能转换语法（如箭头函数、class）**，但无法自动补充 ES6 新增的 **全局对象**（如 `Promise`、`Map`）和 **实例方法**（如 `Array.prototype.includes`）。这些需要通过 **polyfill**（垫片）来实现。

```
core-js` 是最常用的 polyfill 库，`regenerator-runtime` 用于支持 `async/await
```

###### 基础组件 业务组件

基础组件是 **高度通用、无业务逻辑** 的原子级或分子级组件，通常封装 UI 交互和视觉效果，不依赖具体业务场景。
数据通常由父组件通过 `props` 传递，自身不管理业务状态高度可定制

业务组件是 **特定业务场景下的复合组件**，通常由多个基础组件组合而成，包含具体业务逻辑和数据处理。来自父组件传递、API 请求或本地状态管理（如 Vuex、Redux）

###### 保证组件稳定性

**功能正确性、性能可靠性、接口一致性、可维护性**

- **单一职责原则**：组件应只负责一个明确的功能（如按钮只处理点击交互，不包含业务逻辑）。
- **避免功能膨胀**：拒绝 “万能组件”，过度复杂会导致维护困难。例如，将表单验证逻辑封装在表单组件内，而非让按钮组件处理验证。
- **参数类型明确**：使用类型系统（如 TypeScript）定义 `props` 的类型、默认值和必填项，避免运行时错误。
- **空值处理**：确保组件在 `props` 为 `null`、`undefined` 或空字符串时正常工作（如显示默认值或占位符）。
- **大数据量**：测试组件在数据量过大时的性能（如长列表滚动、大量选项的下拉菜单）。

# 第七章：异步编程 (Asynchronous Programming)

---

## 1. Promise：承诺与背叛

### 1.1 基本概念
Promise 是异步编程的一种解决方案。
*   **状态**：Promise 对象的状态不受外界影响。只有异步操作的结果可以决定当前是哪一种状态（pending, fulfilled, rejected）。
*   **不可逆**：一旦状态改变，就不会再变。任何时候都可以得到这个结果。
*   **立即执行**：Promise 新建后就会立即执行。

### 1.2 基本用法
```javascript
const promise = new Promise(function(resolve, reject) {
  // ... some code

  if (/* 异步操作成功 */){
    resolve(value);
  } else {
    reject(error);
  }
});

promise.then(function(value) {
  // success
}, function(error) {
  // failure
});
```

### 1.3 执行顺序：节奏感 (The Rhythm)
听好了，你这个没有节奏感的蠢货。Promise 的执行顺序就像一场精心编排的性爱。

1.  **Executor (立即执行)**：`new Promise(executor)` 中的 `executor` 函数是**同步**执行的。
2.  **`.then()` (微任务 - Microtask)**：当 Promise 状态改变后，回调放入微任务队列。在当前同步代码执行完后执行。
3.  **`setTimeout` (宏任务 - Macrotask)**：下一轮事件循环。

```javascript
console.log("1. 前戏开始 (同步)");
const promise = new Promise((resolve, reject) => {
  console.log("2. 进入 Executor (同步)");
  resolve("Success"); 
  console.log("3. 还在 Executor 里 (同步)");
});
promise.then((value) => {
  console.log("5. .then() 回调 (微任务)");
});
console.log("4. 脚本结束 (同步)");
setTimeout(() => {
  console.log("6. setTimeout (宏任务)");
}, 0);
```

### 1.4 `resolve(promise)`：命运绑定 (The Bondage)
这不是简单的链式调用，这是**完全的服从**。
当你 `resolve(p1)` 时，当前 Promise 的状态完全由 `p1` 决定。

### 1.5 错误处理 (Error Handling)
1.  **吃掉错误 (Swallowing Errors)**：Promise 内部的错误不会让整个进程崩溃（虽然会有警告）。
2.  **`catch` 的恢复机制**：`catch` 处理完错误后，返回一个新的 resolved Promise，后续的 `.then` 会继续执行。
3.  **中断链条**：想中断？必须在 `catch` 里 `throw error`。

## 2. Promise API

### 2.1 `Promise.prototype.finally()`
不管成功还是失败，都会执行。不接受参数。总是返回原来的值（除非抛错）。

### 2.2 `Promise.all()`
*   **全都要**：接受一个数组。只有所有 Promise 都成功，才算成功。
*   **一票否决**：只要有一个失败，立刻失败。

### 2.3 `Promise.race()`
*   **竞速**：谁跑得快就听谁的（无论是成功还是失败）。

### 2.4 `Promise.allSettled()`
*   **我在乎结果，不在乎输赢**：等待所有 Promise 结束（无论成功失败）。
*   返回数组：`[{ status: 'fulfilled', value: ... }, { status: 'rejected', reason: ... }]`。

### 2.5 `Promise.any()` (ES2021)
*   **只要有一个赢家**：只要有一个成功，就成功。
*   **全员失败**：所有都失败，才抛出 `AggregateError`。

## 3. Generator 函数：控制权的交接

### 3.1 基本语法
```javascript
function* helloWorldGenerator() {
  yield 'hello';
  yield 'world';
  return 'ending';
}
var hw = helloWorldGenerator();
```
*   调用生成器函数返回一个**遍历器对象 (Iterator)**。
*   **分段执行**：`next()` 方法让函数从上次暂停的地方继续执行，直到遇到下一个 `yield`。

### 3.2 `next` 方法
*   **返回值**：`{ value: yield后面的值, done: boolean }`。
*   **参数注入**：`next(arg)` 可以把 `arg` 传回给**上一个** `yield` 表达式的返回值。
    *   这允许我们在函数外部控制函数内部的行为。

### 3.3 `throw` 与 `return`
*   `g.throw(err)`：在生成器内部抛出错误。
*   `g.return(val)`：提前终结生成器，并返回指定值。

### 3.4 `yield*`
*   **委托**：在一个生成器里调用另一个生成器。
*   常用于递归遍历（如二叉树扁平化）。

### 3.5 Generator 有什么用？(Why do we need this?)
你可能会问：“这玩意儿除了把代码搞复杂，还有什么用？”
**哼，浅薄。**

1.  **状态机 (State Machine)**：
    *   以前你需要用复杂的 `if/else` 或 `switch` 变量来记录状态。
    *   现在，生成器函数本身就“记住”了状态。
    *   比如红绿灯：`yield 'Red'` -> `yield 'Green'` -> `yield 'Yellow'`。无限循环，永不迷失。

2.  **无限序列 (Infinite Sequence)**：
    *   你可以写一个 `while(true)` 的生成器，产生无限的 ID 或斐波那契数列。
    *   **惰性求值**：虽然它是无限的，但只要你不调用 `next()`，它就不会计算。内存不会爆。

3.  **自定义迭代器 (Custom Iterator)**：
    *   想让你的对象能被 `for...of` 遍历吗？
    *   给它加个 `[Symbol.iterator]` 属性，指向一个 Generator 函数。简单优雅。

4.  **控制流管理 (Control Flow)**：
    *   这也是最重要的：它是 `async/await` 的基石。
    *   它允许我们**暂停**函数的执行，去干点别的（比如等一个 Promise），然后再**回来**。如果没有它，就没有现在的 `async/await`。

## 4. 异步编程进化史

### 4.1 回调地狱 (Callback Hell)
强耦合，难以维护。

### 4.2 Promise
链式调用，解决了缩进问题，但依然是一堆 `.then()`。

### 4.3 Generator + Co (协程：自动驾驶)

Generator 本身是**手动挡**（你需要自己按 `next()` 才会动）。
但在实际开发中，谁想每一步都手动按一下？我们想要**自动挡**（`async/await`）。

在 `async/await` 出现之前，聪明的程序员（比如 TJ Holowaychuk）发明了 `co` 模块。
**原理很简单：**
1.  **Yield Promise**：你在 Generator 里 `yield` 一个 Promise。
2.  **暂停**：Generator 自动暂停，把 Promise 扔给外面的执行器。
3.  **等待**：执行器等待 Promise 解决（resolve）。
4.  **推回**：一旦 Promise 解决了，执行器把结果通过 `it.next(result)` **塞回**给 Generator。
5.  **恢复**：Generator 拿到结果，继续执行，直到下一个 `yield`。

这就像是一个**接球游戏**：
*   **Generator**: "给你个球（Promise），我歇会儿。" (yield)
*   **Executor**: "好，我等着球变色（Resolve）... 好了！接着！" (next(data))
*   **Generator**: "谢了，我继续跑..."

```javascript
// 你的代码（看起来像同步的！）
function* myTask() {
  const data1 = yield fetch('url1'); // 暂停，等 fetch 完
  console.log(data1); 
  const data2 = yield fetch('url2'); // 再暂停
  console.log(data2);
}

// 自动执行器（简陋版 Co）
function run(gen) {
  const it = gen();
  function next(data) {
    const { value, done } = it.next(data); // 把上次的结果塞进去
    if (done) return;
    // value 是个 Promise，等它结束了，再调用 next
    value.then(res => next(res)); 
  }
  next(); // 启动！
}
```

### 4.4 Async/Await (终极形态)

**“语法糖” (Syntactic Sugar)** 这个词听起来很甜，但实际上它意味着：
`async` 函数的本质，就是 **Generator 函数 + 自动执行器 (Spawn)**。

如果我们把 `async` 函数这层皮剥掉，它的真面目是这样的：

```javascript
// 1. 我们平时写的 Async 函数
async function fn(args) {
  // ...
}

// 2. 剥掉糖衣后的真面目
function fn(args) {
  // spawn 是一个自动执行器
  return spawn(function* () {
    // ... 所有的 await 变成了 yield
  });
}
```

**核心：自动执行器 (Spawn) 的实现**
这个函数非常关键，它负责让 Generator 自动跑起来。
它的逻辑是：**递归调用 `next`，直到 `done` 为 true**。

```javascript
function spawn(genF) {
  // 返回一个 Promise，因为 async 函数总是返回 Promise
  return new Promise(function(resolve, reject) {
    const gen = genF(); // 启动生成器
    
    // 递归函数：负责一步步往下走
    function step(nextF) {
      let next;
      try {
        next = nextF(); // 尝试执行 gen.next()
      } catch(e) {
        return reject(e); // 如果生成器内部报错，直接 reject
      }
      
      // 如果生成器结束了 (return)，就 resolve 最终结果
      if(next.done) {
        return resolve(next.value);
      }
      
      // 如果没结束，就等待 yielded 的 Promise 完成
      // Promise.re solve 包裹一下，防止 yielded 的不是 Promise
      Promise.resolve(next.value).then(
        function(v) {
          // 成功了：把结果 v 传回给生成器，继续下一步
          step(function() { return gen.next(v); });
        }, 
        function(e) {
          // 失败了：把错误 e 扔回给生成器 (让 try...catch 捕获)
          step(function() { return gen.throw(e); });
        }
      );
    }
    
    // 启动第一步
    step(function() { return gen.next(undefined); });
  });
}
```

**总结**：
`async/await` 并没有什么魔法，它只是帮你自动执行了 Generator，并处理了 Promise 的回调。它是一个**自带司机的豪车**。

### 4.5 顶层 Await (Top-level Await)
允许在模块顶层直接使用 `await`，解决模块异步加载问题。
```javascript
const dynamic = await import(someMission);
export const output = someProcess(dynamic.default);
```

## 5. 最佳实践与坑

### 5.1 错误处理
*   `await` 命令最好放在 `try...catch` 代码块中。

### 5.2 并发执行
不要傻傻地串行：
```javascript
// 傻瓜写法 (串行)
let foo = await getFoo();
let bar = await getBar();

// 聪明写法 (并发)
let [foo, bar] = await Promise.all([getFoo(), getBar()]);
```

### 5.3 循环中的 Await (Sequence vs Parallel)

为什么 `forEach` 不行？
*   **`forEach` 是个无情的发令枪**：它只负责把你的回调函数（callback）发射出去。
    *   即使你的回调是 `async` 的，`forEach` **根本不在乎**。
    *   它调用了回调，拿到一个 Promise（因为 async 函数返回 Promise），然后直接丢掉，立刻跑去调用下一个。
    *   它**不会等待** Promise 完成。
    *   所以结果是：所有任务几乎同时开始（并发），而 `forEach` 这一行代码瞬间就执行完了。
    *   那时候根本就没有迭代器（ES6）。
它的实现非常简单粗暴：就是用一个 for (let i = 0; i < length; i++) 循环来跑下标，然后把每一项扔给你的回调函数。

*   **`for...of` 是耐心的等待者**：
    *   它本质上是迭代器（Iterator）消费。
    *   当你写 `await` 时，整个 `for` 循环的执行流程**真的暂停了**。
    *   它会等待当前的 Promise 解决（resolved），才会进入下一次循环。
    *   这就是所谓的**串行执行**。

```javascript
// forEach: 瞬间发射所有任务！
arr.forEach(async (item) => {
  await doSomething(item); // 这里的 await 只暂停了回调函数内部，没暂停 forEach 循环！
});
console.log("我先执行了！"); // 这行代码会比上面的任务先打印出来

// for...of: 一个接一个，乖乖排队
for (const item of arr) {
  await doSomething(item); // 真的暂停了整个循环
}
console.log("我最后执行！"); // 等所有任务都做完了才打印
```

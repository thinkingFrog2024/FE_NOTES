##### 组件库打包工具选型（npm/pnpm/yarn）的区别和技术考量

#### **1. 依赖管理机制**

| 维度           | npm（v7+）                | yarn（v2+）              | pnpm                                    |
| -------------- | ------------------------- | ------------------------ | --------------------------------------- |
| **依赖结构**   | 扁平化 + 嵌套（混合模式） | 严格扁平化               | 内容寻址（硬链接 + 符号链接（软链接）） |
| **重复依赖**   | 可能重复安装              | 尽可能扁平化去重         | 仅存储一份依赖副本                      |
| **依赖隔离**   | 弱隔离（嵌套可能冲突）    | 强隔离（通过 workspace） | 强隔离（每个项目独立视图）              |
| **锁文件格式** | package-lock.json         | yarn.lock                | pnpm-lock.yam                           |

若项目 A 依赖`lib@1.0`，项目 B 依赖`lib@2.0`，pnpm 会将两个版本存储在全局 store 中，通过符号链接让不同项目引用各自版本，而 npm/yarn 可能需要重复安装。

- **安装速度**：pnpm > yarn > npm
  pnpm 通过**硬链接复用**依赖（如 Node_modules/.pnpm 目录结构），减少磁盘 IO；yarn 利用缓存和并行安装优化；npm 早期版本因嵌套依赖导致性能较差，v7 后有所改进。
- **内存占用**：pnpm < yarn ≈ npm
  pnpm 的 store 机制避免重复加载依赖代码，内存占用更低。
- **更新效率**：pnpm > yarn > npm
  pnpm 只需更新变化的依赖，而 npm/yarn 可能重新安装整个依赖树。

硬链接指的是多个文件名指向同一个物理文件 

#### **1. 硬链接（Hard Link）**

在 Unix/Linux 文件系统中，每个文件有一个 inode（索引节点），包含文件的元数据（大小、权限、时间戳等）和数据块地址。硬链接通过创建新的文件名，使其指向同一个 inode：

```plaintext
# 创建硬链接示例
$ touch original.txt  # 创建原始文件
$ ln original.txt hardlink.txt  # 创建硬链接

# 查看inode信息
$ ls -i original.txt hardlink.txt
12345 original.txt  12345 hardlink.txt  # inode编号相
```

**特点**：

- 硬链接与源文件完全等价，无法区分。
- 删除任意一个链接，不会影响其他链接或文件内容。
- 文件系统通过 inode 引用计数（link count）管理文件存在性，当所有链接被删除，文件才真正被删除。

在传统 npm/yarn 中，依赖会被重复安装（即使版本相同），导致磁盘空间浪费。pnpm 通过**内容寻址存储（Content-Addressable Storage）** 和符号链接解决了这个问题：



1. **全局存储依赖**：将所有依赖安装到全局 store 目录（如`~/.pnpm-store`），每个版本只存一份。
2. **硬链接复用文件**：相同版本的依赖文件在 store 中只存储一次，不同项目通过硬链接共享。
3. **符号链接构建目录结构**：在项目的`node_modules`中，通过符号链接构建出符合 Node.js 模块解析规则的目录树。



软链接指的是新文件指向另一个文件的路径 也就是快捷方式  符号链接是一个特殊文件，其内容是另一个文件的路径（绝对或相对）。当访问符号链接时，系统会自动将路径解析到源文件：



为什么要这样？

因为传统的npm下载结构是把依赖安装到nodeModules里面 但是pnpm是把依赖下载到pnpm文件夹下 为了达到一样的使用效果 使用软链接创建一个传统的存储结构：


##### Monorepo架构

**Monorepo（单一仓库）** 是一种代码管理模式，指将多个项目或模块的代码存储在同一个代码仓库中，而非每个项目独立维护一个仓库（即 Multirepo 模式）

- 案例：Vue.js 仓库中，`vue-core`与`vue-router`可直接互相引用，无需通过 npm 版本控制。

```
│   ├── button/      # 按钮组件（子项目1）

        └── src/
        
        
//配置文件  pnpm-workspace.yaml
```

当你在子项目中声明对其他子项目的依赖时，包管理工具会自动创建符号链接，而非从 npm 安装

- **共享**：所有子项目共享根目录的`node_modules`缓存，避免重复安装相同依赖。
- **隔离**：每个子项目有独立的`package.json`，可定义自己的依赖版本（如`button`用 React 17，`input`用 React 18）。





技术实现：

1. 工作区支持 ：npm yarn ppnpm 都支持通过配置文件定义子项目目录
2. 依赖自动链接：工具会自动将工作区内的模块依赖解析为本地路径，无需 npm 安装
3. 构建工具集成：**Turbo**：基于缓存的高性能构建工具，比 Lerna 快 5-10 倍（如 Vite 团队使用）。
4. 版本管理策略
   1. **独立版本**：每个模块有独立版本号（如`@utils:1.0.0`、`@components:2.0.0`）。
   2. **单版本**：整个仓库共用一个版本号（如 React 仓库始终为`18.x.x`）。
   3. **工具**：`lerna version`、`np`等可自动化管理多模块版本。

##### 对事件冒泡和委托的理解

事件冒泡是 DOM 事件流的一种机制，指的是当一个元素接收到事件时，该事件会从触发元素开始，沿着 DOM 树向上传播，依次触发父元素、祖父元素，直到传播到`document`或`window`对象。并不是所有事件都支持冒泡 比如focus blur

DOM 事件流包含三个阶段（W3C 标准）：



1. **捕获阶段（Capture Phase）**：事件从`window`开始，向下传播到目标元素（默认不触发，需手动开启）。
2. **目标阶段（Target Phase）**：事件到达触发事件的元素。
3. **冒泡阶段（Bubbling Phase）**：事件从目标元素开始，向上传播到`window`。

**事件冒泡的优缺点**

- **优点**：天然支持事件委托，简化事件处理逻辑。
- **缺点**：可能导致意外触发父元素的事件（如多层嵌套的按钮和容器）。

事件委托是一种利用事件冒泡机制的事件处理模式，它将事件监听器绑定到父元素而非每个子元素，当子元素触发事件时，事件会冒泡到父元素，由父元素的监听器统一处理。



**核心思想**：事件冒泡 + `event.target`（指向实际触发事件的元素）。

#### **事件委托的优势**

- **减少内存占用**：无需为每个子元素创建监听器。
- **动态响应**：新添加的子元素自动获得事件处理能力（无需重新绑定）。
- **代码简洁**：集中管理事件逻辑，提高可维护性。





##### @submit.prevent=handleSubmit@submit.prevent=handleSubmit 的区别

- **`@submit.prevent="handleSubmit"`**：直接绑定方法引用，事件触发时自动传递原生事件对象（`$event`）。
- **`@submit.prevent="handleSubmit()"`**：绑定内联 JavaScript 表达式，事件触发时执行该表达式（调用方法），默认不传递事件对象（除非显式传入`$event`）。

Vue 在处理内联表达式（如`handleSubmit()`）时，会自动注入`$event`作为第一个参数（除非表达式中已显式使用`$event`）。因此，即使写法 2 中未显式传递`$event`，Vue 也会将事件对象传递给`handleSubmit`方法。

在vue的编译器进行处理的时候 会把指令解析成指令对象：

```javascript
// AST中的指令对象
{
  name: 'on',
  rawName: '@submit',
  value: 'handleSubmit', // 直接是方法名
  arg: 'submit',
  modifiers: { prevent: true }
}
// AST中的指令对象
{
  name: 'on',
  rawName: '@submit',
  value: 'handleSubmit()', // 内联JavaScript表达式
  arg: 'submit',
  modifiers: { prevent: true }
}
```

编译器会根据`value`是否包含括号`()`、运算符（如`+`、`&&`）等判断是否为内联表达式。

编译器将 AST 转换为 JavaScript 渲染函数（`render`方法），事件指令会被转换为`vnode.data.on`属性。

```javascript
// 简化后的渲染函数
render() {
  return createVNode('form', {
    on: {
      submit: [
        // 1. 处理.prevent修饰符：自动调用event.preventDefault()
        function preventDefault(e) {
          if (e && e.preventDefault) e.preventDefault();
        },
        // 2. 绑定方法引用
        this.handleSubmit
      ]
    }
  });
}
// 简化后的渲染函数
render() {
  return createVNode('form', {
    on: {
      submit: [
        // 1. 处理.prevent修饰符：自动调用event.preventDefault()
        function preventDefault(e) {
          if (e && e.preventDefault) e.preventDefault();
        },
        // 2. 内联表达式：包装为函数并自动注入$event
        function($event) {
          return this.handleSubmit($event); // 自动注入$event
        }.bind(this)
      ]
    }
  });
}
```

##### promise async await的关系：**从回调到 Promise 再到 Async/Await**

ES2017（ES8）引入 Async/Await，本质是**Promise 的语法糖**，通过`async`和`await`关键字，让异步代码的写法更接近同步代码

- 代码更符合直觉（按同步顺序编写，无需`.then()`链式调用）。
- 支持`try/catch`捕获异步错误，与同步代码错误处理方式一致。
- 调试更友好（可在`await`行打断点，与同步代码调试体验相同）。

`async`函数的返回值**自动封装为 Promise**，`await`关键字只能在`async`函数内部使用，用于 “等待” Promise 的结果（暂停函数执行，直到 Promise 解决）。

`await promise`的行为等价于`promise.then(result => result)`，但更简洁。

await会抛出promise被拒绝的错误 因此可以使用try catch进行捕获

##### promise是怎样解决回调地狱的 

Promise 之所以能够解决回调地狱（Callback Hell），本质上是通过**重构异步代码的控制流**和**统一状态管理**，将嵌套的回调结构转化为线性的链式调用

promiise解决回调嵌套的关键在于使用resolve和reject把成功的值和错误的原因返回给promise 这样子就不需要嵌套调用 并且得益于promise的错误状态传递 可以使用一个catch接受并且处理错误

1. **链式调用：通过返回值重构控制流**

Promise 的 `then()` 方法返回一个**新的 Promise**，该 Promise 的状态取决于回调函数的执行结果。这使得异步操作可以通过链式调用（`then().then().catch()`）按顺序 “平铺”，而非 “嵌套”。

2. **统一的状态管理：避免回调重复调用**

Promise 有三种状态（`pending`/`fulfilled`/`rejected`），且状态一旦改变就会 “冻结”，不会再次变化。这避免了回调函数可能被重复调用的问题（如网络请求成功后，回调不会被再次触发）。

3. **统一的错误处理：catch () 捕获全流程错误**

Promise 通过 `catch()` 方法捕获整个链式调用中的错误，无论错误发生在哪个 `then()` 中，都可在最后一个 `catch()` 中处理，避免了回调地狱中每个回调都需单独处理错误的问题。

4. **更灵活的组合能力：并行与竞争**

Promise 提供 `Promise.all()`、`Promise.race()` 等 API，支持并行或竞争执行多个异步操作，而回调函数难以实现此类组合逻辑。

##### 协商缓存和强缓存在在客户端上怎么实现

缓存是通过浏览器实现的 

1. **检查本地是否有缓存**：浏览器先查缓存数据库，若没有则请求服务端。

2. 验证缓存是否过期

   ：

   - 解析`Cache-Control: max-age=3600`，计算当前时间是否超过缓存有效期。
   - 若未过期，直接从本地读取资源（如图片、JS 文件），**不发送 HTTP 请求**。
   - 若过期，再检查`Expires`字段（HTTP 1.0 兼容策略），若仍过期则进入协商缓存流程

   若强缓存过期，客户端会主动向服务端发送带验证字段的请求：

   

3. **提取缓存中的标识**：从本地缓存中取出服务端之前返回的`ETag`或`Last-Modified`。

4. 添加条件请求头

   ：

   - 在请求头中携带`If-None-Match: "abc123"`（ETag 值）或`If-Modified-Since: Tue, 20 Jun 2023 00:00:00 GMT`。

5. 根据服务端响应处理

   ：

   - 若服务端返回`304 Not Modified`，客户端直接使用本地缓存，并更新缓存有效期。
   - 若返回`200 OK`，则使用新资源并覆盖本地缓存。

##### vue react的区别

Vue 的核心设计理念是**渐进式**（Progressive），即允许开发者根据项目需求逐步引入功能（如路由、状态管理），无需一次性采用所有特性。Vue 采用**声明式语法**，通过模板（Template）或 JSX 描述 UI 与数据的绑定关系，代码更直观、易读。

React 的核心设计理念是**函数式编程**（Functional Programming）和**组件化**（Component-Based）。React 将 UI 视为数据的函数（`UI = f(data)`），通过`render()`方法返回 JSX（JavaScript XML）描述 UI 结构，强调 “状态不可变” 和 “单向数据流”。

Vue 使用**Object.defineProperty()**（Vue 2）或**Proxy**（Vue 3）实现响应式系统，当数据变更时，自动追踪依赖并更新相关 DOM。Vue 的虚拟 DOM（Virtual DOM）更轻量，编译时会做静态分析（如标记静态节点），减少运行时比较开销。

React 使用**虚拟 DOM**作为中间层，当状态变更时，通过**Diff 算法**（协调算法）比较新旧虚拟 DOM 的差异，只更新需要变化的真实 DOM。React 16.8 引入**Fiber 架构**，将渲染过程拆分为多个阶段，支持中断和恢复，提升复杂应用的性能。

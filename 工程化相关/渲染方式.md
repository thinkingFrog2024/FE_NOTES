### 渲染模式

| 模式 | 描述 | 适用场景 |
| :--- | :--- | :--- |
| **CSR** | 在浏览器端执行 JavaScript 渲染页面 | 交互性强、对 SEO 要求不高的应用 |
| **SSR** | 在服务端预先渲染完整 HTML 页面 若在服务器渲染过程中报错 可以尝试降级成CSR（现代框架一般采用这个策略） | 对首屏性能和 SEO 要求高的网站 |
| **Streaming SSR** | 边渲染边返回，更快显示初始 UI | 需要更快首屏感知速度的应用（SSR 默认模式） |
| **RSC** | 组件在服务端渲染，减少客户端 JS 体积；数据与组件逻辑高内聚，减少状态传递 | 追求极致性能、需要减少客户端代码的项目<br>从表面来看 RSC和SSR有相似性 二者均是在服务端进行执行 能够更加快的访问数据。两者的区别在于：SSR只是在服务端提前运行一次并且返回得到的静态html，后续浏览器也需要请求js文件才可以让网页变成可交互的 但是RSC是不会产生对应的js的  是一个纯静态的 数据展示组件，如果一个组件不满足这个要求 就不应该使用这个方案 |
| **SSG** | 构建时生成静态页面，可被 CDN 缓存 | 内容相对静态的网站，如博客、文档站点 |

### 性能对比

| 渲染技术 | 核心优势 | 主要瓶颈 |
| :--- | :--- | :--- |
| **SSR** | 服务端预渲染，有利于 SEO | 需等待所有数据加载完成才能响应 |
| **Streaming SSR** | 边渲染边返回，更快首屏 | JS 体积仍然较大 |
| **Streaming SSR + RSC** | 减少客户端 JS 体积 | - |

### 选择策略

- **内容静态 + 需要 SEO** → SSG
- **需要 SEO + 首屏性能** → Streaming SSR（默认）
- **追求极致性能** → Streaming SSR + RSC
- **CSR 项目渐进优化** → CSR + RSC
- **高度交互 + 无需 SEO** → CSR




#### question part

##### 1. ssr既然是在服务端执行 那么代码里面依赖浏览器环境的part怎么处理？

1. 可以使用条件判断语句（如 `if (typeof window !== 'undefined')`）来判断是否在浏览器环境中执行。
2. 可以将依赖浏览器环境的代码放在客户端组件中，在服务端渲染时不执行。或是使用服务端同构的版本
3. 可以使用 React 提供的 `useEffect` 钩子函数，将依赖浏览器环境的代码放在其中，只有在浏览器环境中执行。（useEffect在ssr的时候不会执行是react的底层决定的 react在csr的时候 一次渲染经过的历程是：虚拟dom构建=>fiber树构建=>commit渲染=>从底向上执行useEffect 在ssr过程里面 是不存在commit阶段的 自然之后的useEffect也不存在了。从useEffect的作用来看：响应副作用，执行一次次难过的操作 等等 ，这些也不应该在ssr的过程中运行）
 - CSR（客户端） ：你调用的是 ReactDOM.createRoot(...).render(...) 或 hydrateRoot 。这玩意儿会走完整的 Render 阶段 和 Commit 阶段 。
- SSR（服务端） ：你调用的是 ReactDOMServer.renderToString(...) （或者流式 API）。这个函数的设计目的就是 把组件变成字符串 ，它 只执行 Render 阶段 ，生成 HTML 扔给浏览器就完事了
   

##### 2. 服务端流式渲染的优势是什么
1. 更快感知速度：流式渲染可以在渲染过程中逐步显示内容，能够以最快的速度显示业务首页
2. 更好的用户体验：通过流式渲染，用户可以更快地看到页面上的内容，而不需要等待整个页面都渲染完成后才能交互
3. 更好的性能控制：流式渲染可以让开发者更好地控制页面加载的优先级和顺序，从而更好地优化性能和用户体验
4. 更好的适应性：流式渲染可以更好地适应不同网络速度和设备性能，使得页面在各种环境下都能有更好的表现


##### 3. RSC的优势是什么
1. 零客户端 JavaScript：Server Components 的代码不会被打包到客户端，显著减少客户端 bundle 体积
2. 直接访问服务端资源：可以直接访问数据库、文件系统、内部 API 等服务端资源，无需额外的 API 层
3. 更好的性能：数据获取更接近数据源，减少客户端数据瀑布流，提升首屏加载速度
4. 自动代码分割：基于实际渲染的数据进行代码分割，而不仅仅是路由，实现更精细的代码优化
5. 更高的内聚性：与数据、权限、缓存等紧密相关的逻辑可以留在 Server Component 中，组件内聚度更高，减少状态上浮和跨层级传递

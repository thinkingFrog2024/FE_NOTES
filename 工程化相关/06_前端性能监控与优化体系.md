# 06_前端性能监控与优化体系

前端性能优化是一个从构建态到运行态，再到监控态的全链路闭环过程。本笔记总结了规范化的性能优化策略与监控方案。

---

## 1. 性能优化策略 (Performance Optimization Strategies)

### 1.1 构建与产物优化
构建优化的核心在于**并发**与**缓存**，而产物优化的核心在于**精简**与**合理拆分**。

- **构建提速**：
  - **并发处理**：使用多进程加载器（如 `thread-loader`）加速编译过程，分摊计算压力。
  - **持久化缓存**：在现代构建工具（如 Webpack 5 或 Rspack）中启用文件系统持久化缓存，大幅减少二次构建耗时。
  - **耗时监控**：通过构建插件分析各个阶段的编译耗时，针对性优化瓶颈。
- **产物拆分 (Chunk Splitting)**：
  - **公共包提取**：将 React、Polyfill、i18n 等基础库抽离为独立的公共包，利用浏览器缓存。
  - **路由级拆分**：基于动态导入（Dynamic Import）实现按需加载。
- **资源精简**：
  - **图片压缩**：通过图片压缩插件（如 `ImageCompressPlugin`）在构建阶段自动压缩静态资源。
  - **日志移除**：在生产环境配置移除 `console.log`，并剔除测试专用的属性（如 `data-testid`）。

### 1.2 资源加载优化
- **预连接与预加载**：
  - **Preconnect**：对 CDN 和后端域名进行预连接，提前完成 DNS 解析和 TCP 握手。
  - **Preload/Prefetch**：通过 `<link rel="preload">` 提前加载关键资源，或在空闲时预获取未来可能需要的资源。

## 2. 运行时优化 (Runtime Optimization)

运行时优化的关键在于**提前获取**与**空闲调度**。

### 2.1 数据预取 (Data Prefetching)
**Q: 为什么不在 React 组件挂载后再请求数据？**
**A**: 组件挂载后再请求会导致“请求瀑布流”，即：JS 加载 -> React 启动 -> 请求发送 -> 数据返回 -> 重新渲染。通过**首屏数据同步化**，可以在 HTML 阶段或路由匹配阶段提前注入数据。

```javascript
// 示例：首屏数据预取方案
// 1. 服务端或路由层提前获取数据并注入 window
window._APP_INITIAL_DATA = { user: {...}, config: {...} };

// 2. React 组件直接消费同步数据
function App() {
  const data = window._APP_INITIAL_DATA;
  // 无需等待 useEffect 中的 fetch
  return <div>{data.user.name}</div>;
}
```

### 2.2 空闲期调度 (Idle Scheduling)
利用浏览器空闲时间处理非关键任务，避免阻塞主线程。

- **智能预取钩子 (onIdle)**：
  不仅依赖 `requestIdleCallback`，还应结合业务生命周期。例如：在页面首屏渲染完成且用户开始产生交互后再执行预取。
- **组件延迟加载**：
  基于 `IntersectionObserver` 监听组件可见性，仅在进入视口或接近视口时加载相关资源。

## 3. 监控体系架构 (Monitoring Architecture)

### 3.1 SDK 设计与注入
为了捕捉到最真实的性能数据，监控脚本必须具备**高优先级**和**低侵入性**。

- **Inline 注入**：将核心监控逻辑（如资源错误捕获、首屏时间点计算）以 inline 脚本形式注入 HTML `head` 顶部。
- **插件化架构**：监控 SDK 采用插件化设计，不同业务按需开启“性能监控”、“错误上报”或“埋点分析”。

### 3.2 用户核心指标 (User-Centric Metrics)
除了常规的 FCP (First Contentful Paint) 和 LCP (Largest Contentful Paint)，还需要针对特定业务定义专项指标：

| 指标名称 | 描述 | 计算方式 |
| :--- | :--- | :--- |
| **TTV (Time to Visible)** | 用户看见核心内容的时间 | 使用 `PerformanceObserver` 监听特定元素的绘制。 |
| **TTU (Time to Usable)** | 用户可进行核心操作的时间 | 结合主线程空闲状态与业务逻辑 Ready 事件。 |

## 4. 专项监控场景 (Specialized Scenarios)

### 4.1 链路与实时连接监控
针对重度依赖长连接（如 WebSocket）的应用：
- **全链路追踪**：记录消息从发送到服务器响应的全链路耗时、成功率。
- **连接健康度**：监控 WebSocket 的重连频率、在线时长以及意外关闭的原因。

### 4.2 内存与复杂应用
针对文档编辑器、AI 交互等内存密集型场景：
- **内存泄漏监控**：定期采样 `performance.memory`，分析内存增量与 Diff。
- **交互卡顿分析**：监听 `Long Task`，并结合 Sourcemap 定位耗时最长的函数。

---

## 5. 总结与最佳实践 (Summary & Best Practices)

- **性能是功能的一部分**：在设计阶段就应考虑资源预取与按需加载。
- **数据驱动**：不要盲目优化，应通过监控数据发现真正的瓶颈（如 95 分位数的耗时）。
- **持续回归**：利用 CI 阶段的产物分析工具，防止性能劣化代码进入主干。

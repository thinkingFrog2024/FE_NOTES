## 一、React 核心原理

### 1.1 虚拟DOM (Virtual DOM)

#### 1.1.1 什么是虚拟DOM

虚拟DOM是真实DOM的JavaScript对象表示。它是React的核心概念之一，本质上是一个轻量级的JavaScript对象树。

```javascript
// 真实DOM元素
const realDOM = document.createElement('div');
realDOM.className = 'container';
realDOM.innerHTML = '<p>Hello</p>';

// 虚拟DOM表示
const virtualDOM = {
  type: 'div',
  props: {
    className: 'container',
    children: {
      type: 'p',
      props: {
        children: 'Hello'
      }
    }
  }
};
```

#### 1.1.2 为什么需要虚拟DOM

**核心优势：**

1. **批量更新（Batching）**: 将多次状态变更合并为一次DOM操作
2. **跨浏览器兼容**: 抽象了不同浏览器的差异
3. **声明式编程**: 开发者只需描述UI应该是什么样子，不需要关心如何更新
4. **Diff算法优化**: 通过对比找出最小变更集 diff本质上 其实是弥补虚拟dom带来的大量的dom操作

#### 1.1.3 虚拟DOM的工作流程

```
State/Props 变更 → 生成新的 Virtual DOM → Diff 对比旧 Virtual DOM → 计算最小变更 → 批量更新真实 DOM
```

**详细步骤：**

```javascript
// 1. 初始渲染
const oldVNode = render(<App />);

// 2. 状态变化触发重新渲染
setState({ count: count + 1 });

// 3. 生成新的Virtual DOM
const newVNode = render(<App />);

// 4. Diff算法对比
const patches = diff(oldVNode, newVNode);

// 5. 应用补丁到真实DOM
patch(domElement, patches);
```

### 1.2 Fiber 架构

#### 1.2.1 Fiber是什么

Fiber是React 16中引入的新协调引擎，它是对Stack Reconciler的重写。Fiber不仅是一个数据结构，还是一个工作单元。

**Fiber节点的数据结构：**

```javascript
function FiberNode(tag, pendingProps, key, mode) {
  // 实例属性
  this.tag = tag;                    // 组件类型标记
  this.key = key;                    // 唯一键值
  this.elementType = null;           // 元素类型函数/类
  this.type = null;                  // 函数/类本身
  this.stateNode = null;             // 真实DOM或组件实例

  // Fiber结构
  this.return = null;                // 指向父Fiber
  this.child = null;                 // 第一个子Fiber
  this.sibling = null;               // 下一个兄弟Fiber
  this.index = 0;                    // 在父节点中的索引

  this.ref = null;                   // ref引用

  // 新的props
  this.pendingProps = pendingProps;
  this.memoizedProps = null;         // 上次渲染使用的props

  // 更新队列
  this.updateQueue = null;
  this.memoizedState = null;         // 上次渲染使用的state

  // 副作用
  this.mode = mode;                  // 并发模式标志
  this.effects = null;               // 子树的副作用列表
  this.effectTag = NoEffect;         // 副作用类型标记
  this.deletions = null;             // 要删除的子节点

  // 调度相关
  this.lanes = NoLanes;              // 优先级车道
  this.childLanes = NoLanes;         // 子树的优先级车道

  // 双缓存
  this.alternate = null;             // 指向workInProgress或current fiber
}
```

#### 1.2.2 Fiber的双缓存机制

React使用双缓存技术来优化渲染过程：current 树是「只读的真相来源」，workInProgress 树是「可丢弃的草稿」。双缓存让 React 可以放心地在草稿上随意修改、随时中断、随时重来，而永远不会污染真相。最终只需 O(1) 切换指针就能让草稿"转正"

```javascript
// current树：正在屏幕上显示的Fiber树
// workInProgress树：正在构建中的新Fiber树

//这个函数的目的就是完成虚拟dom到fiber的转变 只是数据的一些基础的处理
function performUnitOfWork(fiber) {
  // 1. 开始阶段：处理当前fiber
  beginWork(fiber);

  // 2. 如果有子节点，返回子节点
  if (fiber.child) {
    return fiber.child;
  }

  // 3. 如果没有子节点，完成当前fiber
  let temp = fiber;
  while (temp) {
    completeWork(temp);

    if (temp.sibling) {
      return temp.sibling;
    }
    temp = temp.return;
  }
}
```

**双缓存的优势：**

- 避免在构建过程中直接修改current树：避免对current树的修改 可以避免
  - 在更新时若是直接更新更新current树 如果更新中途时间片用完了 那么更新将会被打断 此时current树就处于：一部分是新的 一部分是旧的
  - 直接在current上改的话 就不能回溯
- 可以随时中断和恢复工作
- 完成后直接切换指针，实现无缝替换
  - 直接修改指针的指向 时间复杂度1
  - 没有双缓存 就需要得到每一个更新之后的节点 再进行复制 时间复杂度n
  - 并且 旧的树在这个过程 并不会被销毁 而是通过alternate指针 和原来的树进行关联 这个就像算缓冲图形渲染 （后台缓冲区绘制下一帧 → 绘制完毕 → 交换前后缓冲区指针 → 显示器立即展示新帧）

#### 1.2.3 Fiber树的结构示例

#### 1.2.4 工作循环（Work Loop）深度解析

Fiber 的核心是**可中断的工作循环**，这是 React 实现并发渲染的基石。（react的并发渲染并不是🈯️在同一时有多个更新过程 而是吧原来的一个不可以打断的 长时间运行的任务 拆分成很多小小的片段 每个片段之间可以插入任务。这里的并发指的是多个任务交替执行 同时存在 ）

**从用户点击到时间切片生效 —— 完整调用链路（逐行追踪版）：**

整个协调过程涉及 **3 个包** 的协作：`React Core`（渲染）→ `Reconciler`（协调）→ `Scheduler`（调度）。下面用「函数调用栈」的方式一步步跟踪：

reactCore:定义公共api 组件模型

reconciler：fiber树的创建和协调

Scheduler:优先级队列 时间切片 messageChanel

<br />

Q：react怎么计算优先级？

A: react内有一个lane（车道）模型 ,这个模型其实就是一个位运算，来计算优先级

这是一个31位的二进制掩码，使用二进制进行运算 能够通过与运算快速找到优先级最高的任务，通过或运算合并所有的任务，以及判断包含关系。

其次 分配优先级 要看当前代码在什么环境下运行 ，这个要看触发的事件的类型 以及react内部维护的ExcutionContext的标识。

首先事件分成

离散事件：用户进行点击 按键 不能延迟

连续事件：拖拽 滚动 

&#x20;默认情况：setTimeout

过渡更新 ：使用了 startTransition 或 useTransition 包裹的更新、

<br />

其次ExcutionContext一共有：空闲 批处理 渲染 提交 同步渲染。表示的是react当前的工作状态。

这个标志的作用是：防止嵌套调用 导致的混乱

就比如：setState调用了 那么我们会认为：状态更新了 需要重新进行渲染 但是如果这个状态更新的时候 react正处于渲染更新状态？或者说react现在完全没有执行任何的任务？ 对于第一种情况 把这个setState添加进更新对列就可以了。 对于第二种任务 react还需要发起一个新的调度任务。

<br />

在进行优先级判断的时候 react会首先读取这个标志:

```
function requestUpdateLane(fiber) {
  // ...
  // 如果处于 Render 阶段，说明是渲染过程中产生的更新（例如在函数体里调 setState）

    return workInProgressRootRenderLanes; // 返回当前正在渲染的优先级
  }

}
```

**第 3 步：Scheduler 接管 —— 创建任务并注册宏任务**

```javascript
// scheduleCallback：Scheduler 包的核心函数
function scheduleCallback(priorityLevel, callback) {
  // 1️⃣ 获取当前时间
  const currentTime = getCurrentTime();  // 比如：1000.000ms

  // 2️⃣ 根据 priorityLevel 计算"过期时间"
  //    不同优先级的任务有不同的等待时限
  let timeout;
  switch (priorityLevel) {
    case ImmediatePriority: timeout = -1; break;    // 立即过期，不能延迟
    case UserBlockingPriority: timeout = 250; break; // 用户交互，250ms内必须处理
    case NormalPriority: timeout = 5000; break;     // 普通，5秒
    default: timeout = 5000;
  }
  const expirationTime = currentTime + timeout;
  // 假设 NormalPriority: expirationTime = 6000.000ms

  // 3️⃣ 创建任务对象
  const newTask = {
    id: taskIdCounter++,           // 唯一 ID: 42
    callback: callback,            // = performConcurrentWorkOnRoot 🔑
    priorityLevel: priorityLevel,  // = NormalPriority
    startTime: currentTime,        // = 1000.000ms
    expirationTime: expirationTime, // = 6000.000ms
    sortIndex: expirationTime,      // 用于最小堆排序
  };
  // 此时 newTask 还没被执行，只是被创建好了

  // 4️⃣ 把任务放入优先级队列（最小堆）
  push(taskQueue, newTask);
  // taskQueue 现在是: [ newTask ] （只有一个元素）
  //
  // 最小堆按 sortIndex 排序，值最小的在堆顶
  // 所以 expirationTime 最小的（最紧急的）任务会被先取出

  // 5️⃣ 📢📢📢 请求宿主环境安排一次宏任务！
  // isHostCallbackScheduled这个标志表示现在浏览器事件循环里面是否只有一个即将触发的flushwork
  // 如果不使用这个标志 假设调用一百次setState 那么触发一百次onMessage 
  if (!isHostCallbackScheduled) {
    isHostCallbackScheduled = true;
    requestHostCallback(flushWork);  // ← 关键调用！
  }
  // isHostCallbackScheduled = true（防止重复注册）

  return newTask;  // 返回任务对象（可用于后续取消等操作）
}
```

此时状态：

- 任务对象已创建，放在了 Scheduler 的优先级队列中
- **但还没有开始执行任何代码**
- 只做了一件事：调用了 `requestHostCallback(flushWork)`

**第 4 步：requestHostCallback —— 注册 MessageChannel 宏任务**

```javascript
// requestHostCallback：请求浏览器提供一个宏任务槽位
function requestHostCallback(callback) {
  // 把回调保存起来（稍后在 onmessage 中使用）
  scheduledHostCallback = callback;  // = flushWork 函数
  
  if (!isMessageLoopRunning) {
    isMessageLoopRunning = true;
    
    // 📢📢📢📢📢 这是最关键的一行！！！
    port.postMessage(null);
    //
    // postMessage 会做什么？
    // → 向 MessageChannel 的另一个端口发送一条消息
    // → 浏览器会把 port1.onmessage 回调
    //   放入"宏任务队列"
    // → 当前 JS 继续执行（不会立即触发 onmessage）
    // → 等"本轮事件循环"结束后，
    //   浏览器才会从宏任务队列取出 onmessage 并执行
    //
    // 这就实现了"推迟到下一轮事件循环"！
  }
}

// 对应的接收端（初始化时设置好的）
const channel = new MessageChannel();
const port = channel.port2;           // 发送端（我们用的这个）
const port1 = channel.port1;          // 接收端

port1.onmessage = function () {
  // 这个函数会在"下一轮事件循环"中被执行
  // 具体逻辑见第 6 步
};
```

此时状态（第 4 步结束）：

- `scheduledHostCallback` = `flushWork` 函数
- 宏任务已注册到浏览器队列
- **当前 JS 同步代码全部执行完毕**
- 用户的事件处理函数（onClick）返回
- 浏览器进入"事件循环的间隙期"

```javascript
// 从这里开始，React 正式接管主线程（但只占用一小段时间）
port1.onmessage = function () {

    // = flushWork（在第 4 步保存的）

    // ⏱️ 设置本帧的"截止时间"
    startTime = currentTime;              // = 1005.200ms
    deadline = currentTime + yieldInterval; // = 1010.200ms（+5ms）
    // yieldInterval 默认是 5ms
    // 意味着：这轮最多干 5ms 的活
    
    let hasMoreWork = true;
    try {
      // 📢 调用 flushWork，传入起始时间
      hasMoreWork = scheduledHostCallback(currentTime);
      // scheduledHostCallback 就是 flushWork
      // 返回值：true=还有活没干完 / false=全部干完了
      
    } finally {
      if (hasMoreWork) {
        // 还有任务没做完 → 再注册一个宏任务，下一轮继续
        port.postMessage(null);  // 又是这一行！形成循环
      } else {
        // 全部完成了 → 清理状态
        isMessageLoopRunning = false;
        scheduledHostCallback = null;
      }
    }
  }
};
```

**第 7 步：flushWork —— Scheduler 的任务分发中心**

当开始分发任务的时候 会把isPerforming打开=》 找到已经到期的任务 =〉找到这些任务里面优先级最高的任务=》进行执行=〉在执行过程中 如果时间到期且任务的过期时间还没到 这轮执行可以退出 这个任务留给下一轮 ，对于中途取消的任务 他们的callback属性将会被设置成undefined 不会执行。

```javascript
// flushWork：Scheduler 包的任务执行入口
// 职责：从优先级队列取任务 → 执行 → 处理结果
function flushWork(initialTime) {
  // 标记正在工作中
  isPerformingWork = true;
  
  try {
    // 合并 timerQueue 中已到期的任务到 taskQueue
    advanceTimers(initialTime);
    
    // 🔑 调用 flushWorkHelper 做真正的执行
    return flushWorkHelper(initialTime);
    
  } finally {
    isPerformingWork = false;
    
    // 如果还有剩余任务，再调度一轮
    if (scheduleHostCallbackIfNeeded !== null) {
      scheduleHostCallbackIfNeeded = true;
      // 不在这里发 postMessage，由 onmessage 的 finally 块负责
    }
  }
}
```

```javascript
// flushWorkHelper：真正从队列取任务并执行的函数
function flushWorkHelper(initialTime) {
  let currentTime = currentTime; // 初始时间
  
  // 从优先级队列（最小堆）取出最高优先级的任务
  let currentTask = peek(taskQueue);
  // currentTask = 第 3 步创建的那个 newTask 对象
  // 其 .callback = performConcurrentWorkOnRoot
  
  // 循环处理任务队列中的所有任务
  while (currentTask !== null) {
    
    // 检查 ① 是否该让出了
    if (currentTask.expirationTime > currentTime 
        && shouldYieldToHost()) {
      // 任务还没过期 且 时间片用完了
      // → 退出！不执行这个任务了，留给下一轮
      break;
    }
    
    // 检查 ② 任务有没有回调
    const callback = currentTask.callback;
    if (callback === null) {
      // 任务被取消了，直接移除
      pop(taskQueue);
      currentTask = peek(taskQueue);
      continue;
    }
    
    // ✅ 开始执行任务的回调！
    currentTask.callback = null;  // 先清空（防止重复执行）
    currentPriorityLevel = currentTask.priorityLevel;
    
    // 📢📢📢 执行回调！
    // callback = performConcurrentWorkOnRoot
    // 这一步会进入 React Core 的领域
    const continuationCallback = callback(
      currentTask.expirationTime   // 参数：过期时间
    );
    
    // 分析返回值：
    currentTime = getCurrentTime();  // 更新当前时间
    
    if (typeof continuationCallback === 'function') {
      // 🔑 返回了函数 → 任务还没完成！
      // 把返回的函数作为新的 callback 存回去
      // 下次继续从这个函数执行
      currentTask.callback = continuationCallback;
      
      // 为什么返回函数而不是直接完成？
      // 因为 workLoop 可能被 shouldYield 打断
      // 返回的函数包含了"恢复执行所需的所有上下文"
      
    } else {
      // 返回的不是函数（通常是 undefined 或 null）
      // → 任务彻底完成，从队列移除
      pop(taskQueue);
    }
    
    // 取下一个任务
    currentTask = peek(taskQueue);
  }
  
  // 返回：是否还有未完成的任务
  return currentTask !== null;
  // true  = 还有任务（onmessage 会再发一个 postMessage）
  // false = 全部完成（onmessage 会清理状态）
}
```

```javascript
  //    如果当前是高优先级更新（如用户点击），用同步模式（不可中断）
  //    如果是低优先级更新（如过渡更新），用并发模式（可中断）
  
  const originalCallbackNode = root.callbackNode;
  
  // 2️⃣ 执行渲染（renderRoot）

  if (root.callbackPriority === ImmediatePriority) {

    exitStatus = renderRootSync(root, lanes);
  } else {
    // 并发模式：可中断！🔑
    exitStatus = renderRootConcurrent(root, lanes);
  }
  
  // 3️⃣ 判断渲染是否完成
  if (exitStatus !== RootInProgress) {
    // 渲染已完成（或出错）
    
    // Commit 阶段：不可中断地执行 DOM 操作
    const finishedWork = root.current.alternate;
    commitRoot(root);
    
    // 返回 null 表示"任务完成，不需要恢复"
    return null;

    // 🔑 返回一个"续跑函数"
    // 下次 Scheduler 调用时，直接调用这个函数就能接着干
    return performConcurrentWorkOnRoot.bind(null, root);
  }
}
```

```javascript
// ⭐⭐⭐ workLoopConcurrent：时间切片的核心实现 ⭐⭐⭐
//
// 这个函数做的事情很简单：
//   while (还有节点要处理 && 该让出了吗?) {
//     处理一个节点;
//   }
//
// 但它的精妙之处在于：
//   - 每次循环只处理 1 个 Fiber 节点
//   - 每次循环都检查 shouldYieldToHost()
//   - 一旦应该让出，立刻退出，保留现场
//   - 下次进来时从退出的地方继续

function workLoopConcurrent() {
  // workInProgress 是一个全局变量
  // 指向当前正在处理的 Fiber 节点
  // 初始值 = 根 Fiber（在 prepareFreshStack 中设置的）
  
  while (workInProgress !== null && !shouldYieldToHost()) {
    // ┌──────────────────────────────────────────┐
    // │ 条件1: workInProgress !== null           │
    // │   还有节点没处理完                        │
    // │                                          │
    // │ 条件2: !shouldYieldToHost()              │
    // │   时间还没到，不需要让出                   │
    // │                                          │
    // │ 两个条件都满足 → 继续干活                 │
    // │ 任一条件不满足 → 退出循环                │
    // └──────────────────────────────────────────┘
    
    // 处理当前节点，返回下一个要处理的节点
    workInProgress = performUnitOfWork(workInProgress);
    //
    // performUnitOfWork 内部：
    //   1. beginWork(current, wip)  → "向下进入"
    //      - 处理当前节点
    //      - 创建/复用子 Fiber 节点
    //      - 返回第一个子节点（或 null）
    //   
    //   2. 如果 beginWork 返回了子节点
    //      → 循环继续，下一步处理那个子节点
    //   
    //   3. 如果 beginWork 返回 null（叶子节点）
    //      → 调用 completeUnitOfWork
    //      → completeWork（向上回溯）
    //      → 处理兄弟节点或回到父节点
    //      → 返回下一个要处理的节点（或 null）
    //
    // 每次 performUnitOfWork 大约耗时 0.1~2ms
    // （取决于组件复杂度）
  }
  
  // 循环退出时：
  // - 要么 workInProgress === null（整棵树处理完毕）✅
  // - 要么 shouldYieldToHost() === true（时间到了，被打断）⏰
  //
  // 如果是被打断的：
  //   workInProgress 停留在当前节点
  //   所有局部变量都保存在内存中
  //   下一轮宏任务触发时，直接进入这个 while 循环
  //   从 workInProgress 指向的节点继续！
}
```

**第 10 步：shouldYieldToHost —— 让出判断器**

```javascript
// shouldYieldToHost：判断"现在是不是该停下来，把控制权还给浏览器了"
//
// 这个函数会在 workLoopConcurrent 的每次循环中被调用
// 也就是说：每处理完 1 个 Fiber 节点，都会问一次这个问题
//
// 返回 true  → 该让出了！workLoop 立即退出
// 返回 false → 还能继续！workLoop 继续处理下一个节点

function shouldYieldToHost() {
  // 1️⃣ 第一关：计算已经花了多长时间
  const timeElapsed = getCurrentTime() - startTime;
  // startTime 是 onmessage 触发时记录的（第 6 步）
  // 比如：currentTime = 1008.500ms, startTime = 1005.200ms
  // timeElapsed = 3.3ms
  
  if (timeElapsed < yieldInterval) {
    // 还不到 5ms → 不需要让出，继续干活
    return false;  // ⚡ workLoop 继续
  }

  // 到了这里说明已经 >= 5ms 了
  // 但还不急着让出，再做几个检查...

  // 2️⃣ 第二关：浏览器是否需要绘制？
  if (needsPainting()) {
    // React 通过一些启发式方法检测浏览器是否准备绘制
    // 如果是 → 必须让出，否则掉帧
    return true;  // ⏸️ workLoop 退出
  }

  // 3️⃣ 第三关：是否有输入事件等待处理？
  if (hasInputEvent()) {
    // 检测是否有 click/keypress/scroll 等事件在排队
    // 如果是 → 必须让出，否则用户感觉卡顿
    return true;  // ⏸️ workLoop 退出
  }

  // 4️⃣ 第四关：纯超时
  // 以上都没触发，那就是纯粹的时间到了
  return timeElapsed >= yieldInterval;  // >= 5ms → true
}
```

**performUnitOfWork 的两阶段处理 —— beginWork 与 completeWork**

`performUnitOfWork` 是工作循环中处理单个 Fiber 节点的核心函数，它将每个节点的处理拆分为两个阶段。这种「向下进入、向上完成」的设计对应了深度优先遍历的**递进过程**和**回溯过程**

<br />

<br />

<br />

<br />

### beginWork —— 「向下进入」阶段

`beginWork` 在**首次到达某个 Fiber 节点时**被调用，负责**创建或复用子节点**。可以理解为"进入一个房间，看看里面有什么"。它主要做以下事情：

1. **判断是否可以跳过（Bailout 优化）**：对比当前节点的旧 props（来自 current 树的 `memoizedProps`）和新 props（`pendingProps`），如果 props、context （通过createContext创建的上下文）都没有变化 且子树也没有待处理的更新，直接复用旧的子节点引用，跳过整棵子树的重建 ，简单的来说 判断的条件有三个 props state/queue(更新队列 setState执行之后 状态并不会立即更新 而是在队列进行排队 这个派对的队列 在数据结构层面 是一个环形链表 如果判断链表里面没有数据 那么这个检查通过) context
2. **根据组件类型分发处理逻辑**：通过 `workInProgress.tag` 判断节点类型——函数组件会执行函数体拿到返回的 React 元素、类组件会调用实例的 `render()` 方法、原生 DOM 节点（HostComponent）则处理属性差异、文本节点直接处理内容更新
3. **将返回的 React 元素转化为子 Fiber 节点**（调和过程 reconcileChildren）：对比新旧子元素列表，决定哪些节点可以复用、哪些需要新建、哪些需要标记删除
4. **返回第一个子 Fiber 节点**：让工作循环继续向树的下层深入遍历；如果该节点没有子节点（叶子节点），返回 `null`

简单来说，beginWork 的产出是：**当前节点的子 Fiber 树结构已经构建好了**。

### completeWork —— 「向上回溯」阶段

当 `beginWork` 返回 `null`（说明到达了叶子节点或没有更多子节点），工作循环转入 `completeUnitOfWork`，其中会调用 `completeWork`。这个阶段在**一个节点的所有后代都处理完毕后**被调用，负责**收尾和收集副作用**。可以理解为"离开一个房间前，把该收拾的都收拾好"。它主要做以下事情：

1. **根据节点类型执行具体的「完成」逻辑**：
   - **HostComponent（DOM 节点）**：创建或复用真实的 DOM 实例（`stateNode`），处理样式、事件绑定等属性更新，将 DOM 实例挂载到父 DOM 节点上
   - **HostText（文本节点）**：创建或更新文本内容的 DOM 实例
   - **FunctionComponent / ClassComponent**：处理 ref 的挂载/更新，处理 useLayoutEffect 等同步副作用
   - **其他类型**（如 Fragment、Portal 等）**：执行对应的收尾逻辑**
2. **标记副作用标志位（flags）**：根据 Diff 结果给当前节点打上标记——新增的节点打上 `Placement`（插入）、属性变化的打上 `Update`（更新）、需要删除的打上 `Deletion`
3. **构建 effectList 副作用链表**：将当前节点的副作用信息以单向链表的形式拼接到父节点的 effectList 上。这样 Commit 阶段只需要从头到尾遍历这条链表，就能按顺序执行所有需要 DOM 操作的节点，无需再遍历整棵 Fiber 树
4. **处理兄弟节点回溯**：如果当前节点还有兄弟节点（sibling），就转向兄弟节点继续 beginWork；如果没有兄弟节点了，就一直向上回溯到父节点，直到根节点

Q: 在completa的过程中进行真实dom的创建和挂载？ 可是当一个fiber进入complete阶段的时候 它的父节点必然没有complete 也就是说 它的父节点必然不存在真实dom节点 这样怎样添加dom节点呢

A:是的 所以react的解决方案是：不在complete里面执行添加当前dom到父节点 而是把当前的节点的子fiber的dom元素append进来

Q: 副作用列表effectList是什么？

A:render结束之后 其实一整颗fiber树 应该只有少部分节点发生了改变 

但是如果commit 也就是把最终的生成的树映射到视图上面的时候 再进行对比 耗时十很大的

所以react会在completework阶段就把所有有副作用的节点用链表串起来

```
// 每个 Fiber 节点上有这些字段（来自文档 [01-核心原理.md#L490-L493](file:///Users/bytedance/Downloads/弱智学习日记/code-react/01-核心原理.md#L490-L493)）
class FiberNode {
  flags = NoFlags;           // 当前节点的副作用标记（Placement/Update/Deletion...）
  
  // 子树的副作用链表（当前节点的"收集成果"）
  firstEffect = null;        // 指向子树中第一个有副作用的节点
  lastEffect = null;         // 指向子树中最后一个有副作用的节点
  
  nextEffect = null;         // 全局链表中的下一个节点（串联用）
}
```

举个例子：

假设这棵树在这次更新中有这些变化：

```
         App (无变化)
        /    \
      A(新增!)  B(无变化)
             /   \
          C(属性变了!)  D(删除!)
```

标记：A=Placement, C=Update, D=Deletion, 其余=NoFlags

处理 A 自身的 effect：
A 有副作用！把它加入链表

A 的 effectList:
firstEffect ──→ A ──→ lastEffect
A.nextEffect = null

简单表示：\[A]

C.flags = Update（属性更新）

C 的 effectList:
firstEffect ──→ C ──→ lastEffect

简单表示：\[C]

D.flags = Deletion（删除）

D 的 effectList:
firstEffect ──→ D ──→ lastEffect

简单表示：\[D]

<br />

B 本身没有变化（flags = NoFlags），但它的 子树里有 C 和 D 有副作用 ！

<br />

<br />

```
B 开始 completeWork 时：
  B 自己的 flags = NoFlags（B 没变）

但 B 要检查自己的子节点 C 和 D：
  
  处理完 C 后：C 的 effectList = [C]
  处理完 D 后：D 的 effectList = [D]

现在回到 B 的 completeUnitOfWork，开始合并：

  completedWork = D（刚完成的子节点）
  
  D 有副作用（flags = Deletion）：
    ① 把 D 的 effectList 接到 B 上
    
    B.firstEffect 原来是 null（还没接过任何东西）
    → B.firstEffect = D.firstEffect = D
    → B.lastEffect = D.lastEffect = D
    
    此时 B 的 effectList: [D]

  然后回溯到 C（D 的兄弟？不，C 是先处理的）
  
  实际顺序是：C 先完成 → D 后完成 → 回溯到 B
  
  当处理 C 时（C 已经完成了，它的 effectList=[C]）：
    C 有副作用（flags = Update）
    
    B.lastEffect 已经是 D 了（不是 null）
    → B.lastEffect.nextEffect = C.firstEffect = C   （D 的后面接上 C）
    → B.lastEffect = C.lastEffect = C              （更新尾指针）
    
    此时 B 的 effectList: [D] → [C]
    
  B 自己没有额外副作用（flags = NoFlags），不需要把自己加进去

最终 B 的 effectList:  D → C → null
                        ↑           ↑
                   firstEffect   lastEffect
```

<br />

```
App 开始合并：

  先处理了 A：A 的 effectList = [A]
  再处理了 B：B 的 effectList = [D → C]

  合并 A 到 App:
    App.firstEffect = null → App.firstEffect = A
    App.lastEffect = A
    
    App: [A]

  合并 B 到 App:
    App.lastEffect(=A).nextEffect = B.firstEffect(=D)   // A 后面接 D
    App.lastEffect = B.lastEffect(=C)                     // 尾变成 C
    
    App: [A → D → C]
```

最终：

```
                    App
                   /   \
                 A      B
                       / \
                      C   D

effectList（从 root.firstEffect 开始遍历）:

  root.firstEffect ──→ A ──→ D ──→ C ──→ null
  (Placement)      (Deletion)(Update)
       ↑                          ↑
  root.firstEffect          root.lastEffect

Commit 阶段只需遍历这条链：
  1. A → 执行插入 DOM
  2. D → 执行删除 DOM
  3. C → 执行更新属性

✅ 不需要遍历整棵 Fiber 树！
```

这样的架构 允许我们在commit的时候 只需要获取根节点的副作用链表 进行遍历。

简单来说，completeWork 的产出是：**真实 DOM 已经准备好（或已标记好变更），所有副作用已串联成一条链表等待 Commit 阶段消费**。

我们可以把遍历看作是一个“找路”的过程：

- workLoop (主驱动轮) ：负责“走下一步”。它不断调用 performUnitOfWork 。
- performUnitOfWork (向导) ：它负责调用 beginWork 。
- beginWork (探路者) ：它的职责是“向下走一步”。它只处理当前节点，并返回 第一个子节点（child） 。
- completeUnitOfWork (回溯者) ：它的职责是“向上或向右走”。当没有子节点时，它负责调用 completeWork ，然后找 兄弟节点（sibling） ；如果也没兄弟，就回退到 父节点（return） 。

```
    App
     |
   Header
   /    \
Title  Nav
```

执行流向（DFS 顺序）：

1. workLoop -> performUnitOfWork(App)
2. beginWork(App) -> 产出 Header -> 返回给 workLoop
3. beginWork(Header) -> 产出 Title -> 返回给 workLoop
4. beginWork(Title) -> 没有子节点了 -> 返回 null
5. completeUnitOfWork(Title) ：
   - 执行 completeWork(Title)
   - 发现 Title 有兄弟 Nav -> 返回 Nav
6. beginWork(Nav) -> 没有子节点了 -> 返回 null
7. completeUnitOfWork(Nav) ：
   - 执行 completeWork(Nav)
   - 发现 Nav 没兄弟了 -> 回到父节点 Header
   - 执行 completeWork(Header)
   - 发现 Header 也没兄弟了 -> 回到父节点 App
   - 执行 completeWork(App)

### 两阶段的协作关系

```
        App (root)
       /    \
     A      B
          /   \
         C     D

遍历顺序与两阶段交替执行的过程：

 1. beginWork(App)   → 处理App，生成A、B子节点 → 返回A
 2. beginWork(A)     → 处理A，A无子节点 → 返回null
 3. completeWork(A)  → A是叶子，收尾A，收集A的副作用 → 回到App
 4. 没有兄弟了？→ 回到App层级的兄弟B
 5. beginWork(B)     → 处理B，生成C、D子节点 → 返回C
 6. beginWork(C)     → C无子节点 → 返回null
 7. completeWork(C)  → 收尾C，收集C的副作用 → 回到B
 8. beginWork(D)     → D无子节点 → 返回null
 9. completeWork(D)  → 收尾D，收集D的副作用 → 回到B
10. B的所有子节点完成 → completeWork(B) → 收尾B，合并C/D的effectList到B
11. 回到App → App的所有子节点完成 → completeWork(App) → 整棵树完成！
```

```javascript
function performUnitOfWork(unitOfWork) {
  const current = unitOfWork.alternate;

  // ========== 阶段一：beginWork（向下遍历） ==========
  // 根据组件类型调用不同的处理函数
  let next = beginWork(current, unitOfWork, renderLanes);

  unitOfWork.memoizedProps = unitOfWork.pendingProps;
  if (next === null) {
    // 如果没有子节点，进入完成阶段
    next = completeUnitOfWork(unitOfWork);
  }

  return next;
}

// completeUnitOfWork：向上回溯，完成当前节点及其兄弟/父节点
function completeUnitOfWork(unitOfWork) {
  let completedWork = unitOfWork;
  do {
    const current = completedWork.alternate;
    const returnFiber = completedWork.return;

    // ========== 阶段二：completeWork（节点完成） ==========
    // 根据 tag 类型执行不同的完成逻辑
    completeWork(current, completedWork, renderLanes);

    // 收集副作用到 effectList
    if (returnFiber !== null && (returnFiber.flags & ChildDeletion) === NoFlags) {
      if ((completedWork.flags & Deletion) !== NoFlags) {
        returnFiber.flags |= Deletion | ChildDeletion;
      }
      if ((completedWork.flags & Placement) !== NoFlags) {
        if (returnFiber.flags & Placement) {
          // 父节点已经有Placement标记，不需要重复标记
        } else {
          returnFiber.flags |= Placement;
        }
      }

      // 将当前节点的副作用链表挂载到父节点的 effectList 上
      if (completedWork.effectTagList !== null) {
        if (returnFiber.lastEffect === null) {
          returnFiber.firstEffect = completedWork.firstEffect;
          returnFiber.lastEffect = completedWork.lastEffect;
        } else {
          returnFiber.lastEffect.nextEffect = completedWork.firstEffect;
          returnFiber.lastEffect = completedWork.lastEffect;
        }
      }

      // 处理自身的 effect
      if (completedWork.flags & (Placement | Update)) {
        if (returnFiber.lastEffect === null) {
          returnFiber.firstEffect = completedWork;
          returnFiber.lastEffect = completedWork;
        } else {
          returnFiber.lastEffect.nextEffect = completedWork;
          returnFiber.lastEffect = completedWork;
        }
        completedWork.nextEffect = null;
      }
    }

    const siblingFiber = completedWork.sibling;
    if (siblingFiber !== null) {
      // 有兄弟节点，返回兄弟继续处理
      workInProgress = siblingFiber;
      return siblingFiber;
    }
    // 没有兄弟了，回到父节点
    completedWork = returnFiber;
    workInProgress = completedWork;
  } while (completedWork !== null);

  // 返回null表示整棵树处理完毕
  return null;
}
```

**Commit 阶段的运行逻辑：**

Commit 阶段是 React 将内存中计算好的 Fiber 树变更同步到真实 DOM 的过程。与 Render 阶段不同，Commit 阶段是**同步且不可中断**的，以确保 UI 的一致性。

整个 Commit 阶段可以划分为三个核心子阶段，其执行逻辑紧密围绕“DOM 变更”这一关键时刻展开。

### 1. Before Mutation 阶段（DOM 变更前）
这是在实际修改 DOM 之前的最后准备阶段。
- **任务 A：处理快照**。调用类组件的 `getSnapshotBeforeUpdate` 生命周期方法，允许开发者在 DOM 改变前捕获信息（如滚动位置）。
- **任务 B：异步调度 Effect**。通过调度器异步预约 `useEffect` 的回调。注意，此时 `useEffect` 还没执行，只是“预约”了在浏览器渲染后的某个时刻执行。

### 2. Mutation 阶段（真正的 DOM 操作）
这是最核心的阶段，React 会遍历副作用链表（Effect List），根据标记执行具体的 DOM 操作。
- **插入/移动 (`Placement`)**：调用 `appendChild` 或 `insertBefore`。React 会智能地寻找正确的“兄弟节点”作为参照物，以最小化 DOM 位移。
- **更新 (`Update`)**：将 Render 阶段计算出的属性差异（Props Diff）应用到真实 DOM 上（如修改 style、className、事件绑定等）。
- **删除 (`Deletion`)**：将标记为删除的节点从 DOM 树中移除。这是一个递归过程，会触发子组件的 `componentWillUnmount` 并清理相关的 Ref。

### 3. 双缓存切换（current 指针切换）
在 Mutation 阶段结束后，Layout 阶段开始前，执行：`root.current = finishedWork`。
- **逻辑意义**：从这一刻起，React 正式认为“新树”变成了“当前树”。
- **必要性**：确保在接下来的 Layout 阶段执行生命周期钩子时，组件内部通过 `this` 或 `ref` 访问到的都是已经更新后的最新数据。

### 4. Layout 阶段（DOM 变更后）
此时 DOM 已经更新完毕，但浏览器尚未进行绘制（Paint），页面看起来还是旧的。
- **任务 A：执行同步副作用**。同步触发 `useLayoutEffect` 的回调。因为此时 DOM 已经是新的了，开发者可以安全地读取布局信息（如 `offsetWidth`）而不会导致页面闪烁。
- **任务 B：执行挂载/更新钩子**。触发类组件的 `componentDidMount` 或 `componentDidUpdate`。
- **任务 C：绑定 Ref**：将真实的 DOM 实例或组件实例赋值给 `ref.current`。

```javascript
function commitRoot(root) {
  const { finishedWork } = root;
  
  if ((finishedWork.flags & Marked) === NoFlags) {
    // 没有任何副作用需要处理
    commitBeforeMutationEffects_begin();
    
    // ========== 第一阶段：Before Mutation ==========
    // DOM变更前执行（getSnapshotBeforeUpdate、useLayoutEffect清理等）
    commitBeforeMutationEffects(finishedWork);
    
    // ========== 第二阶段：Mutation（真正的DOM操作）==========
    // 这个阶段会实际修改DOM
    commitMutationEffects(finishedWork);
    
    // 切换current指针（双缓存切换的关键时刻！）
    root.current = finishedWork;
    
    // ========== 第三阶段：Layout（DOM变更后）==========
    // 可以安全地读取DOM布局信息
    commitLayoutEffects(finishedWork);
  }

  // 重置状态
  root.current.finishedWork = null;
}
```

**Mutation 阶段的详细实现：**

```javascript
function commitMutationEffects_begin(root) {
  while (nextEffect !== null) {
    const flags = nextEffect.flags;

    // 处理删除
    if (flags & ContentReset) {
      commitResetTextContent(nextEffect);
    }
    if (flags & Ref) {
      markRef(nextEffect);
    }
    if (flags & Deletion) {
      const prevSibling = nextEffect.sibling;
      const nextSibling = nextEffect.nextEffect;
      // 执行删除操作
      commitDeletion(root, nextEffect, nearestMountedAncestor);
      nextEffect = prevSibling || nextSibling;
      continue;
    }

    // 处理插入/移动
    const isParent = (flags & Placement) !== NoFlags;
    const isContentReset = (flags & ContentReset) !== NoFlags;
    const isCallback = (flags & Callback) !== NoFlags;

    if (isParent || isContentReset || isCallback) {
      // 执行实际的DOM操作
      commitReconciliationEffects(nextEffect);
    }

    // 处理更新
    if (flags & Update) {
      const current = nextEffect.alternate;
      commitWork(current, nextEffect);
    }

    nextEffect = nextEffect.nextEffect;
  }
}
```

#### 1.2.6 调度器与时间切片原理

React 的调度器（Scheduler）是一个独立于 React Core 的包，负责管理任务的优先级调度和时间切片。

当我们说协调过程 指的是render阶段里面 计算fiber树变化的整个过程 这个过程是可以中断的。

不可中断的协调完成之后的commit

**调度器底层 API 选型对比：**

React Scheduler 需要一个「能产生宏任务」的机制来把调度回调推迟到下一个事件循环执行。以下是各方案的对比：

| 方案                           | 类型       | 最小延迟                 | 触发可靠性      | 标签页不活跃时    | 浏览器支持                      |
| ---------------------------- | -------- | -------------------- | ---------- | ---------- | -------------------------- |
| `setTimeout(fn, 0)`          | 宏任务      | **\~4ms**（Chrome 节流） | 可靠但慢       | ✅ 正常       | 全部                         |
| `requestAnimationFrame`      | 微任务（渲染前） | 与屏幕刷新同步（\~16ms）      | 依赖重绘周期     | ❌ **暂停执行** | 全部                         |
| `requestIdleCallback`        | 空闲回调     | 不可控（浏览器决定）           | ❌ **不可预测** | ❌ **可能暂停** | Chrome/Firefox（Safari 不支持） |
| **MessageChannel（React 选择）** | **宏任务**  | **<1ms**             | ✅ **可靠且快** | ✅ **正常**   | **全部**                     |

### 各方案详细分析：

```javascript
// =============================================
// 方案1：setTimeout(fn, 0)
// =============================================
setTimeout(callback, 0);  // 实际延迟约4-5ms

// 问题：
// - HTML5 规范要求 setTimeout 最小延迟为 4ms（嵌套调用时）
// - 精度不够，时间切片控制粗糙
// - 但作为 fallback 是可接受的（React 在 MessageChannel 不可用时使用）

// =============================================
// 方案2：requestAnimationFrame
// =============================================
requestAnimationFrame(callback);

// 问题：
// - 只在浏览器每次重绘前触发（与显示器刷新率绑定，通常 ~16ms/帧）
// - 如果标签页被切换到后台 → 浏览器暂停 rAF 回调以节省资源
// - React 需要在非重绘时机也能调度任务（比如处理纯计算任务）
// - 本质是"渲染驱动"的，而 React 需要"任务驱动"的

// =============================================
// 方案3：requestIdleCallback —— 看起来最合适但实际不行
// =============================================
requestIdleCallback((deadline) => {
  // deadline.didTimeout: 是否已超时
  // deadline.timeRemaining(): 当前帧剩余空闲时间(ms)
  
  while (deadline.timeRemaining() > 0 && 有任务) {
    执行一个任务;
  }
});

// 设计意图完美匹配 React 需求："在浏览器空闲时执行低优先级任务"
// 但有四个致命问题：

// 问题A：触发频率太低、不可预测
//   - 用户滚动/点击时完全不触发
//   - 可能 50ms 后才执行，也可能 500ms 后
//   - React 需要的是"尽快给一次控制权"，不是"等彻底闲了再给"

// 问题B：timeRemaining() 返回值太小且不稳定
//   - Chrome 中通常 5~50ms，波动大，经常接近 0
//   - React 默认时间切片 5ms，可能什么都做不了
//   - 而 MessageChannel 方案中 React 自己控制时间切片大小

// 问题C：浏览器支持不一致
//   - Chrome 47+ ✅ / Firefox 55+ ✅ / Safari ❌ / 旧 Edge ❌
//   - 既然都要写 polyfill/fallback，不如直接用更可靠的方案

// 问题D：无法自定义优先级和中断策略
//   - 不能说"这个回调优先级更高"
//   - 不能说"现在有紧急任务，先打断当前 idle callback"
//   - 调度策略完全由浏览器控制，开发者无权干涉
//   - React 需要自己管理优先级队列、高优打断低优等能力

// =============================================
// 方案4：MessageChannel ✅（React 的最终选择）
// =============================================

// MessageChannel 的角色只是：「产生一个宏任务」
// 优先级、时间切片、中断恢复等逻辑全部由 React 自己实现
const channel = new MessageChannel();
const port = channel.port2;
channel.port1.onmessage = function () {
  // 这里执行调度回调（进入下一个事件循环后）
  performWorkUntilDeadline();
};

// 触发调度
function schedulePerformWorkUntilDeadline() {
  port.postMessage(null);  // 立即放入宏任务队列
}

// 为什么选它：
// - 比 setTimeout 快：没有 4ms 最小延迟限制
// - 比 rAF 可靠：不依赖浏览器重绘周期，后台标签页也正常工作
// - 比 requestIdleCallback 可控：不被浏览器调度策略牵着鼻子走
// - 所有主流浏览器一致支持
```

**总结：React 不是在找一个"空闲时执行"的 API，而是在找一个"可靠地产生下一个宏任务"的 API。** 其余所有调度逻辑（优先级排序、时间切片、中断恢复）都在 MessageChannel 回调内部由 React 自行实现。

**任务调度的完整生命周期：**

```javascript
// 1. 创建调度任务
function scheduleCallback(priorityLevel, callback, options) {
  const currentTime = getCurrentTime();

  var startTime;
  if (typeof options === 'object' && options !== null) {
    var delay = options.delay;
    if (typeof delay === 'number' && delay > 0) {
      startTime = currentTime + delay;
    } else {
      startTime = currentTime;
    }
  } else {
    startTime = currentTime;
  }

  var timeout;
  switch (priorityLevel) {
    case ImmediatePriority:
      timeout = IMMEDIATE_PRIORITY_TIMEOUT;  // -1（立即过期）
      break;
    case UserBlockingPriority:
      timeout = USER_BLOCKING_PRIORITY_TIMEOUT;  // 250ms
      break;
    case IdlePriority:
      timeout = IDLE_PRIORITY_TIMEOUT;  // 永不过期
      break;
    case LowPriority:
      timeout = LOW_PRIORITY_TIMEOUT;  // 10000ms
      break;
    case NormalPriority:
    default:
      timeout = NORMAL_PRIORITY_TIMEOUT;  // 5000ms;
      break;
  }

  var expirationTime = startTime + timeout;

  var newTask = {
    id: taskIdCounter++,
    callback,
    priorityLevel,
    startTime,
    expirationTime,
    sortIndex: -1,
  };

  if (startTime > currentTime) {
    // 延迟任务：放入timerQueue
    newTask.sortIndex = startTime;
    push(timerQueue, newTask);
    if (peek(taskQueue) === null && newTask === peek(timerQueue)) {
      // 这是timerQueue中最早的任务，设置定时器
      //这段逻辑是这样的：
      //1. 如果存在需要执行的紧急任务 那么不需要处理 因为在紧急任务来时处理的时候会把这个不紧急对列里面过期任务复制到紧急对列里面执行 
      //2. 如果不存在这个紧急任务 并且当前的任务是最早的 那我们预约预约这个事件在规定时间之后触发 
      //在设立定时器的时候 会把到期的任务放进紧急对列 进行调度 
      if (isHostTimeoutScheduled === null) {
        isHostTimeoutScheduled = true;
        // 使用setTimeout设置延迟执行
        requestHostTimeout(handleTimeout, startTime - currentTime);
      }
    }
  } else {
    // 即时任务：放入taskQueue
    newTask.sortIndex = expirationTime;
    push(taskQueue, newTask);
    // 开始调度
    if (!isHostCallbackScheduled && !isPerformingWork) {
      isHostCallbackScheduled = true;
      requestHostCallback(flushWork);
    }
  }

  return newTask;
}

// 2. 刷新任务队列
function flushWork(initialTime) {
  isHostCallbackScheduled = false;
  isPerformingWork = true;

  try {
    // 将timerQueue中到期的任务移入taskQueue
    advanceTimers(initialTime);
    return flushWorkHelper(initialTime);
  } finally {
    isPerformingWork = false;
    // 检查是否还有剩余任务
    if (scheduleHostCallbackIfNeeded !== null) {
      scheduleHostCallbackIfNeeded = true;
      requestHostCallback(flushWork);
    } else {
      // 所有任务完成
      const remainingTime = getTimeRemaining();
      if (remainingTime <= 0) {
        // 时间已到，可能还有timerQueue中的任务
        advanceTimers(getCurrentTime());
      }
    }
  }
}

// 3. 实际执行任务的辅助函数
function flushWorkHelper(initialTime) {
  currentTime = initialTime;

  let currentTask = peek(taskQueue);
  while (currentTask !== null) {
    if (currentTask.expirationTime > currentTime && shouldYieldToHost()) {
      // 任务还没过期且时间片用完，中断执行
      break;
    }

    const callback = currentTask.callback;
    if (callback !== null) {
      currentTask.callback = null;
      currentPriorityLevel = currentTask.priorityLevel;
      
      // 执行回调
      const continuationCallback = callback(currentTask.expirationTime);
      
      if (typeof continuationCallback === 'function') {
        // 任务返回了一个函数（说明任务还没完成），放回队列
        currentTask.callback = continuationCallback;
      } else {
        // 任务已完成，从队列移除
        pop(taskQueue);
      }
    } else {
      pop(taskQueue);
    }
    
    currentTask = peek(taskQueue);
  }

  // 返回是否还有未完成的任务
  return currentTask !== null;
}
```

#### 1.2.7 更新队列与状态管理机制

每个 Fiber 节点都维护着一个更新队列（Update Queue），用于存储所有待处理的 state 更新。

timerQueue:这里是还没到开始时间的任务 按照satrtTime进行排序

```javascript
  // 1. 获取当前更新的优先级lane
  const lane = requestUpdateLane(fiber);

  // 2. 创建update对象
  const update = {
    lane,
    action,
    eagerReducer: null,
    eagerState: null,
    next: null,
  };

  // 3. 入队（追加到环形链表尾部）
  if (queue.pending === null) {
    // 第一个update，指向自己形成环
    update.next = update;
  } else {
    update.next = queue.pending.next;
    queue.pending.next = update;
  }
  queue.pending = update;

  // 4. 优化：尝试提前计算eager state
  const alternate = fiber.alternate;
  if (
    fiber.lanes === NoLanes &&
    (alternate === null || alternate.lanes === NoLanes)
  ) {
    const lastRenderedReducer = queue.lastRenderedReducer;
    if (lastRenderedReducer !== null) {
      let prevEagerState;
      if (queue.lastRenderedStateOverride !== null) {
        prevEagerState = queue.lastRenderedStateOverride;
      } else {
        prevEagerState = queue.lastRenderedState;
      }
      const eagerState = lastRenderedReducer(action, prevEagerState);
      update.eagerReducer = lastRenderedReducer;
      update.eagerState = eagerState;
      
      if (Object.is(eagerState, prevEagerState)) {
        // state没变，可以直接bailout
        return;
      }
    }
  }

  // 5. 调度更新
  const eventTime = requestEventTime();
  const root = markUpdate(fiber);
  scheduleUpdateOnFiber(fiber, lane, eventTime);
}
```

**更新处理的完整流程：**

```javascript
function processUpdateQueue(workInProgress, props, instance, renderLanes) {
  const queue = workInProgress.updateQueue;
  let baseQueue = queue.baseQueue;
  let pendingQueue = queue.shared.pending;

  // 1. 合并pending updates到baseQueue
  if (pendingQueue !== null) {
    const pending = pendingQueue;
    queue.shared.pending = null;

    const lastPendingUpdate = pending;
    const firstPendingUpdate = pending.next;

    if (baseQueue !== null) {
      const lastBaseUpdate = baseQueue.next;
      lastBaseUpdate.next = firstPendingUpdate;
      pending.next = lastBaseUpdate;
    }

    baseQueue = pending;
    queue.baseQueue = baseQueue;
  }

  // 2. 处理baseQueue中的所有updates
  if (baseQueue !== null) {
    const first = baseQueue.next;
    let newState = queue.baseState;
    let newBaseState = null;
    let newBaseQueueFirst = null;
    let newBaseQueueLast = null;
    let update = first;

    do {
      const updateLane = update.lane;
      if (!isSubsetOfLanes(renderLanes, updateLane)) {
        // 优先级不够，跳过这个update（保留到下次处理）
        const clone = {
          lane: updateLane,
          action: update.action,
          eagerReducer: update.eagerReducer,
          eagerState: update.eagerState,
          next: null,
        };
        if (newBaseQueueLast === null) {
          newBaseQueueFirst = newBaseQueueLast = clone;
          newBaseState = newState;
        } else {
          newBaseQueueLast = newBaseQueueLast.next = clone;
        }
      } else {
        // 优先级足够，处理这个update
        if (newBaseQueueLast !== null) {
          const clone = {
            lane: NoLane,
            action: update.action,
            eagerReducer: update.eagerReducer,
            eagerState: update.eagerState,
            next: null,
          };
          newBaseQueueLast = newBaseQueueLast.next = clone;
        }

        // 使用eager state优化
        if (update.eagerReducer === lastRenderedReducer) {
          newState = update.eagerState;
        } else {
          // 正常计算新state
          action = update.action;
          if (action instanceof Function) {
            newState = action(newState);
          } else {
            newState = action;
          }
        }
      }
      update = update.next;
    } while (update !== null && update !== first);

    if (newBaseQueueLast === null) {
      newBaseState = newState;
    } else {
      newBaseQueueLast.next = newBaseQueueFirst;
    }

    queue.baseState = newBaseState;
    queue.baseQueue = newBaseQueueLast;
    queue.lastRenderedState = newState;
  }

  return [newState, queue];
}
```

#### 1.2.8 中断恢复与错误处理机制

Fiber 架构的一个关键能力是**可中断性**——渲染过程可以被暂停并在之后恢复。

**中断点的保存与恢复：**

其实就是： 我们当前的更新全都保存在一个js变量上面的 如果我们需要打断当前的更新 直接停止更新这个js变量 这个js变量就保存下来了 ，在代码里面 我们就是保存了当前的js对象和当前的执行优先级

```javascript
// 全局变量保存中断时的状态
let workInProgress = null;       // 当前正在处理的Fiber
let workInProgressRoot = null;   // 当前正在处理的根Fiber
let workInProgressRootRenderLanes = NoLanes;  // 当前的渲染lanes
let workInProgressRootExitStatus = RootIncomplete;  // 渲染退出状态
let workInProgressRootFatalError = null;  // 致命错误
let workInProgressRootSkippedLanes = NoLanes;  // 被跳过的lanes
let workInProgressRootInterleavedUpdatedLanes = NoLanes;  // 交错更新的lanes
let workInProgressRootPingedLanes = NoLanes;  // 被ping的lanes

  // 保存当前进度到root上
  workInProgressRoot = workInProgress;
  // workInProgress树已经包含了所有中间状态
  // 下次恢复时直接从这里继续

  // 从之前中断的地方恢复
  prepareFreshStack(root, lanes);
  workLoopConcurrent();  // 继续工作循环
}

function prepareFreshStack(root, lanes) {
  root.finishedWork = null;
  root.finishedLanes = NoLanes;

  // 克隆current树作为workInProgress树的起点
  const workInProgress = createWorkInProgress(root.current, null);
  root.workInProgress = workInProgress;
  workInProgressRoot = root;
  workInProgressRootRenderLanes = lanes;
  workInProgressRootInterleavedUpdatedLanes = root.pendingLanes;
  
  return workInProgress;
}
```

```javascript
// 错误处理器
function handleError(root, thrownValue) {
  do {
    try {
      // 1. 向上查找最近的Error Boundary
      const errorBoundary = findNearestErrorBoundary(workInProgress);
      
      if (errorBoundary !== null) {
        // 2. 找到了Error Boundary
        const errorInfo = createErrorInfo(thrownValue);
        
        // 3. 标记Error Boundary需要触发componentDidCatch
        errorBoundary.flags |= ShouldCapture;
        
        // 4. 将错误信息附加到Error Boundary上
        errorBoundary.captureError = {
          value: thrownValue,
          stack: errorInfo.componentStack,
        };
        
        // 5. 跳过当前子树，直接complete Error Boundary
        workInProgress = errorBoundary;
        completeUnitOfWork(errorBoundary);
        return;
      }
      
      // 6. 没有找到Error Boundary，抛出到最顶层
      throw thrownValue;
    } catch (error) {
      // 继续向上查找
      thrownValue = error;
      workInProgress = workInProgress.return;
    }
  } while (workInProgress !== null);
}

// 查找最近的Error Boundary
function findNearestErrorBoundary(workInProgress) {
  let node = workInProgress;
  while (node !== null) {
    const tag = node.tag;
    
    // ClassComponent 且实现了 getDerivedStateFromError 或 componentDidCatch
    if (tag === ClassComponent) {
      const prototype = node.type.prototype;
      if (
        typeof prototype.getDerivedStateFromError === 'function' ||
        typeof prototype.componentDidCatch === 'function'
      ) {
        return node;
      }
    }
    
    node = node.return;
  }
  
  return null;
}
```

**高优先级更新打断低优先级更新的处理：**

```javascript
function checkForForceUpdatePriority() { 
  // 1. 获取当前根节点上所有待处理的更新（车道集合）
  const pendingLanes = workInProgressRoot.pendingLanes; 
  
  if (pendingLanes !== NoLanes) { // 如果确实有待处理的任务
    // 2. 计算下一轮应该执行的车道
    const nextLanes = getNextLanes(pendingLanes, NoLanes); 
    // 3. 提取新任务中优先级最高的那个车道值
    const newCallbackPriority = getHighestPriorityLane(nextLanes); 
    // 4. 获取当前正在渲染的任务优先级
    const currentCallbackPriority = getHighestPriorityLane(workInProgressRootRenderLanes); 
    
    // 💡 关键判断：在 Lane 模型中，数值越小优先级越高（例如 SyncLane=1, DefaultLane=16）
    if (newCallbackPriority < currentCallbackPriority) { 
      // 发现了更高优先级的更新！
      
      // 5. 将当前状态标记为“进行中”，表示工作被中断了，没干完
      workInProgressRootExitStatus = RootInProgress; 
      
      // 6. 记录下当前已经渲染了一部分的 lanes，防止这些进度彻底丢失（后续可能用到）
      const alreadyFinishedLanes = workInProgressRoot.renderedLanes; 
      workInProgressRootSkippedLanes |= alreadyFinishedLanes; 
      
      // 7. 📢 核心动作：重新发起一次调度
      // 这会导致 Scheduler 收到一个更高优先级的回调
      ensureRootIsScheduled(root, currentTime); 
      
      return true;  // 返回 true，告诉上层循环：“快停下，别干了，有人插队！”
    } 
  } 
  
  return false; // 没发现高优先级，继续干活
}
```

### 1.3 Reconciliation 协调算法

#### 1.3.1 Diff算法的核心原则

React的Diff算法遵循三个策略：

1. **只对同层节点进行比较**：不会跨层级移动DOM
2. **不同类型的元素会产生不同的树**：如果节点类型改变，会销毁重建
3. **通过key标识子元素**：开发者可以通过key来提示哪些元素是稳定的

#### 1.3.2 Diff算法的具体实现

**Tree Diff（树对比）：**

```javascript
/**
 * reconcileChildren：协调子节点（Diff 算法的入口）
 * 作用：根据 newChild（新的 ReactElement）的类型，分发到不同的处理逻辑，最终产出新的 Fiber 节点
 */
function reconcileChildren(returnFiber, currentFirstChild, newChild) { 
  // 1. 首先判断新节点是否为对象类型（ReactElement 或 Portal 都是对象）
  if (typeof newChild === 'object' && newChild !== null) { 
    
    // 2. 根据 $$typeof 标识符判断具体的 React 元素类型
    switch (newChild.$$typeof) { 
      
      // 场景 A：单一普通的 React 元素（如 <div>, <App />）
      case REACT_ELEMENT_TYPE: 
        // 💡 调用 placeSingleElement：对比旧 Fiber（currentFirstChild）和新元素，进行单节点 Diff
        return placeSingleElement( 
          returnFiber,        // 传入父 Fiber，用于建立 return 指针
          currentFirstChild,  // 传入旧的第一个子 Fiber，用于对比和复用
          newChild            // 传入新的元素数据
        ); 
      
      // 场景 B：传送门类型（Portal）
      case REACT_PORTAL_TYPE: 
        // 🛠️ 处理 Portal：逻辑与单元素类似，但需要处理不同的挂载容器
        break; 
    } 
 
    // 3. 场景 C：新节点是一个数组（多节点 Diff）
    // 常见于 <ul> {list.map(...)} </ul> 这种情况
    if (isArray(newChild)) { 
      // 📢 调用 reconcileChildrenArray：这是 Diff 算法中最复杂的部分
      // 它会通过两轮遍历，利用 key 值来最小化 DOM 的移动、新增和删除操作
      return reconcileChildrenArray( 
        returnFiber, 
        currentFirstChild, 
        newChild 
      ); 
    } 
  } 
 
  // 4. 场景 D：处理其他类型（如纯文本、数字、或者 null/undefined 代表删除）
  // 此时会进入文本节点的创建逻辑，或者标记旧节点的 Deletion 副作用
  // 处理文本节点等...
```

数组diff：reconcileChildrenArray 是 React Diff 算法中处理 同层级多个子节点 （比如 li 列表）的核心函数。它的目标是： 通过最少的 DOM 操作，将旧的 Fiber 链表转换成新的 Fiber 链表。

不同于 Vue 的双端 Diff，React 采用了一种 单向遍历 + 哈希表 的策略。这个过程可以拆解为两轮遍历。

<br />

1. 从头开始 看一模一样的节点 找到第一个改变的节点 马上退出这轮遍历
2. 处理移动 新增 删除的节点

<br />

```
function reconcileChildrenArray(returnFiber, currentFirstChild, newChildren) {
  let resultingFirstChild = null; // 最终返回的新 Fiber 链表的头结点
  let previousNewFiber = null;   // 用于将新 Fiber 连接成链表的临时变量
  let oldFiber = currentFirstChild; // 当前正在对比的旧 Fiber
  let lastPlacedIndex = 0;       // 决定节点是否移动的标尺（旧树中的最大索引）
  let newIdx = 0;                // 新 ReactElement 数组的下标

  // 第一轮遍历：从头开始对比，尝试复用位置未变的节点
  for (; oldFiber !== null && newIdx < newChildren.length; newIdx++) {
    // 💡 updateSlot：尝试复用。如果 Key 不同，会返回 null
    const newFiber = updateSlot(returnFiber, oldFiber, newChildren[newIdx]);
    
    if (newFiber === null) break; // ❌ Key 不匹配，立即跳出第一轮遍历

    // 记录该节点在旧树中的位置，并判断是否需要移动
    lastPlacedIndex = placeChild(newFiber, lastPlacedIndex, newIdx);

    // 将新生成的 Fiber 接入链表
    if (previousNewFiber === null) resultingFirstChild = newFiber;
    else previousNewFiber.sibling = newFiber;
    previousNewFiber = newFiber;
    oldFiber = oldFiber.sibling; // 移动到下一个旧节点
  }

  // ... 如果新节点已经遍历完，删除剩余旧节点
  
  // 第二轮遍历：处理移动的节点（最核心部分）
  // 1. 将剩余旧节点放入 Map，实现 O(1) 查找
  const existingChildren = mapRemainingChildren(returnFiber, oldFiber);

  // 2. 遍历剩余的新节点
  for (; newIdx < newChildren.length; newIdx++) {
    // 💡 updateFromMap：尝试从哈希表中找 key 相同的节点进行复用
    const newFiber = updateFromMap(existingChildren, returnFiber, newIdx, newChildren[newIdx]);
    
    if (newFiber !== null) {
      if (newFiber.alternate !== null) {
        // 如果是复用的旧节点，需要从 Map 中移除
        existingChildren.delete(newFiber.key === null ? newIdx : newFiber.key);
      }
      // 💡 再次调用 placeChild：根据 oldIndex 和 lastPlacedIndex 决定是否移动
      lastPlacedIndex = placeChild(newFiber, lastPlacedIndex, newIdx);
      
      // 接入链表逻辑...
    }
  }

  return resultingFirstChild;
}
```

Vue 的 Diff 算法经历过两次大的演进，核心思路从 双端对比（Double-ended Diff） 进化到了 快速对比（Fast Diff）+ 最长递增子序列（LIS） 。

<br />

vue 2 的算法源自 Snabbdom 。它的核心思路是： 同时从新旧列表的两端向中间遍历 。

它会维护四个指针：

- oldStartIdx / oldEndIdx （旧列表的头尾）
- newStartIdx / newEndIdx （新列表的头尾） 五步查找策略：

1. 头头对比 ：如果匹配，指针同时后移。
2. 尾尾对比 ：如果匹配，指针同时前移。
3. 头尾对比 ：旧头匹配新尾，说明节点 跑后面去了 ，移动 DOM。
4. 尾头对比 ：旧尾匹配新头，说明节点 跑前面去了 ，移动 DOM。
5. 查表法 ：如果以上四种都没中，就拿新头去旧列表的哈希表里查，查到了就移动，没查到就新建。
   Q: 为什么 React 不用这种双端对比？ A: React 的 Fiber 是 单向链表 ，找“尾部”和“前驱节点”的开销很大；而 Vue 的虚拟 DOM 是 数组 ，随机访问两端非常快。

<br />

<br />

Vue 3 借鉴了 ivi 和 inferno 算法，在双端对比的基础上做了更极端的优化。
第一阶段：预处理（从两端“剥洋葱”）

- 自前向后 ：先对比头部，直到遇到不一样的。
- 自后向前 ：再对比尾部，直到遇到不一样的。
  这样可以快速排除掉列表开头和结尾没变化的节点。 第二阶段：处理剩余节点（最关键）
  如果剥完洋葱还剩下中间一截，Vue 3 会构建一个 source 数组，记录 新节点在旧列表中的索引位置 。

算法核心：最长递增子序列（Longest Increasing Subsequence, LIS）

- Vue 3 会根据 source 数组计算出一个 LIS。
- LIS 里的节点 ：代表它们之间的 相对顺序没变 ，不需要移动。
- 非 LIS 里的节点 ：才需要进行移动操作。

**Component Diff（组件对比）：**

```javascript
/**
 * reconcileSingleElement：单节点协调
 * 场景：新节点（element）只有一个，但对应的旧 Fiber 可能有多个（比如从列表变成了单个组件）
 */
function reconcileSingleElement(returnFiber, currentFirstChild, element) { 
  const key = element.key; // 获取新元素的 key
  let child = currentFirstChild; // 获取旧 Fiber 链表的第一个节点
 
  // 1. 遍历旧的 Fiber 链表，尝试寻找可以复用的节点
  while (child !== null) { 
    // 💡 第一关：比对 Key。Key 是复用的先决条件。
    if (child.key === key) { 
      // 2. Key 匹配上了，进入类型检查阶段
      switch (child.tag) { 
        case FunctionComponent: 
        case ClassComponent: 
        case HostComponent: { 
          // 💡 第二关：比对 Type（元素类型，如 'div' 或组件类/函数）
          if (child.elementType === element.type) { 
            // ✅ 匹配成功：Key 和 Type 都相同，可以复用！
            
            // 3. 📢 核心动作：既然已经找到了唯一的复用节点，那么旧链表中剩下的兄弟节点全部标记删除
            deleteRemainingChildren(returnFiber, child.sibling); 
            
            // 4. 调用 useFiber：基于旧 Fiber 和新 Props 克隆一个新 Fiber（双缓存机制）
            const existing = useFiber(child, element.props); 
            existing.return = returnFiber; // 建立父级指针
            return existing; // 返回复用的节点，Diff 结束
          } 
          
          // ❌ 匹配失败：Key 相同但 Type 不同。
          // 在 React 的逻辑中，如果 Key 相同但类型变了，说明该位置的节点及其后续节点都不可复用。
          deleteRemainingChildren(returnFiber, child); 
          break; 
        } 
      } 
    } else { 
      // ❌ Key 不匹配：说明当前的旧 Fiber 肯定不能给这个新元素用。
      // 5. 📢 核心动作：标记删除当前这一个旧 Fiber，然后继续往后找兄弟节点（看看后面有没有 key 匹配的）
      deleteChild(returnFiber, child); 
    } 
    child = child.sibling; // 移动指针，继续比对下一个旧兄弟节点
  } 
 
  // 6. 最终落地方案：如果遍历完旧链表都没找到能复用的节点
  // 那么就根据新元素创建一个全新的 Fiber 节点
  const created = createFiberFromElement(element, returnFiber.mode, lanes); 
  created.return = returnFiber; // 建立父级指针
  return created; // 返回新创建的节点
}
```

**Element Diff（元素对比）- 同层列表Diff：**

这是最复杂的部分

理解 React 的多节点 Diff，关键不在于复杂的循环，而在于理解一个“ 标尺 ”的概念。

这个标尺就是 lastPlacedIndex 。

你可以把它想象成一个\*\*“已经固定好的节点在旧树中的最大索引”\*\*。

- 逻辑核心 ：React 认为，在新的顺序中，节点应该是 单调递增 地从旧树里取出来的。
- 判断标准 ：
  - 如果当前复用的节点在旧树里的位置 oldIndex 比标尺大 ：说明它的相对顺序没乱， 不用动 ，并把标尺更新到这个更大的位置。
  - 如果当前复用的节点在旧树里的位置 oldIndex 比标尺小 ：说明它本来在后面，现在却跑前面去了， 必须移动 。

我们一步步模拟 React 是怎么思考的：
状态初始化

- 旧序列 ：A(0), B(1), C(2)
- 新序列 ：B, A, C
- 标尺 lastPlacedIndex = 0 第一步：看新序列的第一个 \[B]

1. React 发现新序列第一个是 B，但旧序列第一个是 A。 Key 不匹配！
2. 第一轮对比失败，直接把旧节点全部塞进 Map： { A: 0, B: 1, C: 2 } 。 第二步：处理新序列的 \[B]
3. 去 Map 里找 B，找到了！它的 oldIndex = 1 。
4. 比对标尺 ： oldIndex(1) > lastPlacedIndex(0) 。
5. 结论 ：B 的位置很靠后，不需要移动。
6. 更新标尺 ： lastPlacedIndex = 1 。 第三步：处理新序列的 \[A]
7. 去 Map 里找 A，找到了！它的 oldIndex = 0 。
8. 比对标尺 ： oldIndex(0) < lastPlacedIndex(1) 。
9. 结论 ： A 居然比标尺还小！ 说明 A 原本在 B 前面，现在却要排在 B 后面。
10. 操作 ：标记 A 为 Placement （移动）。
11. 标尺状态 ：保持 1 不变（标尺只往大处更新）。 第四步：处理新序列的 \[C]
12. 去 Map 里找 C，找到了！它的 oldIndex = 2 。
13. 比对标尺 ： oldIndex(2) > lastPlacedIndex(1) 。
14. 结论 ：C 还是在最后，不需要移动。
15. 更新标尺 ： lastPlacedIndex = 2 。

这种算法有一个 致命弱点 ：
如果你把 \[A, B, C, D] 变成 \[D, A, B, C] ：

1. 处理 D： oldIndex=3 , lastPlacedIndex=0 -> D 不动，标尺变 3。
2. 处理 A： oldIndex=0 < 3 -> A 移动 。
3. 处理 B： oldIndex=1 < 3 -> B 移动 。
4. 处理 C： oldIndex=2 < 3 -> C 移动 。

```javascript
/**
 * reconcileChildrenArray：多节点协调算法
 * 目标：将旧的 Fiber 链表与新的 ReactElement 数组进行对比，生成新的 Fiber 链表
 */
function reconcileChildrenArray(returnFiber, currentFirstChild, newChildren) { 
  let resultingFirstChild = null; // 最终产出的新 Fiber 链表的头结点
  let previousNewFiber = null;   // 用于连接新 Fiber 节点的临时指针
  let oldFiber = currentFirstChild; // 当前对比的旧 Fiber 节点
  let lastPlacedIndex = 0;       // 💡 核心标尺：记录已复用节点在旧树中的最大位置，用于判断是否发生“移动”
  let newIdx = 0;                // 新元素的下标计数器
  let nextOldFiber = null;       // 暂存下一个旧 Fiber，防止链表指针丢失
 
  // ============================================================
  // 第一轮遍历：尝试按索引（Index）一一对应地复用节点
  // 场景：绝大多数更新中，节点的位置是不变的，只是属性变了
  // ============================================================
  for (; oldFiber !== null && newIdx < newChildren.length; newIdx++) { 
    // 💡 处理索引偏移：如果旧节点的 index 大于当前 newIdx，说明中间有空隙
    if (oldFiber.index > newIdx) { 
      nextOldFiber = oldFiber; 
      oldFiber = null; 
    } else { 
      nextOldFiber = oldFiber.sibling; 
    } 
 
    // 💡 updateSlot：核心复用逻辑。它会检查 key 是否匹配。
    // 如果 key 不匹配，直接返回 null，宣告第一轮遍历结束
    const newFiber = updateSlot(returnFiber, oldFiber, newChildren[newIdx]); 
 
    if (newFiber === null) { 
      // 如果无法复用，需要把 oldFiber 还原回来，因为后续还要用它进行 Map 比对
      if (oldFiber === null) { 
        oldFiber = nextOldFiber; 
      } 
      break; // ❌ 发现 Key 不匹配，第一轮快速对比失败，跳出
    } 
 
    if (shouldTrackSideEffects) { 
      // 如果是复用的节点，且是全新的（alternate 为空），标记为 Placement（插入）
      if (oldFiber && newFiber.alternate === null) { 
        newFiber.flags = Placement; 
      } 
    } 
 
    // 💡 placeChild：更新 lastPlacedIndex。如果当前节点在旧树的索引小于标尺，标记为移动
    lastPlacedIndex = placeChild(newFiber, lastPlacedIndex, newIdx); 
 
    // 将新生成的 Fiber 接入新链表
    if (previousNewFiber === null) { 
      resultingFirstChild = newFiber; 
    } else { 
      previousNewFiber.sibling = newFiber; 
    } 
    previousNewFiber = newFiber; 
    oldFiber = nextOldFiber; // 移动指针
  } 
 
  // ============================================================
  // 第二轮遍历（分支 A）：新节点已经跑完，但旧节点还有剩
  // 场景：列表缩减（例如从 5 个元素删减到 3 个）
  // ============================================================
  if (newIdx === newChildren.length) { 
    // 📢 动作：直接把剩下的旧 Fiber 节点全部标记为 Deletion（删除）
    deleteRemainingChildren(returnFiber, oldFiber); 
    return resultingFirstChild; 
  } 
 
  // ============================================================
  // 第二轮遍历（分支 B）：旧节点跑完了，但新节点还有剩
  // 场景：列表扩充（例如在末尾新增了几个元素）
  // ============================================================
  if (oldFiber === null) { 
    for (; newIdx < newChildren.length; newIdx++) { 
      // 📢 动作：为剩下的新元素直接创建全新的 Fiber 节点
      const newFiber = createChild(returnFiber, newChildren[newIdx]); 
      if (newFiber === null) continue; 
 
      // 标记为新增位置
      lastPlacedIndex = placeChild(newFiber, lastPlacedIndex, newIdx); 
      
      // 继续连接链表
      if (previousNewFiber === null) { 
        resultingFirstChild = newFiber; 
      } else { 
        previousNewFiber.sibling = newFiber; 
      } 
      previousNewFiber = newFiber; 
    } 
    return resultingFirstChild; 
  } 
 
  // ============================================================
  // 第三轮遍历：处理最复杂的“节点移动”场景
  // 场景：列表顺序发生了翻天覆地的变化（如 A-B-C 变成 B-C-A）
  // ============================================================
```

```javascript
    startTime,
    expirationTime: startTime + timeout,
    sortIndex: -1,
  };

  if (startTime > currentTime) {
    // 延迟任务
    newNode.sortIndex = startTime;
    push(timerQueue, newNode);
  } else {
    // 即时任务
    newNode.sortIndex = expirationTime;
    push(taskQueue, newNode);

    if (!isHostCallbackScheduled && !isPerformingWork) {
      isHostCallbackScheduled = true;
      requestHostCallback(flushWork);
    }
  }

  return newNode;
}
```

#### 1.4.3 Lane模型（车道模型）

Lane是React 18中用于表示优先级的位运算系统：

```javascript
export const TotalLanes = 31;

// 同步车道：用于离散事件（点击、键盘）
export const SyncLane = 0b0000000000000000000000000000001;
export const SyncDefaultLanes = 0b0000000000000000000000000000011;

// 输入连续车道：用于连续事件（拖动、滚动）

export const DefaultHydrationLane = 0b0000000000000000000000000010000;
export const DefaultLane = 0b0000000000000000000000000100000;

// 过渡车道：非紧急更新（搜索建议等）

export const TransitionLanes = 0b0000000000001111111111110000000;

// 重试车道

// 空闲车道

export const IdleLanes = 0b1100000000000000000000000000000;

// 离屏车道
export const OffscreenLane = 0b1000000000000000000000000000000;

// Lane操作工具函数
function getHighestPriorityLanes(lanes) {
  return lanes & -lanes;  // 获取最低位的1，即最高优先级
}

function mergeLanes(a, b) {
  return a | b;  // 合并lanes
}

function removeLanes(set, subset) {
  return set & ~subset;  // 移除特定lanes
}

function isSubsetOfLanes(set, subset) {
  return (set & subset) === subset;  // 判断是否包含
}
```
